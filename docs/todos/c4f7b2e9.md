Planners and coders dispatched by `generate-plan` and `execute-plan` should not be exposed to specs, todos, plans, and review files from previous unrelated tasks. Old artifacts on disk are noise that can mislead the dispatched agent or dilute its attention. The dispatch context for each agent should be scoped to only the artifacts immediately relevant to its task.

Spinoff from TODO-01bb6a2b (verdict alignment); separated because the affected agents and skills differ — TODO-01bb6a2b targets reviewers and refiners, this one targets planners and coders.

## Desired behavior

- **Planner (full plan):** sees the latest spec only. No prior plans, review files, or specs for unrelated tasks.
- **Planner (edit round):** sees the latest spec plus the findings from the immediate previous review pass only. No older review files.
- **Coder (full implementation):** sees the latest spec and the latest plan. No prior reviews.
- **Coder (remediation):** sees the latest spec and the findings from the most recent review. No older plans or reviews.

## Open design questions

- How is "latest" resolved? By filename date prefix, by explicit pointer in another artifact, by tracking via workflow state, or by a convention enforced at dispatch?
- What counts as "irrelevant" — only items archived under `.pi/.../archived/`, or also recent-but-unrelated artifacts on the same branch?
- How do existing provenance chains (specs reference todos; plans reference specs; reviews reference plans) interact with this scoping? References stay intact in artifact bodies, but does the dispatch context filter unreferenced artifacts?
- Does scoping happen at dispatch-time (the skill assembles a curated context bundle) or via prompt instruction to the agent (e.g., "do not read other files in `.pi/specs/`")?
- Do refiner edit-pass and remediation dispatches need a different mechanism than initial generate-plan/execute-plan dispatches, or can both use the same scoping primitive?

## Likely affected files

- `agent/skills/generate-plan/SKILL.md` and the planner prompt templates it dispatches with
- `agent/skills/execute-plan/SKILL.md` and the coder prompt templates it dispatches with
- `agent/skills/refine-plan/refine-plan-prompt.md` (planner edit-pass section)
- `agent/skills/refine-code/refine-code-prompt.md` (coder remediation section)
- Possibly `agent/agents/planner.md` and `agent/agents/coder.md` if standing identity rules are needed

## Acceptance criteria

- Planners on full-plan dispatch receive only the latest spec; older specs, plans, and reviews are not in the context.
- Planners on edit rounds receive only the latest spec and the immediately preceding review findings.
- Coders on full-implementation dispatch receive only the latest spec and the latest plan.
- Coders on remediation dispatch receive only the latest spec and the most recent review findings.
- The mechanism is documented in the relevant skill instructions so future authors maintain the scoping discipline.
- Manual smoke run: dispatch a planner or coder in a directory containing several old specs/plans/reviews and confirm the dispatched session does not surface those older artifacts in its working context.
