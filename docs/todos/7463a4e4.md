## Problem

`generate-plan` currently keeps the plan review/edit loop in the main orchestrator session:

1. dispatch `planner` to write the plan
2. dispatch `plan-reviewer`
3. receive the full review in `results[0].finalMessage`
4. write `.pi/plans/reviews/...` from the main session
5. parse review findings in the main session
6. dispatch the planner edit pass
7. repeat until approved or budget exhausted
8. commit the final plan + review artifacts from `generate-plan`

This is true today because `plan-reviewer` is intentionally judge-only/read-only and returns its review through `finalMessage`; the main orchestrator persists the review file. That keeps `plan-reviewer` pure, but it means the user's long-lived main agent context absorbs review text and loop-control details during planning.

By contrast, `refine-code` already has a reusable `refine-code` skill and a short-lived `code-refiner` coordinator subagent that owns review/remediate iteration, writes versioned review artifacts, and returns a compact status/path summary to the caller. Code-review context pressure is contained inside the coordinator.

The plan-review case is more problematic for main-session context because `generate-plan` often happens before substantial implementation work, so preserving the main agent context can matter more.

## Desired direction

Introduce a reusable `refine-plan` skill, analogous to `refine-code`, backed by a narrow `plan-refiner` coordinator subagent.

Intended high-level flow:

```text
generate-plan
  -> planner writes initial draft plan
  -> invoke refine-plan with that plan path
  -> refine-plan reviews/refines the plan and owns the plan-artifact commit gate
  -> generate-plan receives compact completion/commit status
  -> generate-plan offers execute-plan

refine-plan .pi/plans/<plan>.md
  -> can also be run standalone against an existing plan
  -> owns the same plan-artifact commit gate
```

Keep roles separate:

- `planner` writes initial plans and applies plan edit passes.
- `plan-reviewer` remains judge-only/read-only; it reviews and returns findings, but does not write files.
- `refine-plan` is the user/caller-facing skill: gather/validate inputs, resolve model tiers, dispatch the coordinator, parse compact result, validate artifact paths, handle user decisions, and own the commit gate for refined plan artifacts in all normal cases.
- `plan-refiner` is the coordinator subagent: dispatches `plan-reviewer`, writes review artifacts, parses review status/findings, dispatches planner edit passes, tracks iteration budget/eras, appends Review Notes when appropriate, and returns compact status + paths to `refine-plan`.
- `generate-plan` becomes thinner: after initial plan generation it invokes `refine-plan`, then reports the result and offers `execute-plan`. It should not write review files, parse full review text, track review versions, or commit plan/review artifacts itself.

Do **not** make `plan-reviewer` itself write review files unless a later design explicitly accepts the loss of read-only judge boundaries. The oddity to fix is that the main orchestrator owns loop state and review persistence, not that the reviewer is read-only.

## Commit ownership

`refine-plan` should be the single owner of the plan/refinement commit gate, both standalone and when called by `generate-plan`.

Layering:

- `plan-refiner` does **not** commit. It is short-lived and owns coordination/artifact writing only.
- `refine-plan` validates the plan/review artifact paths returned by `plan-refiner`, shows the result to the user, asks whether to commit the refined plan artifacts, and invokes the `commit` skill with only the concrete plan path + concrete review paths from the current run.
- `generate-plan` does **not** have a separate caller-owned commit mode. It delegates refinement and commit gating to `refine-plan` and receives the outcome (e.g. committed with SHA, left uncommitted, issues remaining, failed).

Default policy:

- Do not commit every individual review -> plan-edit attempt. Intermediate plan states are draft states, not durable implementation milestones.
- Commit once at the end of a successful/accepted `refine-plan` run, after user approval.
- If `refine-plan` reaches budget exhaustion and the user chooses to proceed with issues, offer the same final commit gate for the current plan + review artifacts.
- If the user declines the commit gate, leave artifacts uncommitted and return an explicit `uncommitted` outcome to the caller.
- `generate-plan` should only auto-offer `execute-plan` after `refine-plan` finishes; if the plan was left uncommitted, the offer should make that state clear or require explicit user confirmation to proceed with an uncommitted plan.

## Consistency with refine-code

Aim for workflow consistency, not a shared abstraction and not perfect behavioral identity.

Good symmetry to preserve:

| Layer | Code workflow | Plan workflow |
|---|---|---|
| Outer caller | `execute-plan` | `generate-plan` |
| Reusable skill | `refine-code` | `refine-plan` |
| Coordinator agent | `code-refiner` | `plan-refiner` |
| Judge agent | `code-reviewer` | `plan-reviewer` |
| Fixer/editor agent | `coder` | `planner` edit pass |
| Review artifacts | `.pi/reviews/...` | `.pi/plans/reviews/...` |
| Main session gets | compact status/path summary | compact status/path/commit summary |

Intentional asymmetries to preserve:

- `code-refiner` commits remediation batches internally; `plan-refiner` must not commit. Plan generation/refinement commit gating belongs to `refine-plan`.
- `code-refiner` has hybrid re-reviews plus final full-diff verification; `plan-refiner` should probably full-review the whole plan every pass because plan correctness is global (coverage, dependencies, ordering, acceptance criteria).
- `code-refiner` batches code findings; `plan-refiner` can usually send all blocking Error findings to one planner edit pass because the plan is one coherent artifact.
- `code-refiner` copies latest review to an unversioned `.pi/reviews/<name>-code-review.md`; no unversioned plan-review copy is required unless a concrete consumer appears.

Do not introduce a generic refiner abstraction.

## Versioning model to preserve/clarify

Current plan behavior:

- The plan file itself is not filename-versioned. It is overwritten in place during planner edit passes (e.g. `.pi/plans/foo.md`). Git history provides durable version history once committed.
- Plan review files are versioned under `.pi/plans/reviews/` as `.pi/plans/reviews/<plan-basename>-plan-review-vN.md`.
- `vN` is an **era number**, not an individual review attempt number.
- Within one era, up to the configured review/edit iteration budget may run, and the current era's review file is overwritten in place with the latest review output.
- If the budget is exhausted and the user chooses to keep iterating, the workflow starts a new era (`v1` -> `v2`, etc.).
- The final commit includes the stable plan path plus every concrete review file path written by the current run/eras.

This model still makes sense with `refine-plan`, but standalone runs need collision-safe version allocation:

- When `generate-plan` creates a fresh plan basename, `refine-plan` can normally start at `v1`.
- When standalone `refine-plan .pi/plans/<plan>.md` runs against an existing plan, it must scan `.pi/plans/reviews/` for existing `<plan-basename>-plan-review-vN.md` files and start at `max(existing N) + 1` to avoid overwriting old committed review artifacts.
- A budget reset within that standalone run increments again from that starting era.
- The returned review path list contains only concrete review files written during the current run, not older versions and not globs.

## Sketch

Possible compact final signal from `plan-refiner` to `refine-plan`:

```text
STATUS: approved | issues_remaining | failed

## Summary
Iterations: <N>
Errors found: <N>
Errors fixed: <N>
Warnings/suggestions appended: <N>

## Remaining Issues (only if issues_remaining)
<compact summary or references to review file sections>

## Plan File
.pi/plans/<date>-<slug>.md

## Review Files
- .pi/plans/reviews/<plan-basename>-plan-review-vN.md
- .pi/plans/reviews/<plan-basename>-plan-review-vN+1.md
```

`refine-plan` validates the returned paths exist, presents the result, then offers the commit gate. Its own final result back to a caller such as `generate-plan` should include compact status plus commit outcome, for example:

```text
STATUS: approved | issues_remaining | failed
COMMIT: committed <sha> | left_uncommitted | not_attempted
PLAN_PATH: .pi/plans/<date>-<slug>.md
REVIEW_PATHS:
- .pi/plans/reviews/<plan-basename>-plan-review-vN.md
```

## Scope considerations

`refine-plan` should accept at least:

- plan path (required)
- original task artifact/provenance (optional but preferred)
- max iterations (default 3)
- review output directory/base (default derived from plan basename)

Standalone provenance is the main design caveat:

- For plans generated from specs, the plan usually contains `**Spec:** .pi/specs/<filename>`.
- For plans generated from todos, the plan may contain `**Source:** TODO-<id>` and the todo file may still exist.
- For freeform inputs, there may be no durable original source.

`refine-plan` needs an explicit policy for missing provenance. Recommended: fail/ask for the original source unless the user explicitly requests structural-only review. A structural-only plan review should be clearly labeled as such because it cannot verify full source coverage.

If budget exhaustion needs a user decision (`keep iterating` vs `proceed with issues`), prefer returning `issues_remaining` from `plan-refiner` to `refine-plan` and letting `refine-plan` ask the user. If `generate-plan` invoked `refine-plan`, `refine-plan` can surface the same choices and then return the final status/commit outcome to `generate-plan`.

## Acceptance criteria

- A new `refine-plan` skill exists and can be invoked standalone against an existing `.pi/plans/<plan>.md` file.
- A new local coordinator agent (e.g. `agent/agents/plan-refiner.md`) exists for the generate-plan review/edit coordinator role.
- `plan-refiner` runs through a CLI that exposes pi orchestration tools (`subagent_run_serial`), not a Claude-only coordinator path.
- `plan-reviewer` remains read-only/judge-only and does not gain `write` or `edit` solely to persist review files.
- `refine-plan` dispatches `plan-refiner` and receives only compact status plus concrete artifact paths during normal review/edit iterations.
- `refine-plan` validates returned plan/review paths exist before presenting the commit gate.
- `refine-plan` owns the commit gate in both standalone use and when invoked by `generate-plan`; `generate-plan` does not separately commit plan/review artifacts.
- `generate-plan` delegates plan review/edit and plan-artifact commit gating to `refine-plan` instead of receiving full plan-review text in the main session.
- Review text is written by `plan-refiner` to `.pi/plans/reviews/...`; the main `generate-plan` session does not ingest full review text during normal operation.
- Review artifact naming/versioning remains compatible with `.pi/plans/reviews/<plan-basename>-plan-review-vN.md`, where `vN` is an era number.
- Standalone `refine-plan` starts at the next unused `vN` for existing plans rather than overwriting old review artifacts.
- The existing behavior of appending approved warnings/suggestions to the plan as `## Review Notes` is preserved, but performed inside the refine-plan/plan-refiner flow without inlining full review text into the main `generate-plan` session.
- Budget exhaustion still returns control to the user with the same choices (`keep iterating` or `proceed with issues`) or an explicitly improved equivalent.
- The commit skill is invoked only with the concrete plan path and concrete review paths from the current run; no globs/wildcards and no older review versions from prior standalone runs.
- If the user declines the `refine-plan` commit gate, artifacts remain uncommitted and the final result clearly reports `left_uncommitted` (or equivalent) to any caller.
- `generate-plan` reports the `refine-plan` commit outcome before offering `execute-plan`; proceeding with an uncommitted plan requires explicit user confirmation or an equally clear acknowledgement.
- No internal plan-refiner commits are made, and no commits are created for each review/edit attempt by default.
- A smoke run on a small spec/todo confirms the plan is generated, refined through `refine-plan`, review artifact(s) are written, `refine-plan` owns the commit gate, and the main `generate-plan` session receives only compact status/path/commit output.
- A standalone smoke run on an existing plan confirms review version allocation starts at the next unused `vN` and does not overwrite existing review artifacts.

## Non-goals

- Do not generalize into a reusable generic artifact-refiner abstraction.
- Do not merge this with `code-refiner`.
- Do not move final commit responsibility into `plan-refiner`.
- Do not make `plan-refiner` commit internally.
- Do not commit every individual review/edit attempt by default.
- Do not keep a separate caller-owned-commit mode in `generate-plan`; `refine-plan` owns the commit gate.
- Do not change `execute-plan` except where documentation/examples mention the existing `generate-plan -> execute-plan` handoff.
- Do not expand the responsibilities of `plan-reviewer` beyond judging/reviewing.
- Do not add an unversioned latest plan-review copy unless a concrete consumer is identified.
