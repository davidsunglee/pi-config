## Problem

The pi-interactive-subagent spec at `/Users/david/Code/pi-interactive-subagent/.pi/specs/2026-04-29-orchestration-result-artifacts.md` will change orchestration result hand-off semantics so full child `finalMessage` bodies are written by the parent orchestration layer to per-task artifact files, with the model-facing orchestration result pointing at `artifact:` paths.

This should fix the unreliable review hand-off from read-only reviewer subagents to coordinator agents: `plan-reviewer` and `code-reviewer` do not have `write` or `edit`, but the parent orchestration glue will write their final messages to disk. However, the pi-config refiner prompts currently instruct coordinators to read reviewer output from `results[0].finalMessage` directly.

Affected files:

- `agent/skills/refine-plan/SKILL.md`
- `agent/skills/refine-plan/refine-plan-prompt.md`
- `agent/skills/refine-code/SKILL.md`
- `agent/skills/refine-code/refine-code-prompt.md`

## Desired fix

Once orchestration result artifacts are implemented in `pi-interactive-subagent`, update the refiner skills/prompts to consume the new contract explicitly.

1. In `refine-plan-prompt.md`, after each `plan-reviewer` dispatch via `subagent_run_serial`, instruct `plan-refiner` to:
   - read the `artifact:` path / `results[0].artifactPath` for the reviewer task,
   - use the file contents as the full reviewer output,
   - only fall back to inline `results[0].finalMessage` when no artifact path is present and the inline body is non-empty,
   - fail with `plan-reviewer returned empty result` if neither source has content.
2. In `refine-code-prompt.md`, apply the same rule for every `code-reviewer` dispatch: first-pass review, hybrid re-review, and final verification.
3. Update the outer skills' coordinator-result parsing:
   - `refine-plan/SKILL.md` Step 9 should read the `plan-refiner` orchestration artifact path before parsing `STATUS`, `## Plan File`, and `## Review Files`.
   - `refine-code/SKILL.md` Step 5 should read the `code-refiner` orchestration artifact path before parsing `STATUS` and `## Review File`.
4. Keep the existing compact coordinator-output contract intact. The final caller-facing `plan-refiner` / `code-refiner` response should still be status + concrete paths, not full review text.
5. Preserve the no-inline-review hard rules. Artifact fallback is only a transport/readback fallback for a successfully dispatched worker; it is not permission for the coordinator to perform its own review.

## Acceptance criteria

- `rg -n "results\[0\]\.finalMessage|finalMessage" agent/skills/refine-plan agent/skills/refine-code` shows no instruction that relies on inline `finalMessage` as the primary source for orchestration results.
- `plan-refiner` persists the full `plan-reviewer` output by reading the orchestration artifact produced for the reviewer task, even though `plan-reviewer` has no `write`/`edit` tools.
- `code-refiner` persists the full `code-reviewer` output by reading the orchestration artifact produced for the reviewer task, even though `code-reviewer` has no `write`/`edit` tools.
- The outer `refine-plan` and `refine-code` skills parse coordinator status from the coordinator task artifact path, not from truncated inline result text.
- Existing provenance validation still passes: persisted review files begin with `**Reviewer:** <provider>/<model> via <cli>`.
- Manual smoke run: trigger a reviewer output large enough to exceed the old inline preview; confirm the coordinator writes the complete review file and the caller receives only compact status/path output.
