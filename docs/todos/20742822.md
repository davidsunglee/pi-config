# Placeholder: Improve front-end workflow with domain-aware shaping and vertical-slice planning

This todo intentionally has two parts and can be split into separate specs/plans if needed.

## Part 1 — Improve `define-spec`

Goal: make `define-spec` the organic front-end of the workflow by absorbing the best parts of Matt Pocock's `grill-with-docs`, rather than bolting on a separate pre-step.

Recommendations:

- Treat `define-spec` as "shape work into an executable spec," not just "ask questions then write requirements."
- Read domain docs before Q&A:
  - `CONTEXT.md`
  - `CONTEXT-MAP.md` when present
  - relevant `docs/adr/*` files
- During Q&A:
  - ask one question at a time;
  - provide a recommended answer with rationale;
  - challenge vague or overloaded terms;
  - compare user language against existing domain vocabulary;
  - explore the codebase instead of asking when the answer is discoverable;
  - use concrete scenarios to stress-test domain relationships and edge cases.
- Update `CONTEXT.md` inline when domain terms are clarified, using a lightweight glossary/relationships/example-dialogue format inspired by `grill-with-docs`.
- Offer ADRs sparingly when a decision is hard to reverse, surprising without context, and based on a real trade-off.
- Keep `.pi/specs/<date>-topic>.md` as the canonical shaping artifact; do not introduce a mandatory separate PRD step.
- Expand the spec template to include, where useful:
  - `## Problem`
  - `## Solution Shape`
  - `## Domain Language`
  - `## User Stories`
  - `## Requirements`
  - `## Constraints`
  - `## Decisions`
  - optional `## Approach`
  - `## Acceptance Criteria`
  - `## Non-Goals`
  - `## Open Questions`
- Commit the spec plus any updated `CONTEXT.md` / ADR files through the normal review gate.

## Part 2 — Improve `generate-plan`

Goal: make generated plans vertical-slice-first while preserving this repo's stronger executable-plan machinery.

Recommendations:

- Update the planner contract so numbered tasks are preferably thin tracer-bullet vertical slices.
- A good task should deliver the smallest end-to-end behavior that proves a capability works, rather than a horizontal implementation chunk such as "create types," "add UI," or "write tests" in isolation.
- Require each task to map back to one or more requirements, acceptance criteria, or user stories from the spec.
- Require each task to be independently demoable or objectively verifiable where possible.
- Allow horizontal tasks only when explicitly justified, such as migrations, shared infrastructure, or unavoidable sequencing work.
- Preserve strict acceptance-criterion + `Verify:` recipe structure.
- Update plan review to flag:
  - horizontal slicing without justification;
  - tasks that are not independently verifiable;
  - tasks that do not map to spec requirements/user stories;
  - dependency chains that could be converted into vertical slices.
- Keep `execute-plan` mostly unchanged initially; adapt only as needed after the new plan shape is stable.
- Defer optional issue publishing until later. If added, it should be a separate optional skill such as `publish-issues <plan>`, not part of the mandatory path.
