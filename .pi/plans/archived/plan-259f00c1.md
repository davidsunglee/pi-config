# Implementation Plan

## Goal
Create a pi extension at `~/.pi/agent/extensions/custom-footer.ts` that replaces the default footer with one where individual fields are colored using theme token references, so colors auto-adapt on theme switch. The extension must faithfully reproduce the default footer's layout and behavior — including the context usage color escalation (yellow >70%, red >90%) — while making each field's color independently configurable.

## Context

Based on TODO-259f00c1 and the pi source code:

- The **default footer** (`dist/modes/interactive/components/footer.js`) renders 2–3 lines:
  - **Line 1:** `~/path (branch) • session-name` — all dim
  - **Line 2 left:** `↑input ↓output Rcache Wcache $cost context%/contextWindow (auto)` — all dim, except context % which escalates to warning/error
  - **Line 2 right:** `(provider) model • thinking-level` — all dim
  - **Line 3 (optional):** extension statuses — passed through as-is (may be pre-styled)
- The **`setFooter` API** (Pattern 6 in TUI docs) provides `(tui, theme, footerData) => { render, invalidate, dispose? }`
- Extensions have access to `ctx.getContextUsage()`, `ctx.sessionManager.getEntries()`, `ctx.model`, `ctx.cwd`, `ctx.modelRegistry.isUsingOAuth()`, `ctx.getThinkingLevel()`, `ctx.getSessionName()`, and `footerData.getGitBranch()`, `footerData.getExtensionStatuses()`, `footerData.getAvailableProviderCount()`

  ```typescript
    if (count < 1000) return count.toString();
    if (count < 10000) return `${(count / 1000).toFixed(1)}k`;
    if (count < 1000000) return `${Math.round(count / 1000)}k`;
    if (count < 10000000) return `${(count / 1000000).toFixed(1)}M`;
    return `${Math.round(count / 1000000)}M`;
  }
  ```

## Risks

1. **`getEntries()` vs `getBranch()`** — The default footer uses `getEntries()` (all entries including pre-compaction). The example extension uses `getBranch()`. We must use `getEntries()` to match default behavior for cumulative token totals.

2. **`getThinkingLevel()` and `getSessionName()` availability** — These exist on `ExtensionCommandContext` (available in `session_start` handler). Need to verify they're accessible in the `setFooter` render callback via the captured `ctx` closure. If not, we'll need to cache them and update via event listeners.

3. **Extension statuses may be pre-styled** — Other extensions pass already-styled ANSI strings via `ctx.ui.setStatus()`. The default footer passes them through without re-coloring. We should do the same (don't wrap with `theme.fg()`).

4. **Context usage may be `undefined`** — `ctx.getContextUsage()` returns `undefined` when no model is set. Handle gracefully by falling back to `contextWindow` from `ctx.model` or showing `0`.

5. **Theme token names must be valid** — All tokens used (`accent`, `dim`, `warning`, `error`) are standard tokens defined in every theme. The theme system handles missing tokens gracefully.

6. **Width calculation with mixed ANSI** — When symbols and values have different colors, each `theme.fg()` call emits ANSI codes. Use `visibleWidth()` on the assembled string (not character counting) to compute padding correctly.
