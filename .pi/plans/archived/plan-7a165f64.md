# Implementation Plan

## Goal
Update the custom footer extension (`~/.pi/agent/extensions/custom-footer.ts`) to decouple colors from theme token names, support direct color values (named ANSI, hex, ANSI 256), split the `model` field into `modelName`/`thinkingMode`/`modelProvider`, and organize per-theme color overrides via a `THEME_COLORS` map.

## Context

**Current state:** The extension uses `FIELD_COLORS` with theme token names (e.g. `"accent"`, `"dim"`) and applies them via `theme.fg(token, text)`.

**Target state:** Replace `FIELD_COLORS` with:
- `DEFAULT_TOKENS` — maps each field to a theme token for defaults (used via `theme.fg()`)
- `THEME_COLORS` — per-theme overrides using direct color values (applied via a custom `applyColor()` helper)

**Key constraint:** `getResolvedThemeColors()` is **NOT** re-exported from `@mariozechner/pi-coding-agent` (verified — only `.` and `./hooks` are in the exports map). The plan must avoid depending on it. Instead, defaults will continue using `theme.fg(token, text)`, and only user overrides will use `applyColor()`.

### 3. Implement the `applyColor()` helper
- **File:** `~/.pi/agent/extensions/custom-footer.ts`
- **Changes:** Add a function that converts any supported color value to an ANSI-styled string:
  ```typescript
  function applyColor(value: string | number, text: string, mode: "truecolor" | "256color"): string {
    const ansi = colorToAnsi(value, mode);
    return `${ansi}${text}\x1b[39m`;
  }
  ```
  Where `colorToAnsi(value, mode)` handles:
  - **Named ANSI colors** → lookup table: `"red"` → `\x1b[31m`, `"brightCyan"` → `\x1b[96m`, etc. (16 standard + bright variants)
  - **Hex strings** (`"#ff8800"`) → truecolor: `\x1b[38;2;r;g;bm` / 256color: downconvert via weighted RGB distance
  - **ANSI 256 numbers** (`208`) → `\x1b[38;5;Nm`
  
  The named color table (16 entries):
  | Name | Code | Name | Code |
  |------|------|------|------|
  | `black` | 30 | `gray`/`grey` | 90 |
  | `red` | 31 | `brightRed` | 91 |
  | `green` | 32 | `brightGreen` | 92 |
  | `yellow` | 33 | `brightYellow` | 93 |
  | `blue` | 34 | `brightBlue` | 94 |
  | `magenta` | 35 | `brightMagenta` | 95 |
  | `cyan` | 36 | `brightCyan` | 96 |
  | `white` | 37 | `brightWhite` | 97 |

  The reset code `\x1b[39m` resets foreground only, matching `theme.fg()` behavior.

- **Acceptance:** `applyColor("red", "hello", "truecolor")` → `\x1b[31mhello\x1b[39m`; `applyColor("#ff0000", "hi", "truecolor")` → `\x1b[38;2;255;0;0mhi\x1b[39m`; `applyColor(208, "hi", "256color")` → `\x1b[38;5;208mhi\x1b[39m`.

### 4. Implement hex-to-256 downconversion
- **File:** `~/.pi/agent/extensions/custom-footer.ts`
- **Changes:** Port the `hexToRgb`, `rgbTo256`, and supporting functions from the theme module (`theme.js` lines ~118-196). These are needed for `applyColor()` when mode is `"256color"` and the user specifies a hex value:
  - `hexToRgb(hex)` → `{ r, g, b }`
  - `rgbTo256(r, g, b)` → ANSI 256 index (uses 6×6×6 cube + 24-step grayscale with weighted Euclidean distance)
  
  Constants: `CUBE_VALUES = [0, 95, 135, 175, 215, 255]`, `GRAY_VALUES = Array.from({ length: 24 }, (_, i) => 8 + i * 10)`
- **Acceptance:** `hexToRgb("#ff8800")` → `{ r: 255, g: 136, b: 0 }`; hex colors produce reasonable 256-color approximations.

### 5. Implement the `colorize()` render helper
- **File:** `~/.pi/agent/extensions/custom-footer.ts`
- **Changes:** Inside the `setFooter` callback (where `theme` is in scope), create a `colorize` function that picks the right coloring strategy per field:
  ```typescript
  // Computed once per render (or on invalidate)
  const overrides = THEME_COLORS[theme.name ?? ""] ?? {};
  const colorMode = theme.getColorMode();

  function colorize(field: keyof FooterColors, text: string): string {
    const override = overrides[field];
    if (override !== undefined) {
      return applyColor(override, text, colorMode);
    }
    return theme.fg(DEFAULT_TOKENS[field], text);
  }
  ```
- **Acceptance:** Fields with user overrides use `applyColor()`; fields without overrides use `theme.fg()` with the default token.

## Risks

1. **`getResolvedThemeColors` is not importable** — The TODO assumes it's available, but it's NOT re-exported from `@mariozechner/pi-coding-agent` (only `.` and `./hooks` are exported). **Mitigation:** Use `theme.fg(DEFAULT_TOKENS[field], text)` for defaults instead of resolving to hex first. User overrides use `applyColor()`. This avoids the import entirely and is actually cleaner — defaults always track the live theme.

2. **`theme.name` may be undefined** — `Theme.name` is typed as `string | undefined`. **Mitigation:** Use `theme.name ?? ""` when looking up `THEME_COLORS`. An empty string key is unlikely to be used by anyone, so no override will be applied when name is unknown.

3. **Named ANSI colors vs theme tokens** — Users might confuse named ANSI colors (e.g., `"red"`) with theme token names (e.g., `"error"`). The `colorToAnsi` function should throw a clear error for unrecognized strings (not silently treat them as hex or variable refs). **Mitigation:** Validate in `colorToAnsi` — if string doesn't start with `#` and isn't in the named color table, throw with a helpful message.

4. **Extension statuses may be pre-styled** — Other extensions pass already-styled ANSI strings via `ctx.ui.setStatus()`. The current code passes them through without re-coloring. **Mitigation:** Continue passing statuses through as-is (don't wrap with `colorize("statuses", ...)`). The `statuses` field in `THEME_COLORS` is available but only applies if we choose to color the status line wrapper — for now, match default footer behavior and pass through.

5. **Width calculations with mixed ANSI** — When some fields use `theme.fg()` (which emits `\x1b[38;...m....\x1b[39m`) and others use `applyColor()` (same format), the ANSI codes are consistent. `visibleWidth()` already handles ANSI escape stripping. **Mitigation:** No action needed — the mixing is safe.

6. **Hex-to-256 fidelity** — The downconversion algorithm (weighted Euclidean distance) is an approximation. Some colors may look off in 256-color terminals. **Mitigation:** Port the exact algorithm from `theme.js` to ensure consistency with how pi itself handles 256-color fallback.
