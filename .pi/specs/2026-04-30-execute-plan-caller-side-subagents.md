# Move Heavy Caller-Side Work in execute-plan to Subagents

Source: TODO-b943cf24

## Goal

Reduce the work the `execute-plan` caller performs personally so it primarily orchestrates, dispatches, and aggregates. Two specific classes of caller-side work move off the caller: (1) integration test execution at baseline capture (Step 7), post-wave runs (Step 12), the Debugger-first flow's success re-test (Step 12 / Step 16), and the final integration regression gate (Step 16) — moved to a new thin `test-runner` subagent that runs the suite and returns the raw set of failing-test identifiers via an artifact-based handoff; and (2) per-task verifier command-evidence collection (Step 11.1) — folded into the existing `verifier` agent as an explicit phase-1 evidence-collection step so each task is verified in a single dispatch instead of an evidencer-then-verifier pair. The caller continues to own all cross-wave state, the integration regression model's reconciliation arithmetic, the user-facing menus and summaries, and the existing dispatch / commit / wave-gate orchestration.

## Context

Today the caller in `agent/skills/execute-plan/SKILL.md` performs four classes of heavyweight work itself:

1. **Integration suite execution** (Step 7 baseline capture, Step 12 post-wave run, Step 12 Debugger-first re-test on success, Step 16 final-gate re-run). The caller runs the test command, applies the Step 7 identifier-extraction contract (suite-native unique per-test identifier verbatim, no normalization, raw-line fallback for un-named failures) to the runner's failure output, and feeds the resulting identifier set into the integration regression model defined in `agent/skills/execute-plan/integration-regression-model.md`. A full integration suite can produce tens of MB of test-runner output.
2. **Per-task verifier command evidence** (Step 11.1). For every command-style `Verify:` recipe in the wave's tasks, the caller runs the command, captures exit code + stdout + stderr, applies the per-stream 200-line / 20 KB truncation rule (first 100 + last 50 with a marker line that records pre-truncation line and byte counts), and assembles `[Evidence for Criterion N]` blocks (header + `command:` + `exit_code:` + fenced `stdout:` + fenced `stderr:`) that get embedded as `{ORCHESTRATOR_COMMAND_EVIDENCE}` in each per-task verifier prompt. The caller then dispatches one verifier per task in parallel (Step 11.2), bounded by `MAX_PARALLEL_HARD_CAP = 8`.
3. **Verifier-visible files + diff-context assembly** (Step 11.2). The caller runs `git status --porcelain`, `git diff HEAD`, and `git diff --no-index /dev/null -- <file>` for each new file, applies the 500-line / 40 KB diff-truncation rule, and computes the deduplicated union of declared `**Files:**`, worker-reported changes, and orchestrator-observed paths. This is comparatively cheap orchestration plumbing and is explicitly out of scope for this spec.
4. **Reconciliation and state tracking** for the integration regression model. The caller owns the cross-wave `baseline_failures` and `deferred_integration_regressions` sets, computes `current_failing` / `still_failing_deferred` / `cleared_deferred` / `new_regressions_after_deferment` per the model's algorithm on every integration run, and renders the user-facing three-section summary that drives Step 12 and Step 16 menus.

Reviewer→refiner handoff in this codebase recently moved from inline `finalMessage` transport to a file-based artifact pattern (`REVIEW_ARTIFACT: <absolute path>` marker; reviewer writes the body to disk with a prompt-supplied first-line provenance; refiner reads from disk and validates via path-equality + non-empty + first-line discipline) — see `.pi/specs/done/2026-04-30-reviewer-writes-review-artifact.md`, `agent/skills/refine-code/refine-code-prompt.md`, and `agent/skills/refine-code/SKILL.md` Step 6. The same family of failure modes (`response missing marker`, `artifact missing or empty`, `path mismatch`, `provenance malformed`) is the canonical shape for artifact-handoff contracts in this repo.

The existing `verifier` agent (`agent/agents/verifier.md`) is judge-only with tools `read, grep, find, ls` and an explicit "Do NOT run shell commands. Do NOT invoke bash, test, or build tools" rule. That rule depends on a hard tool-surface boundary (no `bash`); folding evidence collection in requires extending the tool surface and replacing the tool-surface guarantee with prompt-encoded recipe-verbatim discipline. The existing per-criterion / overall-verdict output format (`[Criterion N] <PASS|FAIL>` plus `VERDICT: <PASS|FAIL>`) is what SKILL.md Step 11.3's parser depends on, and is preserved unchanged.

The `execute-plan/SKILL.md` file is large, and the user has explicitly flagged growth as a concern. This spec aims to keep net SKILL.md size roughly neutral or smaller: procedural detail moves into the new `test-runner` agent definition and the extended `verifier` prompt; SKILL.md sections shrink as caller-side procedural prose is removed.

The original todo also flagged a hybrid re-review investigation item ("when the hybrid re-review fails, the flow continues to perform a final verification review"). Reading `agent/skills/refine-code/refine-code-prompt.md` Iteration 2..N step 7 ("Assess and remediate — same as iteration 1 steps 4-9") confirms the flow already routes Critical/Important hybrid re-review findings back to remediation rather than to Final Verification — Final Verification is reached only when a review pass is clean. The pathology described in the todo is not present in current code; it is treated as out of scope here per the user's invocation context.

## Requirements

- **New `test-runner` agent** (`agent/agents/test-runner.md`). A thin subagent whose job is: receive a test command + working directory + an absolute write path, execute the test command from the working directory, capture full stdout + stderr + exit code, write the full run output to the supplied path with a brief structured header (the executed command, exit code, the extracted set of failing-test identifiers per the Step 7 identifier-extraction contract, and a timestamp), and emit `TEST_RESULT_ARTIFACT: <absolute path>` as the final anchored line of its `finalMessage`. Tools: `bash`, `write`, `read`. The agent does not perform reconciliation, does not consult `baseline_failures` or `deferred_integration_regressions`, and does not classify pass/fail; it returns the raw failing-identifier set and exit code so the caller can apply the integration regression model.
- **`test-runner` artifact-handoff contract** mirrors the reviewer-artifact contract shape: caller pre-computes the absolute write path and supplies it in the task prompt; agent writes there exactly; agent's `finalMessage` ends with one anchored line `TEST_RESULT_ARTIFACT: <absolute path>` and contains no other structured markers; caller extracts the path via the anchored regex `^TEST_RESULT_ARTIFACT: (.+)$` matched on the LAST such line of `finalMessage`; caller validates path-equality against what it supplied + non-empty file + parseable structured header; on any check failure the caller emits a specific reason from a small failure-mode list and stops the path that called it (no inline fallback, no "I'll just run the test myself" path).
- **`test-runner` invocation points.** Replace the caller-side test-command shell in each of: Step 7 baseline capture; Step 12.2 post-wave integration test; Step 12 Debugger-first flow's success re-test (the re-run after a debug pass commits); Step 16 final integration regression gate (initial run and each `(a) Debug failures now` re-entry). The reconciliation algorithm and the user-facing three-section summary stay caller-side at every call site.
- **`test-runner` model tier.** Every `test-runner` dispatch uses the model resolved from `crossProvider.cheap` in `~/.pi/agent/model-tiers.json`, with `cli` resolved through the existing `dispatch[<provider>]` map. If `crossProvider.cheap` is absent from the model matrix, this work adds it before wiring the dispatch path; do not silently fall back to `cheap`, `standard`, or a CLI default.
- **`test-runner` artifact storage** under `.pi/test-runs/<plan-name>/` with phase-distinct, run-distinct filenames (e.g., `baseline.log`, `wave-<N>-attempt-<K>.log`, `final-gate-<seq>.log`). Concrete filename scheme is at the planner's discretion as long as runs of the same phase produce distinct files (so multiple final-gate re-entries do not overwrite each other) and the `<plan-name>` directory is the only one created by `test-runner` work.
- **Cleanup policy.** Step 16's "Move plan to done" sub-step deletes the per-plan `.pi/test-runs/<plan-name>/` directory on successful plan completion (i.e., after the final integration regression gate has passed and the plan file is moved to `.pi/plans/done/`). Any `(c) Stop execution` exit path — from Step 10's wave gate, Step 12's intermediate-wave or final-wave menu, Step 13's failure-handling prompt, Step 15's review max-iterations menu, or Step 16's final-gate menu — leaves the directory in place so the user can inspect the runs after stop.
- **`verifier` extension for evidence collection.** Extend the existing `verifier` agent to perform a phase-1 evidence-collection pass before its existing judge-only phase:
  - Tool surface gains `bash`. Final tool list: `read, grep, find, ls, bash`.
  - Agent prompt is restructured into two explicit phases. **Phase 1 — evidence collection:** for each command-style `Verify:` recipe in the criterion list, run the recipe text byte-equal verbatim from the supplied working directory, capture exit code + stdout + stderr, apply the per-stream 200-line / 20 KB truncation rule, and produce one `[Evidence for Criterion N]` block per command-style criterion in the same byte-for-byte format Step 11.1 produces today. Skip criteria whose `Verify:` recipe is file-inspection or prose-inspection (gaps in numbering remain expected). **Phase 2 — judgment:** for each criterion, return PASS/FAIL using ONLY the evidence captured in phase 1 (for command-style criteria) or the files in `## Verifier-Visible Files` plus any files explicitly named by a recipe (for file-inspection / prose-inspection criteria). Do not re-run any command after capturing its phase-1 output.
  - **Recipe-verbatim discipline** is added as a hard rule in the agent definition: "You MAY run commands ONLY when they exactly match a `Verify:` recipe text byte-equal from the criterion list. You MUST NOT run any other commands. You MUST NOT re-run a command after capturing its output. You MUST NOT add flags, expand variables, or otherwise transform the recipe text." This rule replaces the prior tool-surface guarantee with prompt-encoded discipline.
- **`verify-task-prompt.md` template update.** Replace the `{ORCHESTRATOR_COMMAND_EVIDENCE}` placeholder section with a section that lists the command-style `Verify:` recipes the verifier must execute in phase 1 (one entry per criterion that has a command-style recipe, numbered to match the criterion index in the existing `{ACCEPTANCE_CRITERIA_WITH_VERIFY}` numbering). Keep `{TASK_SPEC}`, `{ACCEPTANCE_CRITERIA_WITH_VERIFY}`, `{MODIFIED_FILES}`, `{DIFF_CONTEXT}`, and `{WORKING_DIR}` unchanged. The output report shape (`## Per-Criterion Verdicts` with `[Criterion N] <PASS|FAIL>` headers, then `## Overall Verdict` with `VERDICT: <PASS|FAIL>`) is unchanged byte-for-byte so SKILL.md Step 11.3's parser, full-coverage requirement, and protocol-error routing are unaffected.
- **Step 11.1 removal and verifier model tier.** SKILL.md Step 11.1 ("Orchestrator collects command evidence") is removed in full. The truncation rule and `[Evidence for Criterion N]` block format relocate into the verifier agent definition / prompt template. Step 11.2 (verifier dispatch) remains, with the prompt-template change above, and dispatches every verifier at the model resolved from `crossProvider.standard` with `cli` resolved through `dispatch[<provider>]`. This replaces the current `standard` default with `capable` upgrade rule; verifier model selection is no longer based on the model tier used by the task under review. Step 11.3 (verifier output parsing) is unchanged.
- **Step 7 / Step 12 / Step 16 updates.** Caller-side test-command shell-execution prose in Step 7, Step 12.2, the Step 12 Debugger-first flow's success re-test (both intermediate-wave and final-wave entries), and Step 16's gate is replaced with `test-runner` dispatch + artifact-readback + reconciliation-against-caller-state. The reconciliation algorithm (per `integration-regression-model.md`), the three-section user-facing summary format, and the Step 12 / Step 16 menus are unchanged.
- **No caller-side debugging or source edits.** All code-change and diagnostic work surfaced by execution, verification, integration tests, or review continues to be delegated to the existing `coder` agent via the existing paths: Step 10's `(s) Split into sub-tasks`, Step 12 / Step 16's Debugger-first flow (which already dispatches a `coder` for the diagnosis pass and an optional second `coder` for targeted remediation). No new caller-side code-edit, fix-application, or auto-debug logic is introduced.
- **Failure-mode lists.** Both new contracts get explicit failure-mode lists with one-line reason strings, mirroring the reviewer-artifact failure-mode list shape. For `test-runner`: `test-runner dispatch unavailable`, `test-runner dispatch failed`, `test-runner response missing TEST_RESULT_ARTIFACT marker`, `test-runner artifact missing or empty at <path>`, `test-runner artifact path mismatch: expected <X>, got <Y>`, `test-runner artifact header malformed at <path>: <specific check>`. For the verifier-with-bash phase-1 path, a parallel set covering: phase-1 evidence-block malformed in agent output, recipe execution refused/skipped by the agent, and discipline violation (e.g., agent ran a command not matching any recipe) — concrete reason strings at the planner's discretion as long as each maps to a single distinct failure mode.

## Constraints

- The Step 7 identifier-extraction contract (suite-native unique per-test identifier verbatim, no normalization, raw-line fallback for un-named failures) is unchanged. The `test-runner` applies it in-agent so the caller receives identifiers byte-equal to those it would have extracted under the old protocol.
- The integration regression model in `agent/skills/execute-plan/integration-regression-model.md` is unchanged. The three tracked sets (`baseline_failures`, `deferred_integration_regressions`, `new_regressions_after_deferment`), the disjointness/transition rules, the reconciliation algorithm, the pass/fail classification, and the user-facing summary format all live with the caller and are unchanged.
- Cross-wave state ownership is unchanged: the caller owns `baseline_failures` (frozen at Step 7) and `deferred_integration_regressions` (mutated only by `(b) Defer integration debugging` and reconciliation). `test-runner` is stateless across calls and has no knowledge of either set.
- The `[Evidence for Criterion N]` block format (header line, `command:`, `exit_code:`, `stdout:` (fenced), `stderr:` (fenced), in that order) is unchanged byte-for-byte. The per-stream 200-line / 20 KB truncation rule and the criterion-numbering rules (gaps for non-command-style criteria are expected and correct) are unchanged.
- The verifier's per-criterion / overall-verdict output format (`[Criterion N] <PASS|FAIL>` plus `VERDICT: <PASS|FAIL>`) is unchanged byte-for-byte. SKILL.md Step 11.3's parser, the full-coverage requirement (`S == {1..K}`), and the protocol-error routing (malformed verifier output → `VERDICT: FAIL` into Step 13 retry loop) are unchanged.
- Step 11.2's `{MODIFIED_FILES}` and `{DIFF_CONTEXT}` assembly remains caller-side. The 500-line / 40 KB diff-truncation rule, the verifier-visible-files three-source union construction (declared `**Files:**` ∪ worker-reported `## Files Changed` ∪ orchestrator-observed `git status` / `git diff HEAD` paths), and the new-file diff handling (`git diff --no-index /dev/null -- <file>`) are unchanged.
- The wave gate (Step 10), the per-task retry budget and shared counter (Step 13), the partial-progress reporting (Step 14), the Step 15 review/refine-code interaction, and the Step 16 branch-completion handoff are unchanged.
- The reviewer-artifact contract in `refine-code` and `refine-plan` is unchanged. `test-runner`'s artifact contract follows the same shape (anchored marker discipline, path-equality, file-non-empty, header validation) but does not share files, directories, or naming with reviewer artifacts.
- Existing dispatch semantics are otherwise unchanged: model-tier resolution per `~/.pi/agent/model-tiers.json`, dispatch CLI mapping via `dispatch[<provider>]`, `MAX_PARALLEL_HARD_CAP = 8` for parallel fan-out, and the explicit `cli` parameter on every orchestration call. This work relies on `crossProvider.cheap` for `test-runner` and `crossProvider.standard` for `verifier`; if `crossProvider.cheap` is not already present, it is added to the model matrix as part of this change. Neither dispatch path silently falls back to a non-cross-provider tier.
- The hybrid re-review investigation item from the original todo is out of scope. Current `agent/skills/refine-code/refine-code-prompt.md` already loops Critical/Important hybrid re-review findings back to remediation; the pathology described in the todo is not present in current code, and this spec does not modify `refine-code` or `refine-plan`.
- Net `agent/skills/execute-plan/SKILL.md` size after this work is roughly neutral or smaller. Procedural detail moved out of SKILL.md (Step 11.1 entirely; the test-command-execution prose in Step 7, Step 12.2, the Step 12 Debugger-first re-test sub-step, and Step 16's gate) is not re-added elsewhere in SKILL.md; it lives in the new agent definition (`agent/agents/test-runner.md`), the modified agent definition (`agent/agents/verifier.md`), and the modified prompt template (`agent/skills/execute-plan/verify-task-prompt.md`).

## Acceptance Criteria

- A new agent definition `agent/agents/test-runner.md` exists with tools `bash, write, read`, a clear "thin runner" charter (no reconciliation, no pass/fail classification, no consultation of `baseline_failures` or `deferred_integration_regressions`), the artifact-write contract (path supplied in prompt; one write per dispatch; structured header containing the executed command, exit code, the failing-test identifier set extracted per the Step 7 identifier-extraction contract, and a timestamp; full raw run output appended), and the `TEST_RESULT_ARTIFACT: <absolute path>` final-line discipline. Every execute-plan dispatch of this agent passes an explicit `model` resolved from `crossProvider.cheap` and explicit `cli` resolved from that model's provider prefix.
- `agent/agents/verifier.md` is updated: tool surface is `read, grep, find, ls, bash`; the agent prompt has explicit phase-1 (evidence collection per command-style recipe) and phase-2 (judgment) sections; a hard rule is present that bash may execute ONLY command-style `Verify:` recipe text byte-equal verbatim and never any other command and never re-run a command after capturing its output; the per-stream 200-line / 20 KB truncation rule and the `[Evidence for Criterion N]` block format are documented in the agent prompt. Every execute-plan verifier dispatch passes an explicit `model` resolved from `crossProvider.standard` and explicit `cli` resolved from that model's provider prefix.
- `agent/skills/execute-plan/verify-task-prompt.md` no longer contains the `{ORCHESTRATOR_COMMAND_EVIDENCE}` placeholder; it has a section listing the command-style `Verify:` recipes the verifier must execute in phase 1, numbered to match the criterion index in `{ACCEPTANCE_CRITERIA_WITH_VERIFY}`; the `## Per-Criterion Verdicts` and `## Overall Verdict` output sections and their format are unchanged byte-for-byte from today.
- `agent/skills/execute-plan/SKILL.md` Step 11.1 ("Orchestrator collects command evidence") is removed in full. The remaining Step 11 content describes only verifier dispatch (Step 11.2) and verdict parsing (Step 11.3).
- `agent/skills/execute-plan/SKILL.md` Step 7, Step 12.2, the Step 12 Debugger-first flow re-test, and Step 16's final-gate runs all dispatch the `test-runner` subagent and read the returned artifact instead of running the test command in-caller; the surrounding identifier-extraction prose is removed from SKILL.md (it lives in the agent now); the reconciliation prose, three-section summary format, and Step 12 / Step 16 menus are unchanged.
- The caller writes nothing to source files and runs no debugging analysis: there is no `Write` / `Edit` of source files in any caller-side step, and no shell command in any caller-side step is more than orchestration plumbing (`git status`, `git diff`, `git add`, `git commit`, `git rev-parse`, `mkdir -p` for the runs directory, `rm -rf` for the cleanup case).
- `test-runner` artifacts are written under `.pi/test-runs/<plan-name>/` with unambiguous per-phase filenames; multiple runs of the same phase (e.g., final-gate re-entries from `(a) Debug failures now`) produce distinct files (no overwrite).
- Step 16's "Move plan to done" sub-step deletes the per-plan `.pi/test-runs/<plan-name>/` directory on successful gate exit. Every `(c) Stop execution` exit path (Step 10, Step 12, Step 13, Step 15, Step 16) leaves the directory in place. SKILL.md spells out both behaviors at the relevant call sites.
- Failure-mode lists for the two new contracts exist and use the same one-line reason-string shape as the reviewer-artifact failure-mode list. For `test-runner` specifically: `test-runner dispatch unavailable`, `test-runner dispatch failed`, `test-runner response missing TEST_RESULT_ARTIFACT marker`, `test-runner artifact missing or empty at <path>`, `test-runner artifact path mismatch: expected <X>, got <Y>`, `test-runner artifact header malformed at <path>: <specific check>`. For the verifier-with-bash phase-1 path: a parallel set with one distinct reason per failure mode (malformed phase-1 evidence block, recipe-execution refused/skipped, discipline violation), each surfaced into Step 11.3's protocol-error routing as `VERDICT: FAIL`.
- Aggregate `agent/skills/execute-plan/SKILL.md` line count after this work is less than or equal to its current line count.
- A smoke run of `execute-plan` against an existing plan confirms: (a) Step 7 baseline capture writes one artifact and the caller reads back identifiers byte-equal to those produced by the legacy in-caller extraction; (b) Step 11 verifier dispatches return correctly populated `[Evidence for Criterion N]` blocks per the per-stream 200-line / 20 KB truncation rule, and the existing per-criterion verdict format parses cleanly in Step 11.3; (c) Step 12.2 post-wave integration writes one artifact per wave, the three-section user-facing summary renders unchanged, and intermediate-wave / final-wave menus appear at the right times; (d) on plan success `.pi/test-runs/<plan-name>/` is removed; on `(c) Stop execution` it remains.

## Non-Goals

- The caller writing or editing source code, or performing debugging analysis itself. All such work continues to delegate to a `coder` subagent via existing paths (Step 10's split-into-sub-tasks, Step 12 / Step 16's Debugger-first flow). Adding caller-side editing helpers, "smart" caller-side fallback fixes, or auto-debug paths is explicitly excluded.
- Changes to the integration regression model, the three tracked sets, the disjointness/transition rules, or the reconciliation algorithm.
- Changes to the verifier's output format (`[Criterion N]` headers, `VERDICT:` line), Step 11.3's parser, the full-coverage requirement, or the protocol-error routing.
- Moving Step 11.2's `{MODIFIED_FILES}` / `{DIFF_CONTEXT}` assembly off the caller. The 500-line / 40 KB diff-truncation rule and the verifier-visible-files three-source union construction stay caller-side.
- Changes to `refine-code` or `refine-plan`. The original todo's hybrid re-review investigation item is treated as stale (current `refine-code-prompt.md` already loops back to remediation when hybrid re-review reports Critical/Important issues) and is excluded from this spec.
- Changes to the wave gate (Step 10), the retry budget and shared counter (Step 13), the partial-progress reporting (Step 14), the final review / refine-code interaction (Step 15), or the branch-completion handoff (Step 16's branch-completion sub-step).
- Changes to dispatch semantics beyond the explicit model-tier choices in this spec: `test-runner` uses `crossProvider.cheap`, and verifier uses `crossProvider.standard`. Model-tier resolution, CLI mapping, `MAX_PARALLEL_HARD_CAP`, and the explicit `cli` parameter convention otherwise remain unchanged.
- Reviewing or restructuring `.pi/specs/`, `.pi/plans/`, `.pi/reviews/`, `.pi/briefs/`, `.pi/todos/`, or any directory other than the new `.pi/test-runs/`.
- A unified caller-side helper or service abstraction wrapping subagent dispatches; this spec uses the existing `subagent_run_serial` / `subagent_run_parallel` orchestration calls directly.
- A tool-level guard against the verifier running non-recipe commands. The recipe-verbatim discipline is prompt-encoded; introducing a sandboxed-bash tool surface is a separate concern that warrants its own spec if it ever proves needed.
