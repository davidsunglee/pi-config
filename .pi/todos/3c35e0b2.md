## Summary

`agent/skills/execute-plan/SKILL.md` contains several phrases that imply plan execution can be paused and resumed in a later session. There is no resume workflow — no checkpoint file, no stale-baseline handling, no re-entry point at wave N — and per TODO-b4e2d71a (closed), one will not be built. The resumability language is therefore misleading and should be removed so the skill's prose matches its actual capabilities.

## Context

TODO-b4e2d71a tracked the design + implementation of plan resumability (checkpoint state, stale-baseline detection, resume entry point). It was closed 2026-04-19 with a decision not to build it: the existing escape hatch (commit partial work, edit spec, regenerate plan, skip done tasks manually) is acceptable in practice and has not produced a concrete pain report.

This todo is the documentation cleanup that follows from that decision.

## Goal

Update `agent/skills/execute-plan/SKILL.md` so that it never promises or implies resume/checkpoint behavior. "Stop execution" options should describe exactly what they actually do: halt the plan, leave committed waves in git history (because git commits persist — they are not a checkpoint scheme), and leave the plan file in place for manual follow-up.

## Scope (committed)

All edits scoped to `agent/skills/execute-plan/SKILL.md` and, if needed, its sibling files under `agent/skills/execute-plan/`.

### Specific phrases to remove or rewrite

Baseline as of 2026-04-19 (line numbers will shift as edits land — use grep, not line numbers, when executing):

1. **Step 9.5 §4 `(x) Stop execution` menu line** — "halt the plan; committed waves are preserved as checkpoints". Drop "as checkpoints"; wave commits exist because that's how git works, not as a resume mechanism.
2. **Step 9.5 §4 `(x) Stop execution` body** — "All prior wave commits are preserved as checkpoints." Same treatment.
3. **Step 9.7 §2 `(x) Stop execution` menu line** — same "preserved as checkpoints" phrasing.
4. **Step 9.7 §3 `(x) Stop execution` body** — "All prior wave commits remain as checkpoints."
5. **Step 11 intermediate-wave `(c) Stop execution` menu line** — "committed waves are preserved as checkpoints".
6. **Step 11 intermediate-wave `(c) Stop execution` body** — "All prior wave commits are preserved as checkpoints. Report partial progress (Step 13). **The user can resume or fix manually.**" The bolded clause is the strongest promise and must go; "fix manually" is fine as a description.
7. **Step 11 final-wave `(c) Stop execution` menu line** — same "preserved as checkpoints" phrasing.
8. **Step 11 final-wave `(c) Stop execution` body** — "Prior wave commits are preserved as checkpoints."
9. **Step 13 heading and body** — "Report partial progress" section: "Leave the plan file in `.pi/plans/` so it can be resumed." Rewrite to something like "Leave the plan file in `.pi/plans/` for the user's reference." Do not imply automated resumption.
10. **Step 13 persistence note** — "If execution resumes in a new session, do NOT reconstruct `deferred_integration_regressions` from the prior partial-progress report — re-run the full integration suite to re-derive the current failing/deferred state fresh." This block assumes a resume flow exists. Delete the whole persistence note, since there is no "new session" resume to warn about.
11. **Step 15 `(c) Stop execution`** — "all committed wave commits are preserved as checkpoints."

The only legitimate use of "checkpoint" that should survive is Step 9.7's title "Wave-level concerns checkpoint" and Step 9's phrase "Do NOT resolve the checkpoint inline" — both of those refer to a gate, not to resumable execution state. ## Out of scope

- Building any actual resume workflow. That was rejected in TODO-b4e2d71a.
- Changing the menu option letters or the set of options. `(x) Stop execution` / `(c) Stop execution` stay exactly as they are — only the surrounding prose changes.
- Changing how commits are produced per wave. The wave-commit behavior (Step 11 §1) is load-bearing for the current workflow and is not affected by this cleanup.
- Touching other skills (generate-plan, define-spec, refine-code).

## Acceptance criteria

1. `grep -nE 'can be resumed|resume or fix manually|if execution resumes' agent/skills/execute-plan/SKILL.md` returns zero matches.
2. `grep -nE 'preserved as checkpoints|remain as checkpoints' agent/skills/execute-plan/SKILL.md` returns zero matches.
3. `grep -nE 'resum' agent/skills/execute-plan/` (recursive) returns zero matches across all files in that directory.
4. The word "checkpoint" in `SKILL.md` appears only in (a) Step 9.7's title, and (b) Step 9's "Do NOT resolve the checkpoint inline" sentence (or close equivalents after any refactor). No occurrence describes resumable execution state.
5. Step 13's body still explains what happens when execution stops early, but it does NOT claim the plan can be resumed. The persistence note about reconstructing `deferred_integration_regressions` is deleted.
6. Every `(x)` or `(c) Stop execution` menu option across Steps 9.5, 9.7, 11 (intermediate + final), and 15 has consistent phrasing that does NOT use "checkpoint" or "resume". Suggested replacement: `halt plan execution; committed waves remain in git history` or simply `halt plan execution`.
7. No behavior change: the set of menu options, retry budgets, gate ordering, integration-test classification, and the wave-commit protocol are all unchanged. Only prose describing what "stop" means gets updated.
