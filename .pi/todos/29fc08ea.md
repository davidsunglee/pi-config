## Pi skill format

- Each skill is a directory under `~/.pi/agent/skills/` containing `SKILL.md`
- YAML frontmatter with `name` (lowercase, hyphens, must match directory name) and `description` (max 1024 chars, specific about when to use)
- Directory name = skill name (e.g., `verification-before-completion/SKILL.md`)
- Additional files (scripts, references) use relative paths from skill directory

## Adaptation guidelines

### Content trimming (~30-40% reduction per skill)

**Keep â€” effective for model compliance:**
- Iron laws and gate functions (strong compliance anchors)
- Rationalization tables (directly prevent model shortcut behaviors â€” models generate these exact justifications before skipping steps)
- Red flags / stop conditions (concrete trigger patterns)
- The process steps themselves (the substance)
- One concrete example per skill (models anchor on examples)
- The assertive tone for iron laws ("NO COMPLETION CLAIMS WITHOUT FRESH VERIFICATION EVIDENCE") â€” hedging language like "should" signals optionality to models

**Trim â€” useful for humans, not models:**
- Human-motivation content ("you'll be replaced", "trust broken", "tired and wanting work over")
- Duplicate explanations of the same rule in different phrasings
- "Advantages vs manual execution" comparisons (convinces humans to adopt, not useful for agents already told to follow)
- Graphviz dot diagrams â†’ convert to mermaid or structured prose (models parse both, mermaid renders in more contexts)

**Keep language-agnostic.** Original TypeScript/npm examples are fine as concrete illustrations â€” don't port to Java/Maven.
