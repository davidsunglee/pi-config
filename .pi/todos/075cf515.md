## Summary

`agent/skills/define-spec/SKILL.md` currently runs the entire spec-design conversation (context survey, scout-brief consumption, one-question-at-a-time Q&A, spec drafting, commit, generate-plan handoff) inside the orchestrator session. Two concrete costs fall out of that:

- The orchestrator's context fills up with codebase files read during exploration and with the full Q&A turn log, before any planning / execution work even starts.
- There is no clean separation between "the skill that kicks off spec design" and "the skill that conducts spec design" — the orchestrator *is* the interviewer.

With `pi-interactive-subagent` in place, the pane backend can host an interactive subagent in its own mux pane where the user answers questions directly. That gives us a way to move the Q&A out of the orchestrator without losing interactivity. The headless backend does not offer a user-input channel, so we still need an in-place fallback.

## Goal

Split `define-spec` into:

1. A thin orchestrator skill that detects mux availability and dispatches a dedicated `spec-designer` subagent (pane backend), or falls back to running the same spec-design procedure in-place (headless backend).
2. A dedicated `spec-designer` agent definition (pi CLI, `session-mode: lineage-only`, no `auto-exit`) whose job is the interactive Q&A + spec write.
3. A single source of truth for the spec-design procedure itself — loaded as a skill both by the subagent (via `skills:` frontmatter) and by the in-place fallback, so mux / no-mux paths cannot drift.

Outputs (`.pi/specs/<date>-<topic>.md`), commit behavior, and the `generate-plan` handoff stay identical regardless of which branch ran.

## Context (why this shape)

- `pi-interactive-subagent` pane backend already supports a user-interactive child: that is exactly how the bundled `planner` agent works today (`session-mode: lineage-only`, no `auto-exit`, pi CLI). `spec-designer` fits the same mold with a different skill body.
- `session-mode: lineage-only` starts the child with a fresh, empty conversation (no parent turns copied in) but records `parentSession` linkage so session discovery / fork-lineage UX still shows the relationship. That keeps the child's context clean while preserving traceability.
- Skills are forwarded on the pi CLI path but **not** on the Claude CLI path in v1 (`pi-interactive-subagent` README, "Skills" section). Claude-CLI support is in scope for this todo because routing Anthropic models (Opus tier) to Claude CLI is the primary way to access those models, so the orchestrator must deliver the procedure body to Claude children some other way — see the dispatch-time inlining constraint below.
- `caller_ping` is unavailable on Claude-CLI initial runs. The child cannot escalate mid-Q&A back to the orchestrator. Acceptable for spec design since the user is present in the pane, but documented as a known limitation.
- TODO-945326a7 is the existing work item on Claude-backed subagent permission posture (`bypassPermissions` default, tool-surface hardening). This todo inherits that backdrop and should not independently relitigate it — if the `spec-designer` needs a tighter tool surface, align with whatever TODO-945326a7 lands on.
- `brainstorming` (superpowers) was considered as the interactive skill to reuse directly, but it prescribes architecture/components/data-flow and terminates by invoking `writing-plans`, writing to `docs/superpowers/specs/…-design.md`. `define-spec` is deliberately intent-only and terminates by offering `generate-plan`, writing to `.pi/specs/`. Contracts differ too much to drop `brainstorming` in as-is. Specific techniques (scope-decomposition check for multi-subsystem requests, spec self-review pass, explicit user-review gate before handoff) are worth lifting into the new procedure, but the procedure stays ours.

## Scope

### New agent

- `agent/agents/spec-designer.md` (or wherever repo-local pi agents live — confirm during execution): `session-mode: lineage-only`, `auto-exit: false`, loads the define-spec procedure as a skill on the pi path. Tool surface scoped to what the Q&A + write + commit flow actually needs (read, write, bash for commit skill, whatever todo tool access is required to resolve todo IDs). Not a spawner — `spawning: false` unless there is a concrete need.
- The agent must work on **both** pi CLI and Claude CLI, since Opus-tier models are accessed via Claude CLI in this project's `model-tiers.json` routing. CLI selection happens per call (orchestrator may override frontmatter) and via model-tiers `dispatch` map.

### New / relocated skill body

- Extract the current `define-spec` Q&A + spec-write procedure into its own skill (e.g. `agent/skills/design-spec-interactive/` — name to be decided during execution). This becomes the single source of truth for the procedure.
- The new skill body handles: resolving todo ID vs freeform input, scout-brief consumption, codebase exploration, Q&A, spec drafting in the current template, commit via the `commit` skill, report.

### Trimmed `define-spec` orchestrator

Three branches, all producing identical artifacts and calling `generate-plan` the same way in Step 6:

- **pi CLI + mux:** dispatch `spec-designer` via `subagent_run_serial` (blocking, `wait: true`). Procedure skill auto-loads via `skills:` frontmatter. Task body carries only the input (todo ID or freeform text) plus minimal control data. Read back the spec path from `finalMessage`.
- **Claude CLI + mux:** dispatch `spec-designer` via `subagent_run_serial` the same way, but because skills are not forwarded to Claude CLI in v1, the orchestrator reads the procedure skill body from disk and **inlines it into the task prompt** at dispatch time. Same canonical skill file, different delivery mechanism. Read back the spec path from `finalMessage`.
- **No mux (any CLI):** load the same skill body inline and run it in the orchestrator session.

CLI selection for the dispatch resolves from the `capable` (or agent-declared) tier in `~/.pi/agent/model-tiers.json` via the `dispatch` map, consistent with `generate-plan`'s Step 2 resolution. Mux-detection branch must not interactively prompt the user — it should decide silently and announce the chosen mode when it starts.

### Lifted techniques from `brainstorming`

Incorporate into the new procedure body:

- Scope-decomposition check before asking detailed questions (if the request spans multiple independent subsystems, surface that instead of refining details).
- Spec self-review pass after writing the file (placeholder scan, internal-consistency check, scope check, ambiguity check — fix inline).
- Explicit user-review gate on the written spec before offering `generate-plan`.

## Non-goals

- Not adopting `superpowers:brainstorming` as the interactive procedure. Contracts differ (architecture prescription, output location, terminal handoff).
- Not changing the spec template, spec output path, or `generate-plan` input contract.
- Not changing `generate-plan` or `execute-plan` as part of this work (other than updating cross-references if the orchestrator's Step 6 handoff wording changes trivially).
- Not building user-input plumbing for the headless backend. Fallback runs in-place in the orchestrator session; that is the acceptable degraded mode.
- Not solving the Claude-backend permission-posture / guardrails gap (that is TODO-945326a7). Align with whatever posture that todo lands on; do not independently weaken or strengthen it.
- Not solving Claude-CLI `caller_ping` parity. If `spec-designer` ever needs to escalate mid-Q&A, that is a `pi-interactive-subagent` feature request, not part of this work.

## Constraints

- Single source of truth: the procedure body lives in exactly one canonical skill file. All three dispatch branches (pi+mux, claude+mux, no-mux) must execute that same body. Delivery mechanism may differ per branch (pi auto-loads via `skills:` frontmatter; Claude inlines into task prompt; no-mux loads inline into orchestrator), but the content must not diverge — any changes to the procedure happen in the one skill file, not per branch.
- For the Claude-CLI branch, the orchestrator's inlining step must read the skill file fresh at dispatch time (not cache a stale copy). If the skill file is not readable, fail cleanly with a clear error rather than dispatching with an empty / truncated procedure.
- Confirm skill auto-load actually fires on the pi path by dispatching a trivial `spec-designer` task and checking the loaded-skill line in the transcript. Confirm the Claude-CLI inlining path by dispatching a trivial `spec-designer` task with `cli: "claude"` and checking that the procedure body landed in the task prompt (and that Claude does **not** emit the `[pi-interactive-subagent] spec-designer: ignoring skills=...` warning for this dispatch — because the orchestrator should not be passing `skills:` on the Claude path at all).
- Spec output path format (`.pi/specs/<date>-<topic>.md`), provenance lines (`Source: TODO-<id>`, `Scout brief: .pi/briefs/TODO-<id>-brief.md`), and commit contract (commit skill, explicit path, no globs) stay identical. `generate-plan`'s provenance-extraction regex in Step 1b must still match without modification.
- Mux-detection branch must not interactively prompt the user — it should decide silently and announce what mode it picked when it starts.
- No regressions in the "user provided a todo ID and a matching `.pi/briefs/TODO-<id>-brief.md` exists" happy path.

## Acceptance criteria

- A `spec-designer` agent exists and can be dispatched by `subagent_run_serial` with a trivial prompt; the pane opens, the user can interact, and the final message returns a spec path.
- `agent/skills/define-spec/SKILL.md` is substantially smaller (target: orchestration + dispatch only; no Q&A prose, no spec template — those live in the new skill).
- The new procedure skill is invokable directly (by the subagent via `skills:` frontmatter **and** by the orchestrator's fallback path inline).
- Running `define-spec` in a mux session with the pi CLI branch starts the pane, does the Q&A there, writes `.pi/specs/<date>-<topic>.md`, commits it, and returns to the orchestrator with the spec path. Offers `generate-plan`.
- Running `define-spec` in a mux session with the Claude CLI branch (i.e. model-tiers routes the dispatch to `cli: "claude"`) produces the same end-to-end flow — pane opens, user interacts, spec is written, committed, orchestrator receives the spec path — with the procedure body inlined into the task prompt. Artifacts are identical to the pi-CLI run on the same inputs.
- Running `define-spec` in a non-mux session (force with `PI_SUBAGENT_MODE=headless` if that is the right probe) performs the same flow in-place and produces byte-identical artifacts (modulo timestamp fields) to the mux run on the same inputs.
- `generate-plan` consumes the resulting `.pi/specs/<date>-<topic>.md` unchanged. Provenance extraction succeeds on both `Source: TODO-<id>` and `Scout brief: .pi/briefs/TODO-<id>-brief.md` lines.
- Scope-decomposition check, spec self-review pass, and explicit user-review gate are present in the new procedure body and are exercised in at least one dry-run.

## Open questions

- Exact mux-availability probe the orchestrator should use (env var, helper tool, or dispatch-and-detect). `PI_SUBAGENT_MODE` + backend inspection is the likely answer but needs confirmation against `pi-interactive-subagent` internals.
- Where repo-local pi agents live in this project (`agent/agents/`? `.pi/agents/`? both?). Confirm during execution and match existing convention.
- Whether the new procedure skill should live under `agent/skills/` as a sibling of `define-spec` or nested inside it. Sibling favors reuse (other orchestrators could load the same procedure); nested favors clear ownership. Leaning sibling.
- Whether `spec-designer` should get `spawning: false`. It probably should — the procedure reads, writes, and commits, but does not need to dispatch further subagents.
- Whether the orchestrator should still run Step 3's "general survey" itself (as a fast pre-dispatch context check) or hand 100% of exploration to the subagent. Handing it all over is cleaner for context isolation; doing a light pre-dispatch survey helps the orchestrator pick a reasonable mux / no-mux message. Leaning "all over to subagent" but worth confirming.
- How the no-mux fallback should surface to the user — a one-line notice ("no multiplexer detected, running spec design in this session") before proceeding, or silent. Leaning notice.
- Where in the dispatch step to perform the Claude-CLI inlining. Options: (a) inside `define-spec` orchestrator directly, (b) as a generic utility that other skills could reuse (since `generate-plan` and `execute-plan` may hit the same skill-forwarding gap as they move more work to Claude-backed Opus tiers), (c) as a feature request upstream on `pi-interactive-subagent` to forward skills to Claude. (a) is fastest; (b) is the right long-term shape if the pattern repeats; (c) removes the need entirely but is out of our control. Leaning (a) now, with an explicit note in the orchestrator pointing at (b) if the pattern shows up elsewhere.
- Which model tier `spec-designer` should request. The Q&A is judgment-heavy and benefits from Opus, but the spec-write step is mostly structured templating that a cheaper tier could handle. Single-tier is simpler; split-tier would require splitting the subagent. Leaning single-tier Opus (Claude CLI) until a cost problem shows up.
