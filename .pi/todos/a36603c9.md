## Summary

The workflow skills (execute-plan, generate-plan, refine-code) contain duplicated patterns and mechanical operations that could be extracted into reusable primitives and helper tools. The goal is to slim down the skills while preserving their flexibility, mutability, and judgment-handling — not to replace them with code.

Full analysis: [docs/analysis/2026-04-14-skill-decomposition-and-code-boundary.md](/docs/analysis/2026-04-14-skill-decomposition-and-code-boundary.md)

Reinforced by the 2026-04-17 workflow evaluation (see parent `7e1fa3d2`), which re-surfaced the same mechanical-duplication concern and added `extract-provenance` to the helper-tool set (Task 10 below).

## Two categories of extraction

### 1. Converge primitive skill (discipline extraction)

The evaluate-fix-reevaluate loop appears in three skills today:

| Skill | Evaluator | Fixer | Budget |
|-------|-----------|-------|--------|
| refine-code | code-reviewer | coder | 3 |
| generate-plan | plan-reviewer | planner (edit) | 1 |
| execute-plan retries | acceptance criteria | re-dispatch coder | 3 |

Extract a `converge` skill that encodes the loop discipline: never skip evaluation, never exceed budget without returning to caller, each iteration must progress or escalate.

**Critical constraint:** Converge must be an inline protocol (mixin), NOT a dispatched agent. The agent already orchestrating follows the converge rules. No additional dispatch layer. The skill must be explicit about this to prevent future misinterpretation as "dispatch a converge subagent."

### 2. Helper tools (mechanical extraction)

Small TypeScript tools for operations where the LLM adds no value and can introduce errors:

| Tool | What it does | Consumers |
|------|-------------|-----------|
| `resolve-tiers` | model-tiers.json → model string + dispatch target | 4 skills |
| `fill-template` | Reads template file, substitutes placeholders literally | execute-plan |
| `build-waves` | Parses plan dependencies, returns topologically sorted waves | execute-plan |
| `test-baseline` | Runs test command, diffs against stored baseline | execute-plan |
| `extract-provenance` | Bounded preamble read of a spec/plan file, returns `{source_todo, scout_brief}` | generate-plan, execute-plan |

Estimated: 200-300 lines of TypeScript total. Each tool is a pure function — no orchestration state.

### Also: standardize worker status protocol

Extract the DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED protocol into a shared reference document (`agent/protocols/worker-status.md`) that both `coder.md` and consuming skills point to.

## What NOT to extract

- Wave construction — single consumer (execute-plan)
- Settings confirmation UI — single consumer, too thin for a skill
- Git range management — used in multiple skills but too varied to share
- Baseline test capture — better as a tool than a skill

Principle: extract when there are two real consumers AND the shared pattern is about discipline (doing things you might skip), not just deduplication.

### Task 10: Implement `extract-provenance` tool

- TypeScript function: bounded preamble read (stops at first `## ` heading or ~40 lines) of a spec or plan file; returns `{ source_todo, scout_brief }` parsed from `Source: TODO-<id>` and `Scout brief: <path>` lines
- Expose as a tool
- Update generate-plan Step 1 to use the tool for spec-file inputs
- Update execute-plan Step 15 to use the tool to find `**Source:** TODO-<id>` in the plan for todo auto-close
- Eliminates the current implicit contract where planner must hand-write the `Source:` line in a location execute-plan can grep

## Completion criteria

- `converge` primitive exists and is used by refine-code, generate-plan, and execute-plan
- No converge usage adds a dispatch layer — all loops run inline in the existing orchestrating agent
- Five helper tools exist and replace English algorithm descriptions in skills
- Worker status protocol has a single source of truth referenced by all consumers
- Execute-plan drops from ~460 to ~350 lines
- All existing workflow behavior is preserved (no functional changes)
- Mechanical operations (tier resolution, template filling, wave sorting, test diffing) are correct by construction
