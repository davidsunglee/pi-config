## Problem

During execute-plan Step 15 for eval harness v3, the `refine-code` loop produced review files marked:

`**Reviewer:** inline (claude-sonnet-4-6)`

No `gpt-5.5` / `openai-codex` review was produced, even though the model matrix had:

```json
"crossProvider": {
  "standard": "openai-codex/gpt-5.4",
  "capable": "openai-codex/gpt-5.5"
},
"dispatch": {
  "anthropic": "claude",
  "openai-codex": "pi"
}
```

Root cause: `refine-code/SKILL.md` dispatches the `code-refiner` coordinator with plain `standard`, which resolves to `anthropic/claude-sonnet-4-6` and `cli=claude`. Claude-backed coordinator sessions cannot use pi orchestration tools to dispatch `code-reviewer` / `coder` subagents, so the coordinator improvised with an inline Claude review instead of blocking.

`refine-plan/SKILL.md` already has the newer pattern: dispatch the coordinator (`plan-refiner`) with `crossProvider.standard`, resolve `cli` from the dispatch map (normally `pi`), and warn if coordinator CLI is not `pi`.

## Desired fix

1. Update `refine-code/SKILL.md` so the `code-refiner` coordinator uses `crossProvider.standard` (not `standard`) and resolves `cli` via the model matrix dispatch map.
2. Add the same coordinator-CLI guard as `refine-plan`: the coordinator must run under `pi` so it has orchestration tools (`subagent_run_serial`). If the resolved CLI is not `pi`, warn or stop rather than silently dispatching to Claude.
3. Update fallback chain for coordinator dispatch to retry `crossProvider.capable`, then `capable`, re-resolving CLI each time.
4. Harden `refine-code-prompt.md` / `code-refiner` instructions: if subagent dispatch tools are unavailable or dispatch fails, return a failed/BLOCKED status; never perform inline review as a substitute.
5. Consider updating `code-refiner` and `plan-refiner` agent definitions to explicitly include subagent orchestration tools if pi's tool restrictions require it.
6. Add a postcondition/provenance check: review files from refine-code should identify the reviewer model(s), and first/final review should be from `crossProvider.capable` (e.g. `openai-codex/gpt-5.5`) rather than `inline`.

## Question answered

`execute-plan` likely does not need direct changes for this specific bug: it invokes the `refine-code` skill at Step 15 and does not dispatch `code-refiner` itself. Once `refine-code` dispatches the coordinator correctly through `pi`, execute-plan should inherit the fix. A lightweight execute-plan provenance check after Step 15 could still be useful as defense-in-depth.

## Additional finding from corrective run

A corrective `refine-code` run was dispatched as requested with:

- coordinator model: `openai-codex/gpt-5.4`
- coordinator cli: `pi`
- review output base: `.pi/reviews/2026-04-28-2026-04-28-eval-harness-design-v3-code-review-gpt54`

Result: `STATUS: max_iterations_reached` / blocked. The coordinator wrote review artifacts stating that `subagent_run_serial` was unavailable in its session and, per the corrective prompt, did not perform an inline review.

This means dispatching the coordinator through `pi` is necessary but not sufficient in the current configuration: the `code-refiner` agent definition's explicit tool list (`read, write, edit, grep, find, ls, bash`) appears to restrict away orchestration tools. The fix should also update `~/.pi/agent/agents/code-refiner.md` (and likely `plan-refiner.md`) to include `subagent_run_serial` / relevant orchestration tools, or remove/adjust the restrictive tool list if pi uses it as an allowlist.
