## Summary
Broader follow-up: prevent or detect cases where a coordinator dispatched on one backend silently degrades nested subagent model requests (for example, coercing `openai-codex/gpt-5.4` into `opus`).

## Problem
The narrow `refine-code` issue exposed a more general runtime hazard: nested subagent calls may not preserve concrete provider/model ids when they run through a backend with a reduced model vocabulary. In the observed run, the prompt requested cross-provider review models from the model matrix, but the nested runtime accepted only `sonnet`/`opus`/`haiku` and silently substituted an Anthropic tier model.

## Goals
Investigate a generalized hardening approach so coordinators and orchestration skills do not silently violate their model matrix or dispatch contract.

## Potential directions
- Fail fast when a dispatch backend cannot honor a requested concrete model id.
- Add capability validation before nested dispatch.
- Ensure nested subagent calls preserve explicit provider/model ids across backends.
- Improve runtime metadata/artifacts so the actually used model and backend are unambiguous.
- Add regression tests for cross-provider nested review/remediation flows.
- Audit other skills/coordinators that may have the same failure mode when launched via `claude`.

## Candidate scope areas
- `~/Code/pi-subagent`
- skill dispatch guidance in `~/.pi/agent/skills/`
- any extensions/runtime layers that translate nested agent calls differently across `pi` vs `claude`

## Notes
This is the broader infrastructure problem, not the immediate `refine-code` workaround. It may deserve a proper spec/plan once someone decides the intended system-wide behavior.
