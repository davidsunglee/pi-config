## 1. Create a dedicated `plan-reviewer` agent

### Problem

The plan review step in generate-plan (Step 3.5) dispatches a `plan-executor` agent with the `plan-reviewer.md` template. The plan-executor system prompt is oriented toward executing tasks (status codes, file output conventions, escalation rules) — none of which are relevant to reviewing a plan for structural issues.

The existing `plan-reviewer.md` agent file at `agent/agents/plan-reviewer.md` is only 8 lines and delegates entirely to the task prompt, making the agent definition a no-op wrapper.

### Tasks

1. Create a dedicated `plan-reviewer` agent with a system prompt tuned for plan review
2. Have it check plan structure, spec coverage, dependencies, task sizing, and review quality
3. Standardize output as `Approved` or `Issues Found`, with `Error` / `Warning` / `Suggestion` severities
4. Replace the current `plan-executor` + `plan-reviewer.md` wrapper approach in generate-plan

**Related:** This parallels TODO-efc9f70b, where a dedicated `code-reviewer` agent replaces plan-executor-as-reviewer for final code review.

## 2. Replace full regeneration with surgical plan revision

### Problem

In generate-plan Step 3.5, when the reviewer finds errors and the user chooses "Re-generate," the plan-generator runs from scratch with reviewer findings appended. A plan with 8 good tasks and 1 bad task gets fully regenerated, potentially losing good structure or introducing new issues. There is no versioning — the original plan is overwritten.

### Tasks

1. Add a `plan-editor` mode (new agent or extended generator capability) that receives the existing plan plus reviewer findings and produces targeted edits rather than a full rewrite
2. Preserve plan versions by saving the original as `<plan>.v1.md` (or similar) before applying edits
3. Use full regeneration only as a fallback when surgical editing cannot resolve the issues
4. Consider having the reviewer output concrete, actionable diffs rather than prose findings to make surgical edits easier

## 3. Migrate generate-plan to the model matrix

### Problem

Generate-plan currently reads `modelTiers` from `~/.pi/agent/settings.json`. As part of the model matrix extraction (moving model config to `~/.pi/agent/models.json`), generate-plan should be updated to read from the new location.

### Tasks

1. Update Step 3 in `generate-plan/SKILL.md` to read plan-generator model selection from `~/.pi/agent/models.json`
2. Update Step 3.5 to read plan-reviewer model selection from `~/.pi/agent/models.json`
3. Remove any remaining `modelTiers` references in the skill that still point to `settings.json`

## 4. Incorporate `writing-plans` skill

### Problem

Generate-plan owns plan generation behavior, but reusable guidance about how plans should be written can drift if it lives separately or is not consistently invoked. The Superpowers `writing-plans` skill is a concrete reference point for shared planning guidance:

- https://github.com/obra/superpowers/tree/main/skills/writing-plans

Generate-plan should reuse or adapt that shared guidance instead of duplicating plan-writing rules in multiple places.

### Tasks

1. Review Superpowers's `writing-plans` skill and identify the authoritative guidance that should shape plan generation output
2. Decide whether generate-plan should invoke an equivalent local skill directly or inline a maintained subset of its guidance
3. Refactor generate-plan prompts so plan structure and writing standards come from `writing-plans` rather than duplicated prompt text
4. Ensure the reviewer and any future `plan-editor` use the same conventions so generated, reviewed, and revised plans stay consistent
5. Document any intentional deviations from the Superpowers skill so the local behavior stays understandable and maintainable
