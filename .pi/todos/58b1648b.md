## Summary

Follow up on the completed planner-dispatch handoff optimization (`TODO-d718bad4`) by reducing orchestrator context pollution in the later `generate-plan` review/edit loop as well.

The first slice changed only `generate-plan -> planner`, so file inputs are handed off by path instead of embedding full markdown contents. This follow-up should now address the remaining planning-stage passes that can still re-inline large artifacts:

- `generate-plan -> plan-reviewer`
- planner edit pass used after plan review findings

## Motivation

Even with the initial planner dispatch now path-based, the review/edit loop can still repeatedly load large markdown artifacts into the orchestrator context:

- the generated plan contents
- the original spec / design doc contents
- any scout brief carried forward for review context
- review findings plus surrounding context

That means the orchestrator still pays much of the context cost during iterative planning, especially for large specs and plans.

## Goal

Design and, if warranted, implement a file-based handoff contract for the `generate-plan` review/edit loop that preserves correctness, review quality, and approval semantics while reducing repeated markdown inlining in the orchestrator.

## Locked decisions

These are intentional policy decisions, not open questions for the planner:

1. **Freshness / snapshot policy:** Follow the same policy used in the completed planner handoff work. Prefer live file reads by path for durable artifacts. Do **not** introduce new hashing, fingerprinting, or heavier freshness machinery unless analysis finds a concrete need.
2. **Review findings:** Keep review findings inline by default. Treat them as small, ephemeral control data rather than durable artifacts.
3. **How far to push path-based handoff:** Prefer a narrow contract that moves only the large durable artifacts by path.
4. **Todo / freeform runs:** Do **not** introduce temp artifact files just to make todo/freeform review-edit passes path-based. This optimization is primarily for file-based planning runs.
5. **Failure behavior:** Stay consistent with the planner handoff slice — missing required plan/source artifacts should fail the workflow; missing scout briefs should warn and continue.
6. **Readability / auditability bias:** Prefer durable artifacts by reference. Large inlined plans/specs hurt readability more than they help. Small ephemeral guidance may remain inline.

## Preferred direction

Bias toward moving **large durable artifacts** to path-based handoff unless there is a concrete reason to preserve inline snapshot semantics.

In practice, the expected default shape is:

- `plan-reviewer` reads the current plan from disk via path
- `plan-reviewer` reads the original task artifact from disk via path when the planning input was file-based
- planner edit pass reads the current plan from disk via path
- review findings remain inline
- minimal provenance / safety metadata stays inline

If any pass should remain inline, the plan should explain exactly why.

## Decisions to make during planning

The planner / implementer should still resolve the concrete contract details needed to realize the policy above:

1. Exactly what inline provenance metadata should accompany path-based review handoff
2. Whether the edit pass needs any special inline metadata beyond the review findings and output path
3. Whether `plan-reviewer` and/or `planner` need explicit contract updates stating that they must read handed-off files from disk
4. Whether there are any narrow, justified exceptions where inline snapshot semantics should be preserved despite the default policy

## Scope

In scope:

- `agent/skills/generate-plan/SKILL.md`
- `agent/skills/generate-plan/review-plan-prompt.md`
- `agent/skills/generate-plan/edit-plan-prompt.md`
- `agent/agents/plan-reviewer.md` if the contract needs to be made explicit there
- `agent/agents/planner.md` if edit-mode contract wording needs to change
- smoke verification and prompt/transcript inspection for the affected handoffs

Out of scope:

- `execute-plan`
- `execute-plan -> coder`
- unrelated prompt cleanup or workflow redesign
- generic artifact fingerprinting infrastructure beyond what this handoff needs

## Constraints

- Preserve review quality and approval semantics.
- Avoid widening worker scope unnecessarily.
- Prefer plain filesystem-path handoff over new transport mechanisms.
- Keep the change narrowly focused on the review/edit loop.
- Any freshness / provenance guardrails should be lightweight unless stronger protection is clearly necessary.

## Deliverable

Produce either:

1. an implementation that moves the review/edit loop to the chosen file-based contract, or
2. a narrowly justified recommendation explaining why one or more passes should remain inline

The expected default is implementation unless the analysis finds a real semantic or quality regression risk.

## Acceptance criteria

- There is an explicit decision for `generate-plan -> plan-reviewer` handoff.
- There is an explicit decision for the planner edit-pass handoff.
- The final contract clearly states what is passed **by path** vs **inline** for each pass.
- The implementation remains consistent with the planner-only handoff policy for live-file reads, lightweight provenance, and missing-scout-brief handling.
- Any required provenance or freshness guardrails are specified.
- If implementation is chosen, prompts / agent contracts are updated so workers are explicitly told what to read from disk.
- Verification demonstrates the intended behavior with smoke tests and prompt/transcript inspection, including:
  - large plan/source artifacts are no longer unnecessarily inlined into the orchestrator for the updated passes
  - review findings remain inline and that behavior is explicit and verified
  - todo/freeform runs do not gain unnecessary temp-artifact machinery
  - missing scout brief behavior still warns and continues where applicable
  - missing required plan/source artifacts fail clearly
  - review quality / approval flow remains intact
