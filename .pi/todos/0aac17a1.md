## Summary

The current workflow is optimized for medium-to-large planned features: generate plan → review plan → execute in dependency-ordered waves → iterative code refinement → close out. This is the right level of rigor for multi-file, multi-concern work, but it's overkill for the 80% of development tasks that are small: a bugfix, a rename, adding a field, tweaking a config, a small refactor.

Design a lightweight "fastlane" workflow that lets the agent go from task description to committed code without the overhead of plan generation, wave decomposition, or multi-agent orchestration — while still preserving the quality guardrails that matter (TDD, verification, review).

## Motivation

- The full pipeline dispatches 15-25 subagents for a medium feature. A 20-line bugfix doesn't need a planner, plan reviewer, wave scheduler, or code-refiner orchestrator.
- Fresh-context subagent dispatches are expensive. Each one re-reads system prompts, prompt templates, and source files from scratch. Skipping unnecessary dispatches saves real token cost.
- In practice, small tasks are probably already done ad hoc (invoking individual skills piecemeal). A structured fastlane would make this a first-class path rather than an implicit workaround.
- The full pipeline's wave model assumes clean task decomposition. Small tasks rarely decompose — they're one unit of work.

## Design questions to resolve

### 1. Entry criteria: what qualifies as "small"?

How does the agent (or user) decide whether a task should take the fastlane or the full pipeline? Options:

- **User-declared:** User explicitly says "fastlane" or a similar trigger. Simple but requires user judgment.
- **Heuristic-based:** The agent estimates scope from the task description (file count, concern count, estimated diff size) and suggests fastlane if below a threshold. Risks misjudging scope.
- **Default-fastlane with escape hatch:** Start fastlane, escalate to full pipeline if the task turns out to be bigger than expected. Matches how work actually unfolds, but needs a clean escalation path.

### 2. Which guardrails survive?

The full pipeline's value comes from its guardrails. Which ones should the fastlane preserve?

- **TDD** — Probably yes, scaled to scope. Red-green-refactor for a bugfix is lightweight and high-value.
- **Verification before completion** — Yes, this is cheap and prevents false success claims.
- **Code review** — Maybe. A single cross-provider review pass (without the iterative refine loop) could be worth the token cost for anything non-trivial. Or: review only if the diff exceeds a threshold.
- **Git isolation** — Probably optional. A worktree for a one-file change is overhead. A simple branch might suffice, or just commit to the current branch.
- **Commit discipline** — Yes, Conventional Commits format should apply regardless of path.

### 3. Single-agent or minimal-dispatch?

The full pipeline uses 5 specialized agents. The fastlane could:

- **Single-agent:** The host agent does everything inline — reads code, writes the fix, runs tests, commits. Zero subagent dispatches. Cheapest, but loses the independence benefit of fresh-context review.
- **Minimal-dispatch:** The host agent does the implementation, then dispatches one reviewer for a sanity check. One subagent dispatch. Balances cost and quality.
- **Compact pipeline:** Coder subagent + reviewer subagent, no planner or refiner. Two dispatches. Still lighter than the full pipeline but preserves separation of concerns.

### 4. How does this integrate with existing skills?

The fastlane could be:

- **A new skill** (`agent/skills/fastlane/`) with its own SKILL.md — parallel to `execute-plan` as an alternative execution path.
- **A mode within `execute-plan`** — the existing skill detects small scope and short-circuits the wave machinery.
- **An implicit default** — no explicit skill invocation needed; the agent just does the work and invokes `commit`, `verification-before-completion`, and optionally `requesting-code-review` as standalone skills.

### 5. Escalation path

If the fastlane discovers the task is bigger than expected (e.g., touches more files than anticipated, requires interface changes across modules), how does it escalate?

- Stop and suggest generating a plan?
- Automatically transition to the full pipeline?
- Just warn the user and continue?
