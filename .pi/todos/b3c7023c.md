## Context

The review extension (`review.ts`) had a "review sessions" feature — fork context into a fresh session for review, then `/end-review` to return. The concept is interesting but was never used in practice before the extension was removed (see TODO-51cb7cf7).

## Questions to answer
- Is there a natural point in the workflow where session forking adds value? (e.g., forking before a code review so the reviewer has clean context, forking to explore a debugging tangent without polluting the main session)
- Could this be a lightweight skill or small extension rather than the full review session machinery?
- Does pi's built-in `context: "fork"` subagent option already cover this need?

---

## Research: Two mechanisms, different purposes

**`/tree`** — in-place branching within a single session file. Navigate back to any earlier point and continue from there, creating a new branch. The old branch is preserved (optionally summarized). Like git branches but for conversation.

**`/fork`** — creates a *new session file* by copying history up to a selected point. The original session is untouched. A fresh session that starts with the context of an old one.

**`context: "fork"` on subagents** — when dispatching a subagent, `context: "fork"` gives it a copy of the parent session's context. The subagent works in an isolated session but starts with everything the parent knew.

## Research: Practical use cases for manual workflow

1. **"Let me try a different approach"** — Halfway through implementing something, explore an alternative. `/tree` back to before the implementation started, try approach B. If it fails, `/tree` back and approach A is still intact.

2. **Debugging tangents** — Hit a bug and start investigating. The investigation pulls in a lot of context (stack traces, reading files, dead ends). `/tree` back to before the tangent so the debugging noise doesn't pollute the main session's context window.

3. **Pre-review checkpoint** — Before asking for a code review, `/fork` the session. The review discussion happens in the forked session. Main session stays clean for continuing development after.

4. **Subagent with full context** — Working on a feature and want a subagent to handle a subtask (e.g., write tests). `context: "fork"` means the subagent already knows what you've been doing — no need to re-explain the feature.

5. **Spec exploration** — Discussing a design with pi, then branch off to prototype two competing approaches. Fork or branch, try one, go back, try the other, compare.

6. **Recovering from a bad compaction** — Compaction summarized away something important. `/tree` back to before the compaction and continue from there with the full context still available.

## Research: Honest assessment

These are real capabilities, but:

- **Most sessions are short-lived** — if you tend to start fresh sessions per task, branching has less value.
- **Git already handles "try two approaches"** — worktrees and branches may be more natural than session branches for code exploration.
- **`/tree` is already built in** — you don't need a skill to use it. The question is whether there's a *structured workflow pattern* worth codifying.

## Investigation: Does the automated workflow benefit?

**Question:** The generate-plan / execute-plan / other skills that automate workflow spawn subagents for most things (generate a plan, review the plan, re-generate post-review, execute tasks in a wave, review tasks, perform final review, address review issues). Do those subagents fork context, or do they start fresh and use todo / plan / review / etc artifacts to convey information?

**Answer: Everything starts fresh. No skill uses `context: "fork"`.**

Every subagent dispatch uses the default `context: "fresh"`:

| Step | Subagent | Context | How it gets information |
|------|----------|---------|------------------------|
| Generate plan | `plan-generator` | fresh | Task prompt includes full todo body / spec text |
| Review plan | `plan-executor` | fresh | Filled `plan-reviewer.md` template with plan contents + original spec |
| Execute wave tasks | `plan-executor` | fresh | Filled `implementer-prompt.md` template with task spec, plan goal, prior wave summaries |
| Spec review | `plan-executor` | fresh | Filled `spec-reviewer.md` template with task spec + implementer report |
| Final code review | `plan-executor` | fresh | Filled `code-reviewer.md` template with plan contents + git diff range |

The requesting-code-review skill is explicit: *"Use a capable-tier model in a fresh context — the reviewer must see the code without bias from the generation process."*

### The communication medium is files, not context

The entire pipeline is artifact-driven:
- **Plans** (`.pi/plans/*.md`) carry the spec between generation → review → execution
- **Prompt templates** (`implementer-prompt.md`, `spec-reviewer.md`, etc.) are filled with the relevant slice of information
- **Git diffs** carry the code changes to the final reviewer
- **Todos** track lifecycle state

No subagent inherits conversation history. Each one gets a self-contained prompt with exactly the context it needs.

### Why forking doesn't help the automated workflow

Session forking would be useful for exactly **none** of the automated workflow steps. The artifact-based approach is better because:

1. **Focus** — each subagent sees only what's relevant, not 50k tokens of prior conversation
2. **Independence** — the reviewer can't be biased by watching the generation process
3. **Resumability** — if a step fails, you re-run with the same artifact, not a fragile session branch
4. **Debuggability** — you can read the plan file, the review file, the prompt template — all are inspectable artifacts

Forking would arguably make things *worse* — a forked reviewer would carry the generation session's reasoning, which is the opposite of an independent review.

---

## Conclusion

**Closed — no action needed.** The automated workflow is deliberately artifact-driven and doesn't benefit from session forking. The manual use cases (debugging tangents, exploratory branching) are already served by `/tree` and `/fork` built into pi — no skill or extension needed.
