## Context

The review extension (`review.ts`) had a "review sessions" feature — fork context into a fresh session for review, then `/end-review` to return. The concept is interesting but was never used in practice before the extension was removed (see TODO-51cb7cf7).

## Questions to answer
- Is there a natural point in the workflow where session forking adds value? (e.g., forking before a code review so the reviewer has clean context, forking to explore a debugging tangent without polluting the main session)
- Could this be a lightweight skill or small extension rather than the full review session machinery?
- Does pi's built-in `context: "fork"` subagent option already cover this need?
