## Goal

Create a skill at `.pi/agent/skills/generate-plan/SKILL.md` that dispatches the `plan-generator` subagent to analyze the codebase and produce a structured plan file in `.pi/plans/yyyy-MM-dd-<short-description>.md`.

## Architecture Decision

Planning runs on a **`plan-generator` subagent** (not the main agent) because:
- Deep codebase analysis is context-intensive — a fresh context window gives maximum room
- Simple handoff: main agent passes a todo ID or spec path, subagent does the heavy lifting
- Can run async while the user continues other work

The agent definition lives at `~/.pi/agent/agents/plan-generator.md` (already created).

## What the Skill Does

The skill is instructions for the **main agent** on how to:
1. Accept input (todo ID, file path, or freeform description)
2. Dispatch the `plan-generator` subagent via the `subagent` tool
3. Pass the input as the task prompt
4. Report the result back to the user

The skill does NOT define the plan format — that's encoded in the `plan-generator` agent's system prompt.

## Skill Behavior

### Input sources
- A todo ID (e.g., `TODO-7ef7d441`)
- A file path to an existing spec, RFC, or design doc
- A freeform task description

### Dispatch
```
subagent { agent: "plan-generator", task: "<input context>" }
```

The task prompt should include:
- The todo body, spec content, or freeform description
- The current working directory context (repo name, etc.)

### Output
The subagent writes the plan file directly. The skill tells the user where it was saved and suggests using `execute-plan` to run it.

### Async option
The skill should mention that the user can run this async if desired (subagent supports it).

## Plan file location
- Plans live in `.pi/plans/` with filenames like `yyyy-MM-dd-<short-description>.md`
- The `plan-generator` agent handles creating the directory and writing the file

## Plan format
Defined in the `plan-generator` agent system prompt (`~/.pi/agent/agents/plan-generator.md`). Includes:
1. Header — goal, architecture summary, tech stack
2. File structure — files to create/modify with responsibilities
3. Numbered tasks with Files, Steps, Acceptance criteria, Model recommendation
4. Dependencies section
5. Risk assessment
