## Summary

Today the workflow is strictly forward: spec → plan → execute, then done. Execute-plan's skill prose claims plans can be resumed, but no workflow exists — no checkpoint file, no stale-baseline handling, no re-entry point at wave N. Partial failures drop users back to manual git.

## Scope note: revision deferred

This todo originally bundled resumability with a spec-driven revision primitive (amend spec → plan-delta → regen affected tasks → resume). Revision is deferred — revisit reactively if a concrete pain case emerges. Rationale:

- Revision piles semantic spec-diff, plan-delta semantics (unaffected / invalidated / new / removed tasks), per-task revert strategy, and an amend-spec surface on top of already-complex workflow skills. Each is a hard judgment call.
- A workable escape hatch exists today: commit partial work, edit spec, regenerate plan, skip done tasks manually. Ugly but not a dead end.
- Matches the precedent set by `bf68a11b` item 1 (validate-plan): don't build speculative complexity before seeing the failure case.

Resume stays in scope because it's bounded (finite state, one new file, one decision point in execute-plan) and broadly useful on its own.

## Phase 1 — Design spike (produces a spec, not code)

Open questions the spike must answer:

- **State shape.** What must persist to support resume? Candidates: completed waves, per-wave HEAD SHA, baseline SHA, worktree path, plan-version hash, last-known-good task outputs.
- **State location.** `.pi/plans/<plan>.state.json`? Co-located with plan? Per-worktree?
- **Plan lifecycle interaction.** Plan moves to `.pi/plans/done/` on completion. Where does state live mid-flight? Before resume? After resume?
- **Stale-baseline detection.** Repo drifts between runs. When is a stored baseline invalid? How does execute-plan detect and handle that?
- **Entry point.** On `execute-plan <plan>`: if state file exists, offer resume-from-last-successful-wave vs. fresh run. What does the user see?

Spike output: a design doc in `docs/analysis/` and a spec in `.pi/specs/` that the primitive work can consume.
