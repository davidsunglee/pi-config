## Summary

Today the workflow is strictly forward: spec → plan → execute, then done. Two gaps fall out of that:

1. **No resume path.** Execute-plan's skill prose claims plans can be resumed, but no workflow exists. No checkpoint file, no stale-baseline handling, no re-entry point at wave N. Partial failures drop users back to manual git.
2. **No revision path.** If wave 3 reveals the spec was wrong, there is no structured way to amend the spec, regenerate only affected tasks, and continue from the current checkpoint. Users either commit partial work and restart the chain or abandon.

These two are one design problem. Revision requires resume: you can only "regen and continue from wave N" if "continue from wave N" exists. This todo scopes both.

## Phase 1 — Design spike (produces a spec, not code)

Open questions the spike must answer:

- **State shape.** What must persist to support resume? Candidates: completed waves, per-wave HEAD SHA, baseline SHA, worktree path, plan-version hash, last-known-good task outputs.
- **State location.** `.pi/plans/<plan>.state.json`? Co-located with plan? Per-worktree?
- **Plan lifecycle interaction.** Plan moves to `.pi/plans/done/` on completion. Where does state live when plan is revised mid-flight? Before resume? After resume?
- **Stale-baseline detection.** Repo drifts between runs. When is a stored baseline invalid? How does execute-plan detect and handle that?
- **Entry point.** On `execute-plan <plan>`: if state file exists, offer resume-from-last-successful-wave vs. fresh run. What does the user see?
- **Revision surface.** Does amend happen via re-running define-spec on the same target? A new `amend-spec` skill? Editing the spec file directly and a diff tool catches the delta?
- **Plan-delta semantics.** Given spec v1 → spec v2, which tasks are: (a) unaffected (keep committed work), (b) invalidated (need re-run), (c) new (insert into a later wave), (d) removed (revert)?
- **Revert strategy.** `git revert` per invalidated task? Branch reset to a safe SHA? User confirmation required?
- **User-facing skill.** One new skill? Extension of generate-plan? What is the entry point surface area?

Spike output: a design doc in `docs/analysis/` and a spec in `.pi/specs/` that the primitive work can consume.

## Phase 2 — Primitive implementation

Depends on Phase 1 design.

Likely sub-tasks (exact shape depends on design):
- Checkpoint writer in execute-plan (per-wave)
- Stale-baseline detector
- Resume entry point on execute-plan
- Spec-diff tool (semantic comparison of two spec versions)
- Plan-delta producer (given original plan + spec-diff → affected task set)
- Revision skill or generate-plan extension

## Is this spec-worthy?

Yes. Run the full define-spec → generate-plan → execute-plan workflow on this one. New concepts (checkpoint state, revision semantics), multi-skill touchpoints (define-spec, generate-plan, execute-plan), and non-trivial judgment calls in the design phase.

The small workflow-polish items (worktree UX, BLOCKED immediacy, model-tier override) stay in `8ddd2e17` and should be handled as a lightweight plan or direct edits — not spec-worthy on their own.
