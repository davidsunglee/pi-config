## Goal

Create a skill at `.pi/agent/skills/execute-plan/SKILL.md` that reads a plan file from `.pi/plans/`, decomposes it into dependency-ordered waves, and dispatches `plan-executor` subagents to execute each wave.

## Architecture Decision

- The **main agent** (with skill loaded) handles orchestration: reading the plan, building the dependency graph, grouping waves, dispatching workers, handling status codes, retries, and user interaction
- Individual tasks are dispatched to **`plan-executor` subagents** (not the generic `worker`) because they need structured status reporting (DONE, DONE_WITH_CONCERNS, NEEDS_CONTEXT, BLOCKED)
- The `plan-executor` agent definition lives at `~/.pi/agent/agents/plan-executor.md` (already created)

## Plan file location

- Plans live in `.pi/plans/` with filenames like `yyyy-MM-dd-<short-description>.md`
- The companion `generate-plan` skill (TODO-7ef7d441) dispatches `plan-generator` to write plans
- This skill only executes existing plans — it does not generate them

## Expected plan format

Plans must follow the structure produced by the `plan-generator` agent. On load, validate that the plan contains:

1. **Header** — goal, architecture summary, tech stack
2. **File structure** — list of files to create or modify with responsibilities
3. **Numbered tasks**, each with:
   - `**Files:**` section with exact paths (Create / Modify / Test)
   - Checkbox steps (`- [ ] **Step N: ...**`)
   - Acceptance criteria
   - Model recommendation (cheap / standard / capable)
4. **Dependencies** section — which tasks depend on which
5. **Risk assessment**

If the plan is missing required sections, **stop and tell the user** what's missing rather than guessing. Suggest they re-generate the plan with the `generate-plan` skill.

## Startup: ask execution preferences

Before executing, ask the user two questions:

**1. Execution mode:**
- Sequential (one task at a time)
- Parallel (dependency-ordered waves of up to 7 workers)

**2. Wave pacing** (if parallel):
- **(a) Pause between waves** — user reviews before next wave
- **(b) Auto-continue, report failures at end** — all waves run, failures collected and reported when done
- **(c) Auto-continue unless failures** — pause and ask user only when a wave has failures

Default: parallel with pacing (c) unless user specifies otherwise.

## Execution behavior

### Dependency resolution
1. Read the plan file, identify all tasks
2. Read the explicit dependencies section to build the dependency graph
3. Group tasks into waves where all tasks in a wave have their dependencies satisfied by prior waves

### Worker dispatch
- Up to 7 parallel `plan-executor` subagents per wave (subagent parallel task limit)
- Each worker gets a **self-contained prompt** including:
  - Exact source files to read
  - Exact output file path
  - Content spec / acceptance criteria from the plan
  - Cross-linking and formatting conventions
- Dispatch via subagent tool, overriding model per task:
  ```
  subagent { tasks: [
    { agent: "plan-executor", task: "...", model: "<resolved from tier>" },
    { agent: "plan-executor", task: "...", model: "<resolved from tier>" },
    ...
  ]}
  ```

### Model selection

Plans annotate each task with an abstract tier: `cheap`, `standard`, or `capable`. The skill resolves these to actual models **dynamically at execution time**. **Never hardcode model names or versions in the skill itself.**

Auto-detect model family with this priority: **Claude → GPT → Gemini**. Use whichever family is available, then resolve tiers to the **latest available version** of that family's corresponding model class:

- **capable** → latest flagship/opus-class model
- **standard** → latest mid-tier/sonnet-class model (also the `plan-executor` default)
- **cheap** → latest fast/haiku-class model

The user can override the model family if they want, but shouldn't have to — it should just work.

The `plan-executor` agent has a default model, but the skill should always pass an explicit model override per task based on the plan's tier recommendation. This ensures cheap tasks don't waste money on the default model and capable tasks get the best available.

If the plan omits tier recommendations for a task, apply this fallback rubric:
- Touches 1-2 files with a complete spec → cheap
- Touches multiple files with integration concerns → standard
- Requires design judgment or broad codebase understanding → capable

### Existing output files
- If output files from a prior run already exist, **ask the user** whether to skip or re-run those tasks

### Verification after each wave
- **Full content validation** — read each output file and verify it matches the plan spec (not just existence/non-empty checks)

### Handling plan-executor status codes

The `plan-executor` agent reports one of four statuses. Handle each appropriately:

- **DONE**: Proceed to spec compliance review (full content validation).
- **DONE_WITH_CONCERNS**: Read the concerns before proceeding. If correctness/scope concerns, address before review. If observations, note and proceed.
- **NEEDS_CONTEXT**: Provide the missing context and re-dispatch.
- **BLOCKED**: Assess the blocker:
  - Context problem → provide more context and re-dispatch
  - Reasoning problem → re-dispatch with a more capable model
  - Task too large → break into smaller pieces
  - Plan is wrong → escalate to the human

Never ignore an escalation or force the same model to retry without changes.

### Failure handling
- If a worker produces empty, missing, or wrong output: **retry automatically up to 3 times**
- If still failed after 3 retries: **notify the user at end of wave** and ask whether to:
  - Retry again
  - Skip the failed task and continue to next wave
  - Stop the entire plan

### Plan lifecycle
- When execution completes successfully, move the plan file to `.pi/plans/done/`
