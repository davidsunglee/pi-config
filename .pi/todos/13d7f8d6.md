## Problem

execute-plan runs parallel tasks that modify files across multiple waves, but there are no git checkpoints or integration verification between waves. This means:
- A 5-wave plan produces one giant uncommitted diff
- If execution stops at wave 3, you have dirty state with no rollback points
- Wave 4 tasks might silently conflict with wave 3 outputs
- The final code review's `git diff` is a massive wall

## Solution

### 1. Commit after each wave

After a wave's tasks complete and pass spec review (Step 9), commit all changes with a structured message:

```
feat(plan): wave 1 - <plan goal summary>

- Task 1: <task title>
- Task 3: <task title>
```

Subject line uses the plan's Goal section (truncated to fit). Body lists each task completed in that wave as `- Task N: <title>` (dashed list, one per line, blank line after subject per git convention).

**No git repo:** Skip commits silently. Working on files outside version control is anomalous — don't error or warn.

**On main branch:** If executing directly on main/master/develop (no feature branch, no worktree), ask the user to confirm before the first commit: "You're on `main`. Commits will be made directly to main after each wave. Continue? (y/n)". Ask once at the start, not per wave. If declined, skip commits for the entire execution (proceed without checkpoints).

### 2. Test command in the plan

Add a `## Test Command` section to the plan format (optional). The planner specifies the command to run based on its codebase analysis:

```markdown
## Test Command

```bash
npm test
```
```

If the section is missing from a plan, fall back to auto-detect (`package.json` → `npm test`, `Cargo.toml` → `cargo test`, `Makefile` with test target → `make test`, etc.) or ask the user during Step 3 settings. This keeps backward compatibility with existing plans.

### 3. Baseline test capture

Before executing the first wave (after Step 6, before Step 7 dispatch), run the test command and record the result:
- **All pass (exit 0):** Record clean baseline. Any post-wave failure is a regression.
- **Some fail:** Warn the user: "⚠️ N tests already failing before execution. New failures only will be flagged." Record the failing test names/count. Proceed with execution.
- Only flag **new** failures after each wave — compare against the baseline to distinguish pre-existing failures from regressions.

### 4. Integration tests between waves

After committing a wave (or after Step 9 if commits are skipped), run the test command. Strictly pass/fail — exit code 0 is pass, anything else is fail.

**On failure:** Present the user with choices (same pattern as Step 10 retry/skip/stop):
- **Retry** — re-dispatch the wave's tasks with test output appended to the prompt
- **Skip** — proceed to the next wave despite failures
- **Stop** — halt plan execution (changes committed so far are preserved as checkpoints)

Pre-existing failures (from baseline) are excluded from the pass/fail decision — only new failures count.

## Changes

### Plan format
- Add optional `## Test Command` section to the plan structure
- Update plan validation (Step 2) to recognize but not require the new section
- Update `plan-generator` to detect and include a test command when generating plans

### execute-plan SKILL.md
- **Step 2 (validate):** Recognize optional `## Test Command` section
- **Step 3 (settings):** Add "Commit per wave" setting (Enabled default, only shown if git repo detected). Add "Integration tests" setting (Enabled default if test command found). If no test command in plan and no auto-detect, ask for the command or allow disabling.
- **New step after Step 6 / before Step 7:** Baseline test capture — run the test command, record results, warn if pre-existing failures
- **Step 7 (before first dispatch):** If on main and commits enabled, ask for main-branch confirmation
- **New step after Step 9 (after wave verification):** Commit wave changes + run integration tests. On test failure, present retry/skip/stop choices.
- **Step 12 (code review):** The git range now spans multiple commits (one per wave) instead of a single diff — no change needed, `BASE_SHA..HEAD_SHA` already covers this.
