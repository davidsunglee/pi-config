## Goal
Unify the working message and working indicator around a shared runtime state model and shared configuration so they communicate Pi activity more clearly, remain easy to read in code, and are configurable through both `working.json` and a new `/working` command.

## Runtime model
The UI should use four effective modes, ordered by priority:
1. **hidden** — when control is with the user / Pi is idle
2. **thinking** — highest-priority active state
3. **toolUse** — active while Pi is executing tools
4. **active** — default catch-all active state while Pi is working but not in a higher-priority state

### State drivers
- Preserve the existing hide/show lifecycle behavior that currently makes the working UI disappear when the assistant turn ends / control returns to the user.
- `thinking` is driven by the existing model thinking stream events:
  - `thinking_start`
  - `thinking_end`
- `toolUse` is driven by tool execution events:
  - `tool_execution_start`
  - `tool_execution_update`
  - `tool_execution_end`
- `active` is the fallback visible state during an assistant turn when neither `thinking` nor `toolUse` is active.

### State priority
When multiple signals could apply, the priority is:
- hidden > thinking > toolUse > active

## Visual behavior
### Shared state styling
Use one shared style configuration per state for **both** the working message and the working indicator.
Each state should define:
- `color`
- `gleam`
- `rainbow`

### Default state styles
- `active`
  - color: built-in default working color (current Nord blue used by working message)
  - gleam: `false`
  - rainbow: `false`
- `toolUse`
  - color: built-in default working color
  - gleam: `true`
  - rainbow: `false`
- `thinking`
  - color: built-in default working color
  - gleam: `true`
  - rainbow: `true`

### Effect semantics
- `rainbow` visually replaces the configured base color.
- `gleam` and `rainbow` may both be enabled at once.
- For the indicator, **shape controls motion** and **effects control styling/color treatment**.
- If `indicatorShape=dot`, the dot remains static; gleam only affects the styling/color treatment, not motion.

## Indicator shape
Add a top-level shared indicator-shape setting:
- config key: `indicatorShape`
- allowed values: `dot | pulse | spinner`
- default: `spinner`

Remove manual persistent hidden/none selection from the indicator config/command model. Hidden state should be runtime-driven by idle/user-turn state only.

## Configuration: `agent/working.json`
Replace the current schema with a new shared, state-first schema optimized for user readability.

### Naming
- Config keys use **camelCase**.
- Runtime style states in config are:
  - `active`
  - `toolUse`
  - `thinking`

### Desired schema shape
```json
{
  "indicatorShape": "spinner",
  "active": {
    "color": "#81A1C1",
    "gleam": false,
    "rainbow": false
  },
  "toolUse": {
    "color": "#81A1C1",
    "gleam": true,
    "rainbow": false
  },
  "thinking": {
    "color": "#81A1C1",
    "gleam": true,
    "rainbow": true
  }
}
```

### Config rules
- No top-level color fallback/inheritance model is required.
- Each state owns its own `color` value.
- `color=default` should reset a state to the built-in default working color (the current Nord blue used by the working message).
- Supported custom color input format is **hex only**: `#RRGGBB`.
- No backward compatibility with the old `working.json` schema is required.

## Command surface
Replace `/working-indicator` with a new `/working` command.

### Command behavior
- `/working` with no args should print a summary of the current working configuration.
- Command state names should use **kebab-case** for user ergonomics:
  - `active`
  - `tool-use`
  - `thinking`

### Command grammar
- `/working indicator=dot`
- `/working indicator=pulse`
- `/working indicator=spinner`
- `/working active color=default|#RRGGBB`
- `/working active gleam=on|off`
- `/working active rainbow=on|off`
- `/working tool-use color=default|#RRGGBB`
- `/working tool-use gleam=on|off`
- `/working tool-use rainbow=on|off`
- `/working thinking color=default|#RRGGBB`
- `/working thinking gleam=on|off`
- `/working thinking rainbow=on|off`

## Code structure expectations
- Prioritize code that is **readable, DRY, and easy for a human to follow**.
- Centralize shared logic between the working message and working indicator, especially:
  - runtime state tracking
  - event handling / state transitions for thinking and tool execution
  - config loading/parsing/defaulting
  - style/effect resolution
- It is acceptable to keep the implementation in one main logic file if it remains clean.
- It is also acceptable to extract large constants/data tables (for example, working-message strings or rainbow palettes) into separate files if that keeps the main logic file under roughly **500 lines**.
- Naming/model cleanup is explicitly in scope so the concepts of visibility, active state, tool use, thinking, and effects read clearly in code and config.

## Acceptance criteria
- The working message and working indicator use a shared runtime state model with states `active`, `toolUse`, and `thinking`, plus hidden while idle.
- The working indicator changes styling by active state, including reacting to thinking and tool execution.
- Both message and indicator read from the same per-state style config in `working.json`.
- `working.json` supports `indicatorShape` plus shared `active`, `toolUse`, and `thinking` style blocks.
- `/working-indicator` is replaced by `/working`.
- `/working` supports the agreed command grammar and reports current config when run with no args.
- `color=default` resets to the built-in Nord blue default working color.
- `indicatorShape=dot` remains static while still reflecting styling/effect changes.
- Hidden behavior remains aligned with the current assistant-turn lifecycle.
- Shared logic is centralized rather than duplicated across working message and indicator implementations.
- Code and docs clearly reflect the new naming/model.
