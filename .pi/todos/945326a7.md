## Current understanding

After the switch to `pi-interactive-subagent`, the original framing of this todo is no longer fully accurate.

- **Pi-backed subagents are not the main gap anymore.** They start fresh pi sessions that load extensions from the active config root / target cwd. In the current setup, that includes `guardrails.ts`, so pi children should get the same session-level guardrail hook the host gets.
- **Claude-backed subagents are still a gap.** The `pi-interactive-subagent` Claude path launches with permissive permission modes (`--dangerously-skip-permissions` on the pane path, `--permission-mode bypassPermissions` on the headless path), and Claude does not execute pi's `tool_call` extension hook. So `guardrails.ts` does **not** protect Claude-backed subagents.
- **This matters a lot in practice** because `agent/model-tiers.json` currently routes Anthropic models to `cli: "claude"`, so many real workflow agents (`planner`, `coder`, `code-refiner`, etc.) commonly run on the Claude path.

So the updated problem is **not** "subagents in general bypass guardrails." The real problem is:

> Claude-backed subagents lack protection equivalent to `guardrails.ts`, and several current agent tool surfaces are too permissive for that backend.

## Why this matters

- Autonomous subagents are still the highest-risk execution context.
- `code-refiner` has `maxSubagentDepth: 1`, so nested dispatch (`code-refiner` -> child reviewer/coder) is still relevant.
- Several roles are broader than they should be:
  - `planner` has `bash`, so it is not truly read-only in enforcement terms.
  - `plan-reviewer` also has `bash`.
  - `coder`, `code-reviewer`, and `code-refiner` currently rely heavily on prompt compliance / backend defaults rather than tight `tools:` allowlists.
- `verifier` is a useful contrast case: it already has a restricted tool surface, so it is much closer to the desired posture.

## Desired outcome

Establish a deliberate safety posture for subagents under `pi-interactive-subagent`:

- pi-backed children keep benefiting from `guardrails.ts`
- Claude-backed children get a safer default posture than `bypassPermissions`
- agent roles have intentionally minimal tool surfaces
- read-only roles are read-only by enforcement, not just by prompt wording
- nested dispatch follows the same safety rules

## Design questions

### 1. Claude backend safety posture

- Should `pi-interactive-subagent` plumb Claude `permissionMode` and switch the default from `bypassPermissions` to `auto`?
- Is Claude `auto` only an interim mitigation, or is it sufficient for some roles?
- Are there any roles that truly require `bypassPermissions`, and if so, why?

### 2. Backend routing strategy

- Should high-risk writer/orchestrator roles prefer `cli: "pi"` so repo guardrails definitely apply?
- Should model routing become **role-aware**, not just provider-aware?
- Is the current `dispatch` map too coarse if it routes almost every Anthropic-backed worker onto the less-protected Claude path?

### 3. Agent tool-surface hardening

- Should `planner` and `plan-reviewer` lose `bash` entirely?
- Should `code-reviewer` become explicitly read-only via `tools:` frontmatter?
- Should `coder` / `code-refiner` get explicit allowlists instead of inheriting a broad default surface?
- For Claude-backed agents, can `--tools` restrictions plus stricter role definitions bring the risk down enough?

### 4. Nested dispatch

- Ensure the chosen safety posture covers `code-refiner` -> `code-reviewer` / `coder` child launches as well as top-level orchestration.
- Avoid a design where the parent is hardened but nested workers fall back to a more permissive path.

## Non-goals

- Treating prompt instructions alone as a safety solution.
- Reverting to the old `pi-subagent` assumptions.
- Declaring the issue fixed merely because pi-backed children likely load the guardrails extension.

## Completion criteria

- The subagent safety model is documented accurately for the `pi-interactive-subagent` world: pi-backed children are covered by session-loaded guardrails; Claude-backed children are handled explicitly rather than implicitly.
- Claude-backed subagents no longer run with `bypassPermissions` by default **or** there is an explicit, justified exception list.
- Read-only roles (`planner`, `plan-reviewer`, `code-reviewer`, `verifier`) cannot write in practice, not just by prompt convention.
- High-risk roles (`coder`, `code-refiner`, nested remediation workers) have explicit, intentional tool surfaces and backend choices.
- The chosen protection model works for nested dispatch.
- Verification covers both pi and Claude execution paths.
