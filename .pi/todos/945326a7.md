## Problem

The `guardrails.ts` extension intercepts `tool_call` events on the host pi session and blocks or confirmation-gates dangerous operations (destructive shell commands, writes to sensitive paths, force pushes, etc.). However, subagents dispatched via `pi-subagent` run as separate processes with their own tool execution contexts. They do not inherit the host session's `tool_call` hook, so none of the guardrails apply.

This means every `coder` subagent dispatched by `execute-plan` has full, unguarded write access. A hallucinating coder could `rm -rf` a directory, write to `.env`, force-push, or hit any other dangerous operation that the guardrails were designed to catch — and nothing would intercept it.

The same gap applies to all dispatched subagents: `planner` (read-only by convention, not enforcement), `code-reviewer`, `code-refiner` (which itself dispatches sub-subagents), and `plan-reviewer`.

## Why this matters

- The guardrails exist specifically because models sometimes produce dangerous tool calls. Subagents — running autonomously without a human watching — are the highest-risk context for this.
- The `code-refiner` has `maxSubagentDepth: 1`, meaning it dispatches its own sub-subagents. That's two levels of unguarded execution.
- The `planner` agent is "read-only" only because its system prompt says so. There's no enforcement preventing it from writing if it hallucinates past the instruction.

## Design questions

### 1. Where should guardrails be enforced?

Options:

- **In `pi-subagent`:** The subagent dispatch infrastructure could inject guardrail configuration or hooks into every spawned agent. This covers all subagents uniformly but requires changes to the `pi-subagent` package.
- **In pi itself:** If pi supports a project-level guardrail config that applies to all sessions (including subagent sessions) spawned from a project directory, that would be the cleanest solution. May not exist today.
- **In the agent definitions:** Each agent `.md` file could include explicit tool restrictions (e.g., planner gets no write tools). This is already partially done via the `tools` frontmatter field in `planner.md`, but it's opt-in and doesn't cover dangerous uses of allowed tools (e.g., `bash` running `rm -rf`).
- **In the dispatch prompt:** The skill could inject a "forbidden operations" block into every worker prompt. Weakest option — relies on the model obeying the instruction, which is exactly the failure mode guardrails are meant to catch.

### 2. What level of protection is appropriate?

The host guardrails have three tiers: hard-block (sensitive paths), confirmation-gate (dangerous commands), and allow. Subagents can't do interactive confirmation — there's no human to approve. Options:

- **Block everything that would be confirmation-gated in the host.** Conservative, but may block legitimate operations (e.g., a coder that needs to `rm` a file it's replacing).
- **Block only hard-block operations, allow the rest.** Matches the host's non-interactive posture but leaves dangerous commands unguarded.
- **Allowlist approach:** Define a positive set of permitted operations per agent role rather than a blocklist. Most restrictive but hardest to maintain.

### 3. What about the planner?

The `planner` agent declares `tools: read, grep, find, ls, bash` in its frontmatter. It has `bash` access, which means it can write despite being "read-only by convention." Should the planner's tool set be restricted to truly read-only tools, or should bash be scoped (e.g., no write redirections)?

## Completion criteria

- Dispatched subagents are covered by guardrails equivalent to or stricter than the host session
- The solution works for nested dispatch (`code-refiner` → `coder`)
- Read-only agents (planner, plan-reviewer, code-reviewer) cannot write even if they attempt to
- The guardrail enforcement does not depend solely on prompt compliance
