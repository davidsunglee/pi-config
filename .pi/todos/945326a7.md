## Goal

Simplify the subagent safety work into two concrete decisions:

1. Explore changing Claude-backed subagents from bypass permissions to Claude `auto` mode.
2. Decide the intended tool allowlist for each subagent type.

## Questions to answer

### Claude permission mode

- Can pane and headless Claude launches use `auto` instead of `bypassPermissions` / `--dangerously-skip-permissions`?
- What breaks, if anything, when Claude subagents run in `auto`?
- Do any subagent roles genuinely need bypass permissions? If yes, document the exception.

### Subagent tool surfaces

For each local agent in `agent/agents/`, decide the intended tools:

- `spec-designer`
- `planner`
- `plan-reviewer`
- `plan-refiner` â€” added by the refine-plan flow; include it in the coordinator-agent review.
- `coder`
- `verifier`
- `code-reviewer`
- `code-refiner`

Focus on role intent:

- read-only agents should not be able to write
- verifier should remain judge-only
- coder/remediation agents should have only the tools they need to edit and verify
- coordinator agents should have the orchestration tools they need, but not broad unnecessary write access

Coordinator-specific expectations to decide and document:

- `plan-refiner` coordinates one refine-plan era: dispatches `plan-reviewer` and `planner` edit-mode subagents, writes a versioned review file under `.pi/plans/reviews/`, may append approved review notes to the plan, but must not commit or edit implementation code directly.
- `code-refiner` coordinates code review/remediation: dispatches `code-reviewer` and `coder` subagents, manages review artifacts, and may commit remediation batches according to its current role rules.
- Coordinator tool surfaces should be explicit: orchestration tools where needed, read access to required artifacts, narrowly scoped write/edit authority for owned artifacts, and no broad write access when child agents own the actual code or plan edits.
