## Problem

The Gemini section of the usage-bar extension is returning `401 Unauthorized` when `/usage` runs.

Current root cause in the extension:
- `~/.pi/agent/extensions/usage-bar.ts` manually reads raw Gemini access tokens from disk
- it reads from `~/.pi/agent/auth.json` directly and also falls back to `~/.gemini/oauth_creds.json`
- it then calls the Gemini quota endpoint with that raw token
- this bypasses pi’s auth layer, so the extension does **not** benefit from pi’s normal OAuth refresh behavior
- the result is that `/usage` can report Gemini as unauthorized even when the right fix should be handled through pi’s own auth flow

## Goal

Make the Gemini section of the usage-bar extension reflect **what pi itself can actually use**, by relying on pi’s auth/model-registry path instead of manually reading Gemini CLI credential files.

## Decisions from refinement

- **Source of truth:** only use **pi auth**, not Gemini CLI credentials
- **Auth path:** use pi’s `modelRegistry` / `authStorage` path so the extension reflects pi-available auth
- **Fallbacks:** do **not** fall back to `~/.gemini/oauth_creds.json`
- **Error handling:** if Gemini still fails after going through pi auth, stop there and surface a clear pi-centric error
- **Scope:** keep this narrowly focused on making the extension work for the current local pi setup
- **Tests:** no automated tests for now; manual verification only
- **Refactoring:** no broader cleanup beyond what is needed for this fix

## Why this direction

pi already has an auth system that persists `/login` credentials in `~/.pi/agent/auth.json` and manages OAuth token refresh through its auth/model registry. The Gemini usage-bar path should rely on that system instead of re-implementing token handling or consulting unrelated credential files.

The extension should answer: **"what can pi use right now?"**
not: **"what raw Gemini credentials exist somewhere on disk?"**

## Scope

Update:
- `~/.pi/agent/extensions/usage-bar.ts`

Specifically, adjust the Gemini usage path so it:
1. Gets Gemini auth from pi’s runtime auth/model-registry layer
2. Stops manually reading `~/.gemini/oauth_creds.json`
3. Avoids custom Gemini token-refresh logic inside the extension if pi’s auth layer already handles it
4. Returns a clearer error when pi has no usable Gemini auth or the provider call still fails

## Implementation direction

Likely changes:

1. **Replace manual Gemini token loading**
   - remove the current direct-file Gemini auth lookup in `fetchGeminiUsage()`
   - use `modelRegistry` / `authStorage` instead, similar to how Codex already asks pi auth for credentials

2. **Reflect pi-only auth state**
   - if pi has no Gemini auth configured, report that clearly
   - do not consult `~/.gemini/oauth_creds.json`

3. **Use pi-managed token refresh implicitly**
   - rely on pi auth storage behavior instead of hand-rolling Gemini refresh in the extension
   - if pi returns no usable token, surface that outcome clearly

4. **Improve error messaging**
   - prefer accurate messages over opaque raw 401s when possible
   - examples: `No credentials`, `Unauthorized`, `Gemini auth unavailable in pi`
   - keep messages truthful; do not guess at causes you cannot verify

## Acceptance criteria

- Triggering `/usage` uses **pi-managed Gemini auth only**
- The Gemini section no longer depends on `~/.gemini/oauth_creds.json`
- If pi can successfully use Gemini auth, the Gemini usage request succeeds in the extension as well
- If pi has no usable Gemini auth, the extension shows a clear, pi-centric error message
- The change is narrowly scoped and does not broaden into unrelated usage-bar refactoring

## Verification

Manual verification only for now:

1. reload the extension / reload pi as needed
2. run `/usage`
3. confirm one of these outcomes:
   - Gemini usage data loads successfully using pi auth, or
   - Gemini shows a clear auth-related message that reflects pi’s actual auth state

## Explicitly out of scope

- adding Gemini CLI credential fallback support
- custom Gemini token refresh implementation inside the extension, unless absolutely required after proving pi auth is insufficient
- broader provider-auth cleanup in usage-bar
- automated tests for this iteration
