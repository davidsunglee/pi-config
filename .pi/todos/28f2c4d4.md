## Problem

Recent workflow diagnostics found multiple broken or fragile hand-offs between orchestrator/coordinator agents and subagents.

Primary symptom: Claude-pane subagents often finished their textual answer but did not call a real completion callback, so their panes/windows had to be manually closed. Downstream orchestrators/coordinators then sometimes failed to receive a reliable `results[0].finalMessage`, treated the hand-off as missing/failed, and re-launched work.

## Evidence from recent session logs

Relevant Claude transcript examples:

- `~/.pi/agent/sessions/claude-code/bb26fe4e-eb63-47a6-9eec-118df172d085.jsonl`
- `~/.pi/agent/sessions/claude-code/b1e08076-7720-45b6-b0f0-8876af1f7e24.jsonl`
- `~/.pi/agent/sessions/claude-code/3c8dcd80-3a26-4622-a0da-9b5f3bbdbc3b.jsonl`

Observed transcript lines included:

- `No matching deferred tools found` after searching for `subagent_done`
- `No subagent_done tool is available in this environment — it may not be wired up for this session.`
- `The subagent_done tool is not available in this environment. Here is my final summary:`
- Some agents printed literal `<subagent_done>`, which is not a real callback/tool call.

## Root cause hypothesis

The pi subagent plugin files exist and Claude is launched with `--plugin-dir`, but the `subagent_done` MCP tool is not exposed in Claude Code sessions.

Plugin/tool files exist at:

- `/Users/david/Code/pi-interactive-subagent/pi-extension/subagents/plugin/mcp/server.js`
- `/Users/david/Code/pi-interactive-subagent/pi-extension/subagents/plugin/.mcp.json`
- `/Users/david/Code/pi-interactive-subagent/pi-extension/subagents/plugin/hooks/on-stop.sh`

Launch scripts pass:

```bash
claude --plugin-dir /Users/david/Code/pi-interactive-subagent/pi-extension/subagents/plugin ...
```

But actual Claude tool listings contained Gmail/Drive/Calendar/Context7 MCP tools and built-in deferred tools, not either expected lifecycle tool name:

- `mcp__pi-subagent__subagent_done`
- `mcp__plugin_pi-subagent_pi-subagent__subagent_done`

The Stop hook appears to run, but it only writes transcript pointers. It explicitly does not signal completion; completion signaling is delegated to the MCP `subagent_done` tool.

## Outstanding hand-off issues to fix

### 1. Claude subagent completion callback missing

Claude-pane subagents cannot reliably self-complete because `subagent_done` is not available.

Expected behavior:

- Any pi-spawned Claude subagent can call a real `subagent_done` MCP tool.
- The tool writes the sentinel file used by the watcher.
- The parent orchestration receives a normal completion result without manual pane closure.

Investigate/fix:

- Why `--plugin-dir .../subagents/plugin` loads hooks but not the MCP tool.
- Whether Claude requires an explicit `--mcp-config` fallback in addition to, or instead of, `--plugin-dir`.
- Whether plugin manifest/config naming changes are needed for Claude to expose the MCP server.
- Whether `buildClaudeCmdParts` needs to pass extra flags or environment variables.
- Whether Claude’s tool allowlist (`--tools`) is suppressing the lifecycle tool in some cases.

### 2. Manual close causes fragile result extraction

When a Claude subagent is manually closed after producing text, result extraction can be delayed or inconsistent. Coordinators may see no usable `results[0].finalMessage` or may receive fallback text rather than a clean structured result.

Affected flows:

- `plan-reviewer` returning plan review text to `generate-plan` / `plan-refiner`
- `code-reviewer` returning review text to `code-refiner`
- Claude `coder` / `verifier` workers used by `execute-plan`

Expected behavior:

- Manual closure should not be part of the happy path.
- If a subagent exits without calling `subagent_done`, the parent should still produce a clear diagnostic result rather than silently re-launching or treating output as absent.

### 3. `code-reviewer` → `code-refiner` hand-off is brittle

The `code-refiner` protocol expects:

1. Dispatch `code-reviewer`.
2. Read `results[0].finalMessage`.
3. Persist review to `.pi/reviews/...`.
4. Parse findings.
5. Remediate or finish.

When `code-reviewer` runs under Claude and cannot call `subagent_done`, the hand-off can stall or return unusable output. `code-refiner` may think no review was generated and re-launch the review.

Expected behavior:

- `code-refiner` should distinguish:
  - subagent transport/lifecycle failure,
  - missing/empty review,
  - malformed review,
  - valid review with findings.
- It should not repeatedly re-launch without surfacing a clear transport failure.

### 4. `plan-reviewer` hand-off is similarly brittle

`plan-reviewer` is judge-only/read-only and returns review text via `finalMessage`; the coordinator/orchestrator persists the review file.

If completion signaling fails, the caller may not receive the review text, causing manual intervention or repeated review attempts.

Expected behavior:

- Review text should be reliably returned through orchestration results.
- Missing or malformed review output should produce an explicit failure status.
- The plan-reviewer pane should auto-complete without manual close.

### 5. Literal `<subagent_done>` is not a valid fallback

Some Claude agents printed `<subagent_done>` as plain text. This does not call a tool or write the sentinel.

Expected behavior:

- Prompts/system instructions should make clear that `subagent_done` is a tool call, not a text token.
- If the tool is unavailable, agents should report a structured lifecycle/tooling error.
- Parent orchestration should detect this condition and fail clearly.

### 6. Validate Claude tool allowlist behavior

`buildClaudeCmdParts` attempts to include lifecycle MCP tools when `effectiveTools` is set:

- `mcp__pi-subagent__subagent_done`
- `mcp__plugin_pi-subagent_pi-subagent__subagent_done`

Need to verify:

- Whether those names match Claude’s actual names for plugin-loaded MCP tools.
- Whether an absent/incorrect `--tools` list prevents deferred MCP tools from appearing.
- Whether subagents without explicit `tools:` behave differently from those with explicit `tools:`.

### 7. Add regression coverage

Add tests or smoke coverage for both pi and Claude subagent paths:

- Claude pane launch exposes `subagent_done`.
- Claude subagent can call `subagent_done` and parent receives completion without manual close.
- Claude subagent with restrictive `tools:` still has lifecycle tool available.
- `code-refiner` handles missing/empty/malformed review output explicitly.
- `plan-refiner` handles missing/empty/malformed review output explicitly.
- No coordinator silently loops forever/re-launches on transport failure.

## Previously found and fixed hand-off issue

A real `generate-plan` → `refine-plan` contract mismatch was found and fixed:

- Old behavior: `generate-plan` documented passing `--plan-path <path>`.
- Actual `refine-plan` contract: `PLAN_PATH` is positional.
- Fixed by commit `1514b94 fix(review): align refine-plan handoff`.

Keep this in mind as an example of protocol-contract drift to guard against in tests/docs.

## Suggested acceptance criteria

- A pi-spawned Claude subagent can call `subagent_done` successfully in a minimal smoke test.
- The parent orchestration receives the subagent result without manual pane/window closure.
- Claude subagent transcript/tool listing shows the expected pi lifecycle MCP tool.
- `code-refiner` no longer re-launches review when a reviewer completed but lifecycle signaling failed; it reports a clear transport/output error instead.
- `plan-reviewer`/`plan-refiner` flow completes without manual close and persists the expected review artifact.
- Regression tests cover lifecycle tool availability and allowlist behavior.
- Documentation or launcher code clearly explains/falls back between `--plugin-dir` and `--mcp-config` if needed.
