## Problem

Execute-plan Step 9b (post-wave integration test failure) has two issues:

### 1. Blunt retry strategy

When integration tests fail after a wave, the current behavior is:
1. Undo the wave commit
2. Re-dispatch **all** tasks in the wave with raw test output appended
3. Hope the workers figure out what broke

This doesn't analyze which task caused the failure, doesn't do targeted debugging, and re-runs tasks that were probably fine.

### 2. Inconsistent user choice presentation

Step 9b uses `(r)/(s)/(x)` mnemonic letters, while other choice points in the suite use `(a)/(b)/(c)` with imperative verb phrases:

- generate-plan Step 4: `(a) Keep iterating` / `(b) Proceed with issues`
- execute-plan Step 12: `(a) Keep iterating` / `(b) Proceed with issues` / `(c) Stop execution`

## Tasks

### 1. Replace blunt retry with systematic debugging loop

Instead of re-dispatching all tasks blindly, invoke the `systematic-debugging` skill (or a similar targeted approach) to:
- Analyze the test failure output
- Identify which task(s) likely caused the regression
- Attempt targeted fixes
- Re-run tests to verify

### 2. Align user choices with suite conventions

Replace the `(r)/(s)/(x)` options with the consistent `(a)/(b)/(c)` pattern:

- **(a) Debug failures** — kick off systematic debugging loop for the failing tests
- **(b) Skip tests** — proceed to next wave despite failures
- **(c) Stop execution** — skip completion, report partial progress
