## Problem

The plan-generator produces plans that can have structural issues not caught until execution fails or produces wrong output. Examples from the wiki generation:

- Task 1 (overview) needed to link to filenames produced by Tasks 2-14, but the plan declared no dependencies — the worker guessed filenames and got them wrong
- Acceptance criteria were binary checklists ("contains mermaid diagram") rather than quality expectations ("include worked examples where they clarify flow")
- No detection of tasks that are too large for a single worker or that overlap in scope

And from the skill porting:

- Plan said "Add at the bottom: `<!-- attribution -->`" but 3/7 workers put it before the YAML frontmatter, breaking skill loading. The plan stated the frontmatter format but not the constraint ("nothing before the frontmatter").

## Solution: Two changes

### Change 1: Plan-generator prompt improvement

Update `~/.pi/agent/agents/plan-generator.md` to add guidance in the plan output section:

> When tasks create files with specific format requirements, state both the required structure AND common mistakes/constraints that would break it.

This prevents the class of bug where the plan specifies a format (e.g., YAML frontmatter) but doesn't state the footgun (e.g., "frontmatter must be the very first line — do not put anything before it"). Better to prevent at generation time than catch at review time.

### Change 2: Add plan review step between generation and reporting

Modify `~/.pi/agent/skills/generate-plan/SKILL.md` to add a new Step 3.5 between "plan-generator completes" and "report to user."

#### New Step 3.5: Review the plan

**Dispatch mechanism:** Use `plan-executor` agent with the `plan-reviewer.md` prompt template injected as the task (same pattern as `requesting-code-review` uses `plan-executor` + `code-reviewer.md`). No new subagent definition needed.

1. Read the prompt template from `~/.pi/agent/skills/generate-plan/plan-reviewer.md`
2. Fill placeholders with:
   - The full generated plan file contents
   - The original task spec (todo body, file contents, or freeform description from Step 1)
3. Dispatch `plan-executor` with the filled template

#### Reviewer model selection — cross-provider

Use a capable-tier model from a **different provider** than the plan-generator to get an independent perspective:

- Detect which provider family the generator used (parse model string: `claude-*` → Anthropic, `gpt-*` → OpenAI, `gemini-*` → Google)
- Pick the capable tier from the next provider in priority order: Anthropic → OpenAI → Google → Anthropic
- **Fallback:** If the alternate provider is not available, use the same model as the generator

Resolution uses the same tier table as `execute-plan` Step 6 — do not hardcode model version strings in this skill.

#### What the reviewer checks

Generic structural checks (no format-specific knowledge baked in):

- **Spec/todo coverage** — does every requirement in the original todo/spec have a corresponding task? Any gaps?
- **Dependency accuracy** — are there implicit dependencies not declared? (e.g., Task A references output filenames from Task B but says "no dependencies")
- **Task sizing** — any task that reads 4000+ lines of source or produces very large output may need splitting
- **Cross-task consistency** — do tasks that reference each other's outputs use consistent names/paths?
- **Acceptance criteria quality** — are criteria specific enough to verify, or are they vague pass/fail checks?
- **Buildability** — could an engineer/agent follow this plan without getting stuck? Any tasks so vague they can't be acted on?
- **Placeholder content** — any TODOs, TBDs, "implement later", "similar to Task N"

**Calibration:** Only flag issues that would cause real problems during execution. An agent building the wrong thing or getting stuck is an issue. Minor wording and stylistic preferences are not. Approve unless there are serious gaps.

#### Reviewer output format

Structured findings with severity levels:

- **Error** — missing tasks, wrong dependencies, tasks that can't be executed. Blocks execution.
- **Warning** — vague criteria, sizing concerns, consistency risks. Informational.
- **Suggestion** — improvements that would make the plan better but aren't problems.

#### Handling reviewer findings

**Always present findings to the user.** The user decides what to do:

- **If errors found:** Show findings. User chooses to re-generate (with findings attached to the prompt) or to manually fix the plan.
- **If only warnings/suggestions:** Append findings as a `## Review Notes` section at the end of the plan file and continue to Step 4.
- **If clean (no issues):** Continue to Step 4 with no changes.

The `## Review Notes` section format:

```markdown
## Review Notes

_Added by plan reviewer — informational, not blocking._

### Warnings
- **Task 3**: Acceptance criteria "contains diagram" is vague — consider specifying diagram type and what it should show

### Suggestions
- **Task 1 → Task 5**: Both reference `config.json` output — verify path consistency
```

**No changes needed to `execute-plan`** — its Step 2 validation only checks for the 5 required sections (header, file structure, tasks, dependencies, risk). Extra sections at the end are ignored.

### Deliverables

1. **`~/.pi/agent/skills/generate-plan/plan-reviewer.md`** — New prompt template. Accepts `{PLAN_CONTENTS}` and `{ORIGINAL_SPEC}` placeholders. Calibrated for plan documents (not code diffs). Produces: Status (Approved / Issues Found), Issues list with severity, Recommendations.

2. **`~/.pi/agent/skills/generate-plan/SKILL.md`** — Add Step 3.5 (review) between existing Steps 3 and 4. Update Step 4 to include review status in the report.

3. **`~/.pi/agent/agents/plan-generator.md`** — Add constraint/footgun guidance to the plan output section.

### Skill flow after change

```
Step 1: Determine input source (unchanged)
Step 2: Assemble prompt (unchanged)
Step 3: Dispatch plan-generator (unchanged — but generator prompt improved per change 1)
Step 3.5: Dispatch plan-reviewer (NEW)
  - Use plan-executor agent with plan-reviewer.md template
  - Capable-tier model from alternate provider (fallback: same provider)
  - Input: generated plan + original spec/todo body
  - Output: findings (error/warning/suggestion)
  - If errors: present to user, user decides
  - If clean/warnings only: append notes to plan, continue
Step 4: Report result (updated to include review status)
```

### Key design decisions

- **No new subagent definition** — follows the same pattern as `requesting-code-review`: use `plan-executor` with a prompt template.
- **Cross-provider model** — reviewer uses a different provider's capable model to avoid anchoring bias from same-family reasoning patterns. Falls back to same provider if unavailable.
- **User decides on errors** — no auto-retry loop. Errors are presented and the user chooses to re-generate or manually fix.
- **Warnings don't block** — appended as `## Review Notes` to the plan file for transparency; execution proceeds.
- **Reviewer stays generic** — no format-specific knowledge. The "buildability" check is the right framing. Format-specific footguns are prevented by the plan-generator stating constraints explicitly (change 1).
