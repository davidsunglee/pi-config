## Summary

Model-tier + dispatch-CLI resolution from `~/.pi/agent/model-tiers.json` is still described in multiple skills with subtly different strictness and fallback behavior. This todo is now scoped only to canonicalizing that resolution policy and replacing duplicated in-skill algorithms with references to one shared document.

The former verifier command-evidence/context-pressure item has been addressed by the two-phase verifier design (`{PHASE_1_RECIPES}` + verifier-side Phase 1 evidence collection) and is intentionally out of scope here.

## Problem

Resolution from tier name → concrete model string → provider prefix → dispatch CLI is repeated or partially restated across skills. The copies have drifted in strictness:

- `agent/skills/define-spec/SKILL.md` fails loudly if `model-tiers.json` is missing/unreadable, if the selected tier is missing/empty, or if `dispatch.<provider>` is missing/empty. It explicitly forbids CLI-default fallback.
- `agent/skills/generate-plan/SKILL.md` reads `model-tiers.json`, references `execute-plan` for the full dispatch-resolution algorithm, and currently allows softer/default behavior around dispatch lookup.
- `agent/skills/execute-plan/SKILL.md` Step 6 contains the longest inlined algorithm and still documents defaulting to `"pi"` when the dispatch map or provider entry is absent.
- `agent/skills/refine-code/` and `agent/skills/refine-plan/` now share `_shared/coordinator-dispatch.md` for the coordinator hop, but their prompts and provenance validation still contain worker-dispatch tier/CLI resolution language that should be audited for consistency with the canonical policy.
- `agent/skills/requesting-code-review/SKILL.md` also resolves reviewer model/CLI and should be included in the audit.

The drift is real: a missing `dispatch.<provider>` entry stops `define-spec` cleanly but may silently route an `execute-plan` worker to `pi`, which can be the wrong CLI for the selected provider.

## Direction

Create a single canonical reference for general skill dispatch resolution, then update consumers to point to it.

Recommended location: `agent/skills/model-tier-resolution.md`.

The canonical document should cover:

1. Input file location: `~/.pi/agent/model-tiers.json`.
2. Expected JSON shape:
   - top-level tier keys such as `capable`, `standard`, `cheap`
   - optional/required `crossProvider` tiers where used
   - required `dispatch` object mapping provider prefixes to CLI names
3. Resolving a tier path (for example `capable`, `crossProvider.standard`, `crossProvider.cheap`) to a non-empty model string.
4. Extracting the provider prefix as the substring before the first `/`.
5. Looking up `dispatch.<provider>`.
6. The strictness policy and exact failure behavior.
7. Any documented fallback semantics, especially where a skill intentionally tries `crossProvider.capable` then `capable`, or where coordinator-only pi-routing uses `_shared/coordinator-dispatch.md`.

Recommended strictness policy: strict by default. Missing/unreadable `model-tiers.json`, missing/empty tier values, missing/empty `dispatch`, or missing/empty `dispatch.<provider>` should stop with a clear error rather than silently defaulting to `pi`. If any context intentionally needs lenient/default behavior, that exception should be named explicitly in the canonical doc and in the consuming skill.

## Acceptance criteria

- `agent/skills/model-tier-resolution.md` exists and is the canonical reference for tier path → model string → provider prefix → dispatch CLI resolution.
- The canonical doc defines the strictness policy and gives concrete failure messages or failure-message templates.
- `define-spec`, `generate-plan`, and `execute-plan` reference the canonical doc rather than duplicating the full provider-prefix / dispatch-map algorithm.
- Skill-local prose remains only where it describes which tier is used for which role, or explicit skill-specific fallback semantics.
- Strictness is consistent for general worker/reviewer dispatch: missing `model-tiers.json`, missing selected tier, missing `dispatch`, or missing `dispatch.<provider>` no longer silently defaults to `pi` unless the canonical doc explicitly names that context as an exception.
- `_shared/coordinator-dispatch.md` either references the canonical doc for the basic tier-path/provider/dispatch lookup primitives or clearly documents why coordinator dispatch has a special pi-only resolution procedure.
- A grep/audit for `dispatch[`, `dispatch.`, `provider prefix`, and `model-tiers.json` across `agent/skills/` shows no stale duplicated general-resolution algorithms; remaining matches are canonical-doc references, coordinator-specific rules, provenance validation, role-to-tier mappings, or explicit exceptions.

## Open questions

- Should the canonical doc live at `agent/skills/model-tier-resolution.md` or under a broader config location such as `agent/model-tier-resolution.md`? Current recommendation is `agent/skills/model-tier-resolution.md` because skills are the consumers.
- Should every general dispatch use strict policy immediately, or should any existing lenient/default-to-`pi` behavior be preserved as an explicitly documented exception? Current recommendation is strict everywhere except coordinator-specific pi-routing, which already has its own shared procedure.

## Coordinator dispatch requirement

The canonical resolution work must explicitly preserve and document the coordinator-dispatch invariant: coordinator agents that need to launch subagents themselves must be dispatched through the `pi` CLI, not through `claude` or any other CLI.

Coordinator agents currently include at least:

- `code-refiner` (used by `refine-code`)
- `plan-refiner` (used by `refine-plan`)

Reason: these coordinators rely on orchestration tools such as `subagent_run_serial` / `subagent_run_parallel` to dispatch reviewer, planner, or coder workers. Those tools are available to pi-backed coordinator sessions; a non-pi coordinator cannot reliably run the inner loop. If the coordinator cannot be launched on `pi`, the skill must fail loudly rather than perform an inline review/remediation fallback.

The canonical model-tier-resolution documentation should distinguish two dispatch cases:

1. **General worker/reviewer dispatch** — resolve the selected tier path strictly to a model string, derive provider prefix, require a non-empty `dispatch.<provider>` mapping, and use the mapped CLI.
2. **Coordinator dispatch** — resolve candidate coordinator tiers in the documented order, but only dispatch a candidate when its resolved `dispatch.<provider>` is `pi`; skip non-pi candidates silently as coordinator-ineligible, and hard-stop if no pi-eligible candidate succeeds.

`agent/skills/_shared/coordinator-dispatch.md` may remain as the detailed coordinator procedure, but it should either be referenced from `agent/skills/model-tier-resolution.md` or folded into that document so the pi-only coordinator rule is not hidden from the broader model-resolution policy.

Additional acceptance criteria for this coordinator requirement:

- The canonical doc explicitly states that coordinator agents must run via `pi` because they need subagent orchestration tools.
- `refine-code` and `refine-plan` continue to use the shared coordinator-dispatch procedure and do not duplicate or weaken the pi-only invariant.
- The hard-stop behavior for coordinator dispatch remains clear: no pi-eligible tier, or all pi-eligible tiers failing, stops the skill with a specific error; there is no inline review/remediation fallback.
- The grep/audit for model-tier resolution distinguishes allowed coordinator-specific matches in `_shared/coordinator-dispatch.md` from stale duplicated general worker/reviewer resolution algorithms.
