## Summary

Two distinct cleanups in the `define-spec → generate-plan → execute-plan` workflow that share a theme: the workflow already has principled mechanisms (path-based handoff, strict resolution-failure surfacing) that `execute-plan` and `generate-plan` quietly diverge from. Pulling them back into line removes a real context-pressure point and a real drift hazard.

## Item 1 — Path-based handoff for verifier command evidence

**Problem.** `execute-plan` Step 11.1 has the orchestrator collect orchestrator-side command evidence (exact command, exit code, stdout, stderr, with per-stream truncation) for every command-style `Verify:` recipe in a wave, then inline the assembled evidence blocks into each verifier dispatch as `{ORCHESTRATOR_COMMAND_EVIDENCE}`. For a wave with many tasks × many command-style recipes per task, this shuttles non-trivial stdout/stderr through the orchestrator's own context window — the same pressure point that `generate-plan` deliberately solved with path-based handoff (planner / plan-reviewer / planner-edit all read large artifacts from disk, never inlined).

This is the one place `execute-plan` inverts the design philosophy of the surrounding system.

**Direction.** Mirror the `generate-plan` shape:

- Have the orchestrator write evidence blocks to a per-wave directory (e.g. `.pi/runs/<plan-basename>/<wave-id>/evidence/<task-N>/criterion-<K>.txt`, or a single per-task evidence file). The exact path scheme is a design choice — sibling-of-plan or under a transient `.pi/runs/` tree are both reasonable.
- Replace `{ORCHESTRATOR_COMMAND_EVIDENCE}` in the verifier prompt with a path reference plus the verifier's existing `## Verifier-Visible Files`-style contract: a list of evidence file paths the verifier MUST read from disk before judging command-style criteria. The verifier already reads files; this just adds evidence files to that surface.
- Keep the truncation rule, but apply it at write time and let the verifier read the truncated artifact directly. Cite-by-block-number stays intact (the file naming gives the same handle).
- Decide whether evidence files are committed (probably not — they are run-transient artifacts, similar to `.pi/runs/` rather than `.pi/plans/reviews/`) and whether they survive past Step 16.
- Update `verify-task-prompt.md`, `agent/agents/verifier.md` input contract, and the relevant Step 11 prose so the path-based contract is the single source of truth.

**Why this matters.** On a plan with ~20 command-style recipes, the orchestrator currently has to hold all that command output in context on top of the diff context, the file list, and every other in-flight decision. Path-based handoff drops that load to a small list of paths.

## Item 2 — Single canonical model-tier resolution document

**Problem.** Model-tier + dispatch-CLI resolution from `~/.pi/agent/model-tiers.json` is restated in three places with subtly different strictness:

- `agent/skills/define-spec/SKILL.md` Step 3a — fails loudly and stops if `model-tiers.json` is missing/unreadable, if `capable` is missing/empty, or if `dispatch.<provider>` is missing/empty. Three explicit error messages, no CLI-default fallback.
- `agent/skills/generate-plan/SKILL.md` Step 2 — fails on missing `model-tiers.json` ("`generate-plan` requires `~/.pi/agent/model-tiers.json` — see model matrix configuration"), but is softer on the dispatch-CLI side. Step 4.1 has a cross-provider → same-provider fallback with re-resolution. The dispatch-resolution algorithm is referenced *to* `execute-plan` Step 6 ("see execute-plan Step 6 for the full resolution algorithm") rather than restated here.
- `agent/skills/execute-plan/SKILL.md` Step 6 — has the longest dispatch-resolution prose (provider-prefix extraction, dispatch map lookup, default to `"pi"`). Does not have a strict failure mode for missing `capable` or missing dispatch entries — the default-to-`"pi"` rule is a silent fallback rather than a stop.

The strictness divergence is real: a missing `dispatch.<provider>` entry stops `define-spec` cleanly but silently routes `execute-plan` workers to the pi CLI default, which may not be what the operator intended for a Claude-tier task.

**Direction.**

- Create `agent/skills/model-tier-resolution.md` (sibling doc, same shape as `agent/skills/execute-plan/integration-regression-model.md`) as the canonical reference. Cover: input file location, JSON shape expected, resolving a tier name to a model string, deriving the provider prefix, looking up `dispatch.<provider>`, the strict-vs-lenient policy (and which one applies where), the fallback policy for `crossProvider.capable` failures.
- Pick a single strictness policy and document it. Recommended: strict everywhere (matching `define-spec` Step 3a). The whole reason `model-tiers.json` exists is to keep dispatch routing explicit; silent fall-through to pi defeats that. If silent-fallback is genuinely correct in some context, document that context explicitly rather than letting it be implicit by omission.
- Replace the three in-skill restatements with a one-paragraph "see `agent/skills/model-tier-resolution.md`" reference plus only the skill-specific bits (which tier → which role, fallback semantics for plan review).
- Audit existing skills for any other places resolution logic is restated (this todo's review only walked the three orchestrator skills; refine-code, requesting-code-review, and the planner edit pass may also touch this).

**Why this matters.** Three near-copies of the same algorithm have already drifted in strictness. The next change to `model-tiers.json` shape (e.g. adding a new tier, changing the dispatch map structure) requires three coordinated edits with no review check that they actually agree. One canonical doc and three pointers eliminates that.

## Scope

- Item 1 changes: `execute-plan/SKILL.md` Step 11 (and references to `{ORCHESTRATOR_COMMAND_EVIDENCE}`), `execute-plan/verify-task-prompt.md`, `agent/agents/verifier.md` input contract. New evidence file path scheme decided during execution. Decide commit vs. transient.
- Item 2 changes: new `agent/skills/model-tier-resolution.md` (or wherever shared canonical workflow docs live — `integration-regression-model.md`'s placement under `execute-plan/` is plan-specific; this one is workflow-wide and probably belongs higher). Edits to `define-spec/SKILL.md` Step 3a, `generate-plan/SKILL.md` Step 2 + Step 4.1 fallback handling, `execute-plan/SKILL.md` Step 6. Audit of any other resolution sites.

Both items are independent — they can land in either order, or in the same plan if the dependency graph favors it.

## Non-goals

- Not refactoring the rest of `execute-plan/SKILL.md` beyond Step 11 and Step 6. The broader "execute-plan is too big" observation is real but outside this todo.
- Not touching the integration-regression model, retry-budget rules, wave-gate prose, or the Debugger-first flow.
- Not changing the verifier's per-criterion verdict format, the parser contract in Step 11.3, or the Acceptance Criteria authoring contract in `planner.md`.
- Not changing `model-tiers.json` itself.

## Acceptance criteria

### Item 1

- A new evidence-file path scheme is defined and documented (in the skill or in a sibling doc).
- `execute-plan/SKILL.md` Step 11.1 writes evidence to disk; Step 11.2 references the evidence-file paths instead of inlining `{ORCHESTRATOR_COMMAND_EVIDENCE}`.
- `verify-task-prompt.md` and `agent/agents/verifier.md` reflect the path-based contract — the verifier reads evidence files from disk per a `## Verifier-Visible Evidence` (or equivalent) section, and `## Verifier-Visible Files` continues to cover the file-inspection set.
- The truncation rule still applies and is enforced at write time. Truncation markers are still present in the evidence files (so the verifier can detect insufficient evidence and `FAIL` accordingly).
- The cite-by-block contract still works (`evidence: Evidence for Criterion N` or equivalent identifier resolves to a specific evidence file deterministically).
- A dry-run end-to-end on a small plan with at least one command-style recipe per task confirms evidence flows through disk and the verifier reaches the correct verdict.

### Item 2

- `agent/skills/model-tier-resolution.md` (or chosen path) exists and is the canonical reference for tier → model string → provider → dispatch CLI resolution, including strictness policy and `crossProvider.capable` fallback semantics.
- `define-spec/SKILL.md` Step 3a, `generate-plan/SKILL.md` Step 2 (and Step 4.1 fallback), and `execute-plan/SKILL.md` Step 6 each reference the canonical doc and contain only the skill-local specifics (tier-per-role mapping, fallback-on-failure prose).
- Strictness is consistent across all three skills — a missing `capable` field, a missing `dispatch.<provider>` entry, or an unreadable `model-tiers.json` produces the same shape of failure regardless of which skill triggered the resolution.
- A grep for `dispatch.` and `model-tiers.json` across `agent/skills/` returns no inlined resolution algorithms — only skill-local `which tier for which role` and pointers to the canonical doc.

## Open questions

- Path scheme for evidence files. Under `.pi/runs/<plan-basename>/<wave-id>/...`? Under `.pi/plans/reviews/...` (already exists)? Inside the worktree only, or somewhere shared? Leaning `.pi/runs/<plan-basename>/<wave-id>/evidence/<task-N>/criterion-<K>.txt` because it parallels `.pi/plans/reviews/` structurally but is run-transient — needs confirmation against existing conventions.
- Whether evidence files are committed or transient. Probably transient (gitignored) — they are debugging aids, not artifacts of record. The verifier verdict is what enters git history. Confirm during execution.
- Where the canonical model-tier-resolution doc lives. `agent/skills/model-tier-resolution.md` (workflow-wide) or `agent/model-tier-resolution.md` (config-wide, since it talks about `~/.pi/agent/model-tiers.json` directly)? Leaning `agent/skills/` because the skills are its consumers, but `agent/` is defensible if other non-skill prose ever needs to reference it.
- Whether to bring strictness up uniformly to `define-spec`'s level (recommended) or down to `execute-plan`'s default-to-`"pi"` level. The first option fails loudly on misconfiguration; the second is more permissive but risks silent misrouting. Leaning strict.
- Whether the audit will surface other resolution sites (refine-code, code-reviewer dispatch in `requesting-code-review`, the planner edit pass). Likely yes — should be in scope of Item 2's audit step.
