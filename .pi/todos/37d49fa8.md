## Problem

Model resolution is a hallucination trap. execute-plan Step 6 asks the agent to "detect which model family is available" and "resolve to latest version" using abstract tier tables. The agent doesn't have a reliable API to discover models at runtime — it's guessing model name strings.

The cross-provider rotation (Anthropic → OpenAI → Google → Anthropic) didn't match enabled providers, causing failed dispatches and silent fallbacks. We observed a Gemini Flash spec review failure on a trivial task during plan execution for TODO-51cb7cf7.

## Solution: Explicit tier map in settings.json

Add a `modelTiers` section to `~/.pi/agent/settings.json`:

```json
"modelTiers": {
  "capable": "anthropic/claude-opus-4-6",
  "standard": "anthropic/claude-sonnet-4-6",
  "cheap": "anthropic/claude-haiku-4-5",
  "crossProvider": {
    "capable": "openai-codex/gpt-5.4",
    "standard": "openai-codex/gpt-5.4-mini"
  }
}
```

No cheap cross-provider tier (nano not available, and per-wave spec review doesn't use cross-provider anyway).

## Changes

### 1. Add `modelTiers` to settings.json
- Add the tier map structure above to `~/.pi/agent/settings.json`

### 2. Update execute-plan Step 6 (model tier resolution)
- Replace the abstract tier table and "detect which model family" logic
- Instead: instruct the agent to read `modelTiers` from `~/.pi/agent/settings.json`
- Map each task's `cheap`/`standard`/`capable` recommendation directly to the tier map value
- No guessing, no pattern matching on model name strings

### 3. Update execute-plan Step 9 (per-wave spec review)
- Remove cross-provider rotation — use same-provider standard tier
- The spec reviewer model is just `modelTiers.standard`

### 4. Update execute-plan Step 12 (final code review)
- Use `modelTiers.crossProvider.capable` for the review dispatch
- On failure: fall back to `modelTiers.capable` (same provider) and **notify the user** that fallback occurred

### 5. Update generate-plan Step 3.5 (plan review)
- Use `modelTiers.crossProvider.capable` for the review dispatch
- On failure: fall back to `modelTiers.capable` (same provider) and **notify the user** that fallback occurred
- Remove the three-provider rotation logic and model string parsing

### 6. Remove dead logic
- Remove the tier resolution table from execute-plan Step 6
- Remove cross-provider rotation heuristics (Anthropic → OpenAI → Google → Anthropic) from all skills
- Remove model string parsing (`claude-*` → Anthropic, `gpt-*` → OpenAI, etc.)

## Design decisions
- **Tier map over derivation**: avoids hallucination-prone pattern matching on model name strings
- **Cross-provider at capable tier only** (plan review + final code review): per-wave spec review doesn't benefit enough to justify cross-provider friction
- **Notify on fallback**: user should know when cross-provider failed so they can investigate
- **New model versions**: update the tier map manually in settings.json — one place
- **New providers**: add manually to settings.json — the skill doesn't auto-discover providers
