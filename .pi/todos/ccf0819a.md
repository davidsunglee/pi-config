## Problem

Our `execute-plan` skill dispatches `plan-executor` workers with self-contained prompts but lacks several quality mechanisms that superpowers' `subagent-driven-development` skill includes. The main gaps:

1. **No worktree isolation** — execution happens in the main workspace. If a wave fails partway through, partial changes pollute the working tree. Superpowers *requires* `using-git-worktrees` before starting.

2. **No branch completion workflow** — Step 13 just moves the plan to `done/`. No option to merge, create PR, or discard. Superpowers ends with `finishing-a-development-branch`.

3. **No TDD enforcement** — worker prompts don't reference test-driven-development. Superpowers injects TDD into every implementer dispatch.

4. **No per-task spec compliance review** — our Step 9 checks acceptance criteria by reading output files, but doesn't dispatch an independent reviewer to verify the implementation matches the spec. Superpowers has a two-stage review (spec compliance *then* code quality) per task, catching issues before the next wave starts.

5. **No standardized implementer prompt** — superpowers bundles an `implementer-prompt.md` template with self-review instructions, escalation guidelines ("it's OK to say this is too hard"), code organization guidance, and structured status reporting. Our worker prompts are ad-hoc.

## Solution: Five enhancements to execute-plan

### Enhancement 1: Git worktree integration (using-git-worktrees tie-in)

Add a new **Step 0** (pre-flight) to execute-plan.

**Always ask** unless already on a feature branch or in a worktree:
- **(a) Execute in a worktree** (default) — invoke `using-git-worktrees` skill to create an isolated workspace, then execute the plan there. Suggest a branch name derived from the plan filename (e.g., `plan/2026-04-06-plan-review-step`).
- **(b) Execute in current workspace** — proceed as today.

**Auto-detect and skip:** If already on a feature branch or in a worktree, use the existing one. Display "Current workspace (on feature branch `xyz`)" in the settings summary and don't ask.

For option (a), the skill should:
1. Suggest a branch name derived from the plan filename
2. Delegate to `using-git-worktrees` for setup (directory selection, safety verification, project setup, baseline tests)
3. Continue execution in the worktree

**Note:** This is for *manual* worktree setup as described in the `using-git-worktrees` skill. It's separate from pi's built-in `worktree: true` parallel dispatch (which creates ephemeral worktrees per subagent).

### Enhancement 2: Finishing-a-development-branch integration

Replace the current Step 13 (which just moves the plan to `done/`) with:

After all waves complete and optional review passes:
1. **Always** move the plan file to `.pi/plans/done/` (this is plan-lifecycle, unconditional — the plan was executed regardless of what happens to the branch)
2. Invoke `finishing-a-development-branch` skill which presents the 4 options:
   - Merge back to base branch locally
   - Push and create a Pull Request
   - Keep the branch as-is
   - Discard

**Branch completion is offered even if review issues are pending** — the user may want to keep the branch and fix later, or create a PR with known issues noted.

**Only applicable when running in a worktree or feature branch** (Enhancement 1 option a, or if user manually started on a branch). If executing on main/master, skip branch completion and just report completion as today.

### Enhancement 3: TDD injection into worker prompts

When assembling worker prompts (Step 7), include a condensed TDD instruction block (~10 lines, not the full ~200-line skill):

> Follow test-driven development: write a failing test first, implement minimal code to pass, then refactor. No production code without a failing test. If the task includes test files, follow the red-green-refactor cycle for each step.

**Default: enabled.** Can be disabled via the settings confirmation (for non-code plans like documentation, config, content generation).

The condensed version captures the essential discipline without consuming excessive worker context. The full `test-driven-development` skill is too long to inject verbatim.

### Enhancement 4: Per-task spec compliance review

Add spec compliance review to Step 9 (currently "Verify wave output"):

After each wave, for each completed task:
1. **Current behavior (keep):** Read output files and verify against acceptance criteria.
2. **New: Dispatch spec compliance reviewer** — use a prompt template (`spec-reviewer.md`, new file in the execute-plan skill directory) that:
   - Receives the task's full spec (steps + acceptance criteria from the plan)
   - Receives the implementer's status report
   - Is instructed to **not trust the report** and verify by reading actual code/files
   - Reports: ✅ Spec compliant or ❌ Issues found (with file:line references)

If the spec reviewer finds issues, treat as a failure and apply Step 10 retry logic (re-dispatch the worker with the reviewer's findings).

#### Spec reviewer model selection

- **Always standard tier** — focused check, doesn't need capable-tier reasoning
- **Cross-provider if possible** — same rotation as the plan reviewer (Anthropic → OpenAI → Google → Anthropic), fallback to same provider
- Resolve using the execute-plan tier table (no hardcoded model strings)

#### Parallel vs sequential spec reviews

**Match the execution mode the user chose in Step 3:**
- If parallel execution → dispatch all spec reviews for the wave in parallel
- If sequential execution → review each task sequentially

**Default: enabled.** Can be disabled via the settings confirmation.

**This replaces the need for a separate code quality review per task** — the final code review (Step 12) already covers quality for the full implementation. Per-task spec compliance is the higher-value check because it catches "built the wrong thing" early.

### Enhancement 5: Standardized implementer prompt template

Create `implementer-prompt.md` in the execute-plan skill directory. Worker prompts (Step 7) are assembled by filling this template rather than being constructed ad-hoc.

The template should include (adapted from superpowers' `implementer-prompt.md`):

- **Task description** — `{TASK_SPEC}` placeholder (full text of task from plan)
- **Context** — `{CONTEXT}` placeholder (where this fits, dependencies, architectural context)
- **Working directory** — `{WORKING_DIR}`
- **Self-review checklist** — completeness, quality, discipline, testing (from superpowers)
- **Full escalation guidance** (~15 lines) — teaches workers *when* to escalate, not just the status codes. Includes:
  - "It is always OK to stop and say this is too hard. Bad work is worse than no work."
  - When to escalate: architectural decisions with multiple valid approaches, can't find clarity in provided code, uncertain about approach, task involves unanticipated restructuring, reading file after file without progress
  - How to escalate: BLOCKED or NEEDS_CONTEXT with specifics on what's needed
- **Code organization guidance** — follow file structure from plan, one responsibility per file, follow existing patterns
- **Report format** — structured status (DONE / DONE_WITH_CONCERNS / BLOCKED / NEEDS_CONTEXT) with what was implemented, tests, files changed, self-review findings
- **TDD block** (conditional) — included when Enhancement 3 is active

Full escalation guidance is preferred over minimal status codes because workers exhibit a real failure mode without it: they don't know it's OK to stop, so they keep reading files and guessing until they produce something wrong. The ~15 lines of context cost is negligible compared to a wasted retry cycle.

## Configuration UX: Single confirmation with defaults

Replace the current Step 3 (which asks 3 individual questions) with a **single settings confirmation**:

```
Execution settings:

  Workspace:        Worktree (recommended — creates feature branch)
  Execution:        Parallel, auto-continue unless failures
  TDD:              Enabled
  Per-task review:  Enabled (spec compliance after each wave)
  Final review:     Enabled (code quality after all waves)

Accept defaults? (y / customize)
```

- **`y`** → go with all defaults, one interaction
- **`customize`** → ask each question individually (current behavior, expanded)
- Auto-detected situations skip their line as a choice (e.g., already on feature branch → "Workspace: Current workspace (on feature branch `xyz`)")

**Defaults (all enabled):**

| Setting | Default | Can disable for |
|---------|---------|-----------------|
| Workspace | Worktree | Already on branch/worktree (auto-detected, not a choice) |
| Execution | Parallel, auto-continue unless failures | User preference |
| TDD | Enabled | Non-code plans (docs, config, content) |
| Per-task review | Enabled | Small/simple plans where speed matters more |
| Final review | Enabled | User preference |

## Key design decisions

- **Per-task spec review, not per-task code quality review** — code quality review per task (as superpowers does) doubles review cost. Spec compliance per task + final code quality review is a better tradeoff for parallel wave execution where you want fast feedback on "did you build the right thing?"
- **Spec reviewer is standard-tier cross-provider** — focused check doesn't need capable-tier, but cross-provider avoids same-model blind spots.
- **Spec review parallelism matches execution mode** — parallel execution → parallel reviews, sequential → sequential reviews.
- **Worktree default on, auto-skip when redundant** — always suggest worktree isolation, but auto-detect if already on a branch/worktree and skip the question.
- **Branch completion offered even with pending issues** — user may want to keep branch and fix later, or PR with known issues.
- **Plan moves to done unconditionally** — plan-lifecycle (executed) is separate from branch-lifecycle (merge/PR/keep/discard).
- **TDD opt-in with default enabled** — not all plan tasks involve code (docs, config, content). Let the user disable for non-code plans.
- **Condensed TDD, not full skill injection** — the full `test-driven-development` skill is ~200 lines. A condensed ~10-line version captures the essential discipline without wasting worker context.
- **Full escalation guidance in implementer template** — the "when to escalate" list prevents workers from silently producing bad output. ~15 lines of context is negligible vs. a wasted retry cycle.
- **Single confirmation UX** — 6 individual questions is tedious. Show recommended defaults, let user accept with `y` or drill into `customize`. Fast path is one interaction.

## Inspiration

Adapted from [superpowers subagent-driven-development](https://github.com/obra/superpowers/tree/main/skills/subagent-driven-development) by Jesse Vincent. Key adaptations:
