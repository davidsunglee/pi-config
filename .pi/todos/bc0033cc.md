## Summary
Dispatching standard workflow agents with `agentScope: "project"` can fail in this repo because project-scope agent resolution exposes a different visible name set than the expected workflow agents (`coder`, `plan-reviewer`, `code-reviewer`, `verifier`, etc.). This caused `execute-plan` orchestration to fail when `coder` was requested under project-only scope and the runtime reported only `planner`, `reviewer`, `scout`, and `worker`.

## Goal
Make this failure mode hard or impossible to hit again.

## Problem details
- `~/.pi/agent/agents/` contains the standard workflow agents, including `coder.md`.
- `agent/agents/` also contains `coder.md` and related agents.
- But dispatching with `agentScope: "project"` did not expose `coder`; the runtime instead advertised `planner`, `reviewer`, `scout`, and `worker`.
- Default/user scope and `agentScope: "both"` both worked for `coder`.

## Desired outcomes
At least one of the following should happen:
- Standard workflow dispatches never use `agentScope: "project"` unless explicitly required.
- A preflight check verifies that the intended agent names are available under the chosen scope before execution starts.
- If project-only scope is requested and the target agent is unavailable, orchestration falls back automatically to default scope or `agentScope: "both"`.
- Documentation and/or skills clearly encode the safe rule so future sessions do not regress.

## Candidate scope
- Investigate why project-only scope exposes `worker/reviewer/scout` instead of the expected workflow names.
- Decide whether to fix configuration, skill guidance, or subagent-dispatch behavior.
- Add a durable guardrail in the workflow.

## Acceptance criteria
- We have a documented and enforceable rule or mechanism that prevents this exact failure from recurring.
- Future execute-plan style dispatches of `coder` do not fail solely because `agentScope: "project"` was chosen.
