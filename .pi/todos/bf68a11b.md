## Resolution

All three scope items are implemented at HEAD.

- **§1 (verify-recipe AC format):** `review-plan-prompt.md` treats missing or placeholder `Verify:` recipes as Error-severity findings (lines 64-66, 87, 105); `edit-plan-prompt.md` adds concrete recipes when findings call them out (items 6-7); execute-plan Step 10 refuses to dispatch the verifier when any AC lacks a recipe. `generate-plan-prompt.md` itself does not enforce the recipe up-front, but the review→edit→execute pipeline covers it end-to-end — functional outcome achieved.
- **§2 (verify-task subagent):** `agent/skills/execute-plan/verify-task-prompt.md` exists. Fresh-context judge-only agent; input includes task spec, AC-with-recipes, orchestrator-collected command evidence, verifier-visible file set, and diff context; output is per-criterion `[Criterion N] PASS|FAIL` plus overall `VERDICT: PASS|FAIL`. Model tier defaults to `standard`, upgraded to `capable` when the task ran at `capable`.
- **§3 (wiring):** execute-plan Step 10 dispatches a fresh `verifier` subagent per task, in parallel up to `MAX_PARALLEL_TASKS`. `VERDICT: FAIL` (including malformed output treated as `FAIL`) routes to Step 12's retry loop under the shared 3-retry budget. Preconditions preserved: Step 9.5 must exit before Step 10, and Step 10 must pass before Step 11.

Out-of-scope items remain correctly deferred: no `validate-plan` schema gate, no forcing AC into executable-assertion shape.

## Summary

The real gap surfaced by the 2026-04-17 workflow evaluation is that execute-plan Step 10 is orchestrator self-audit against free-form prose AC — the weakest gate in the pipeline. This todo addresses that gap without adopting the full rigidity of the original items 2 and 3, and defers item 1 entirely.

## Recommendation rationale (why this scope, not the original 1/2/3)

- **Item 1 (`validate-plan` schema gate): defer.** Execute-plan Step 2's section check plus the generate-plan review-edit loop already catch malformed plans before execution. There is no observed failure of malformed plans breaking wave ordering or todo auto-close in practice. Build a validator reactively when a concrete failure proves the gap, not speculatively.
- **Item 2 (machine-checkable AC): soften.** Forcing every AC into test / exit-code / regex shape over-constrains plans that touch skills, prompts, and docs — which is the dominant workload in this repo. Keep AC format-flexible but require each AC to include a *verification recipe*: a concrete instruction for how to check it (command to run, file+pattern to inspect, behavior to exercise). Prose is fine as long as the recipe is reproducible by a fresh reader.
- **Item 3 (`verify-task` subagent): keep, and decouple from item 2.** A fresh-context verifier reading AC + produced files + diff is strictly better than the orchestrator grading its own wave, even when AC stays prose. It does not require item 2's rigid format to be useful.

## Scope

### 1. Update AC format (soften item 2)

- Update `generate-plan-prompt.md` and the plan template to require each acceptance criterion to include a **verification recipe** line, e.g.:
  - `Verify: run <command>, expect exit 0`
  - `Verify: grep "<pattern>" in <file>, expect ≥1 match`
  - `Verify: file <path> contains section "## Foo" with subsections X, Y`
  - `Verify: read <path> and confirm <prose description of what to look for>`
- The recipe does not need to be executable — it just needs to be specific enough that a fresh reader (human or agent) can reproduce the check without re-deriving intent.
- Update `review-plan-prompt.md` so the plan-reviewer flags AC without a verification recipe as an Error severity finding.
- Update `edit-plan-prompt.md` so the edit pass adds recipes when findings call them out.

### 2. Add `verify-task` subagent (item 3)

- New subagent prompt under `agent/skills/execute-plan/verify-task-prompt.md` (or equivalent).
- Input contract: task spec (name, Files, AC with recipes), the produced/modified files by path, the wave diff range.
- Output contract: per-AC verdict (`PASS` / `FAIL`) with evidence quoted from files or command output; overall `VERIFIED` / `NOT_VERIFIED` status.
- Fresh context per verification — not reusing coder or orchestrator state.
- Model tier: `standard` default; `capable` if task tier was `capable`.

### 3. Wire verification into execute-plan

- Replace Step 10's orchestrator self-audit with `verify-task` dispatch per task.
- Dispatch pattern: per-task (not per-wave), in parallel with remaining coders or pipelined as each coder returns `DONE`.
- On `NOT_VERIFIED`: route to the existing Step 12 retry logic (counts toward the 3-retry budget shared with Step 9.5).
- Keep Step 10's precondition (Step 9.5 must have exited) and Step 11's precondition (Step 10 must have passed) intact.

## Out of scope

- Rigid schema-level validator for the plan file (original item 1). Revisit only if a concrete failure surfaces.
- Replacing prose AC wholesale with executable assertions. Recipes can be prose as long as they are specific.
- Changes to Step 9.5 (blocked-task gate), Step 11 (post-wave commit/integration tests), or the final refine-code review loop.

## Size

Medium. Three coordinated edits: AC format in generate-plan prompts, new verify-task subagent, Step 10 rewrite in execute-plan.

## Dependencies

None hard. The AC format change (§1) and the verifier (§2) can ship independently; §3 requires both merged to be end-to-end useful, but the verifier works on prose-recipe AC from day one.
