# High-level spec: move `execute-plan` orchestration into a pi extension

Derived from: TODO-8dc863eb
Source exploration todo: TODO-8dc863eb

## Goal

Replace the most mechanical, stateful, failure-prone parts of the current prose-driven `execute-plan` workflow with a code-backed pi extension, while preserving agent-driven judgment where LLM reasoning is actually valuable.

The desired result is parity with the current workflow plus better reliability, resumability, observability/debugging, and user experience.

## Problem statement

Today, `execute-plan` is effectively a large state machine encoded in prose. That works when the orchestrating model follows instructions perfectly, but the system remains fragile in predictable ways:

- skipped procedural steps
- lost or inconsistently reconstructed state between waves
- incorrect or hallucinated model resolution
- brittle retry handling
- weak resumability after interruption or failure
- poor observability into what happened and why
- UX limitations from keeping bookkeeping in prompts instead of code

The goal is not to remove agentic execution. The goal is to move orchestration mechanics into code and leave judgment-heavy decisions with agents.

## Deliverable

A new pi extension that implements `execute-plan` orchestration as shared code, exposed through:

1. a direct user-facing command / command-like entry point
2. an agent-callable tool/API surface
3. a thin skill wrapper so the agent can still discover and apply the workflow naturally

All entry points must share the same core implementation rather than duplicating orchestration logic.

## Core design principle

**Code orchestrates; agents judge.**

### Move into code

The extension should own the parts that are procedural, stateful, mechanical, and easy to verify:

- plan loading and structural validation
- extracting optional test command
- dependency parsing and wave computation
- model tier resolution from settings
- prompt/template loading and placeholder filling
- worker dispatch bookkeeping
- retry counting and retry state
- workspace/worktree bookkeeping
- per-wave commit mechanics
- test execution and baseline comparison
- review dispatch plumbing
- persistence of run state
- resume behavior
- cancellation handling
- progress reporting / TUI / status surfaces

### Keep agent-driven

The extension should continue delegating judgment-heavy work to agents/subagents:

- implementation of plan tasks via `plan-executor`
- interpretation of `DONE_WITH_CONCERNS`
- interpretation of `BLOCKED` / `NEEDS_CONTEXT`
- content-level verification against task acceptance criteria
- spec-compliance review reasoning
- final code review reasoning
- deciding whether a concern is acceptable or needs follow-up

The extension may encode routing heuristics, but should not try to replace LLM judgment with brittle deterministic rules where the task is inherently interpretive.

## Scope

### In scope

- Replace prose orchestration of the current `execute-plan` workflow with extension code
- Preserve the existing plan-file-based workflow shape
- Preserve subagent-based task execution
- Preserve review as part of the workflow, but make review modules reusable
- Persist enough state to resume after interruption
- Improve debuggability and transparency of execution
- Support defaults/config-driven execution while preserving interactive customization
- Design shared primitives that other workflows, especially `generate-plan`, may later reuse

### Out of scope / non-goals

- Replacing `plan-executor`
- Replacing subagents as the execution model
- Redesigning the plan format
- Building a standalone external CLI as the primary solution
- Building for multi-user/team concurrency in v1

## Entry points

The extension should expose one shared orchestration engine through multiple surfaces:

### 1. Direct invocation

A user-facing command should allow users to run plan execution directly.

### 2. Agent-callable tool/API

The agent should be able to invoke the same extension when it determines the workflow is relevant.

### 3. Skill wrapper

The existing `execute-plan` skill should become a thin wrapper or discovery layer that routes into the extension rather than re-encoding orchestration logic in prose.

## User interaction model

The extension should preserve the current "good defaults + optional customization" model:

- users can start with defaults/config
- users can customize execution settings before starting
- the extension should present settings clearly before execution

When the agent chooses the extension autonomously, it should require user confirmation only when risky or destructive choices are involved, rather than always requiring confirmation or always auto-starting blindly.

## Workflow coverage

The extension should own the full `execute-plan` workflow, not only task dispatch.

That includes:

- pre-flight and workspace handling
- plan validation
- settings confirmation/customization
- output collision checks
- wave computation
- model resolution
- baseline test capture
- wave execution
- handling worker statuses
- post-wave verification
- per-wave spec-compliance review
- post-wave commits and integration tests
- retry/skip/stop handling
- final code review
- completion bookkeeping (move plan, close linked todo, report summary)

Review is part of orchestration and should remain in scope, but review machinery should be modular enough to be invoked independently by other workflows.

## Shared primitives to design intentionally

Even if this effort only implements `execute-plan`, the design should factor out reusable primitives that other workflows can later share.

Candidate shared primitives:

- plan file parser / validator
- dependency graph + wave builder
- model tier resolver
- prompt-template loader/filler
- run-state persistence layer
- review-dispatch helper(s)
- git/worktree helper(s)
- test baseline + comparison helper(s)
- execution reporting/progress primitives

These primitives should make later migration of adjacent workflows easier, especially plan review and other orchestration-heavy tasks.

## State persistence

Persist execution state in a dedicated run-state area, e.g. `.pi/plan-runs/...`, rather than embedding state into the plan file itself.

### Persisted state must include

- selected execution settings
- workspace/worktree path
- resolved waves
- per-task dispatch history
- retry counts
- per-wave verification results
- test baseline
- review outputs / review artifact paths
- current execution status and last completed wave

This state should be durable enough to support interruption, inspection, and resume.

## Resumability

V1 resumability target:

- support resume from the **last completed wave**
- do not require arbitrary checkpoint resume in v1
- state should make it obvious why the previous run stopped and what remains

The extension should explicitly detect resumable runs and offer the user a clear resume path.

## Cancellation behavior

Cancellation should support both modes:

1. stop after the current wave finishes
2. stop after the current task finishes

The extension should make the active cancellation mode visible and persist cancellation state in run metadata.

## Concurrency model

V1 should assume **one active `execute-plan` run per repository**.

It is acceptable to reject or defer starting another run while one is active, in exchange for simpler state handling and fewer concurrency hazards.

## Handling worker statuses

For `BLOCKED` and `NEEDS_CONTEXT`, the extension should follow the Step 8 heuristics currently documented in `execute-plan`:

- provide missing context and re-dispatch when the problem is contextual
- re-dispatch with a more capable model when the problem is reasoning-related
- split work when the task is too large
- escalate to the user when the plan is fundamentally wrong or human judgment is required

This decision logic should be implemented explicitly in code as orchestrator policy rather than being left as prose-only instruction.

## Reviews and checks

### Per-wave spec compliance review

Keep this capability in the design, because it may catch worker/spec drift before it compounds across waves.

However, it should be:

- modular
- configurable
- enabled by default initially

If later evidence shows it does not pull its weight, it can be relaxed, but the initial design should preserve it.

### Final code review

Final code review should remain mandatory by default.

It should remain part of the orchestration workflow and should continue to support cross-provider review where configured.

If the final review finds important or critical issues, the workflow should:

1. present the findings clearly
2. offer to launch fix-up tasks
3. not silently proceed as though the run is complete and ready to merge without surfacing those issues

## Settings and policy

The extension should preserve current execution-time configurability, including the ability to run with defaults or customize:

- workspace choice
- execution mode / pacing
- TDD on/off
- spec check on/off
- checkpoint commits on/off
- integration tests on/off
- test command override
- final review behavior (subject to mandatory-by-default policy)

The extension should support config-backed defaults while still providing interactive confirmation/customization.

## Observability and UX goals

The extension should improve visibility into execution by making it easy to inspect:

- current wave and task status
- what model was resolved for each task/review
- retries taken and why
- latest worker status/result
- verification outcomes
- test baseline and regressions
- review artifacts
- why execution stopped, paused, or is resumable

The user should not have to reconstruct orchestration state from scattered prompt history.

## Compatibility target

The extension should aim for behavioral parity with the current `execute-plan` workflow where that workflow is already valuable, while removing as much orchestration fragility as possible.

This is not intended as a reduced-scope v1 unless implementation reality forces it. The target is:

**parity plus improved UX/resume/debugging from day one**

## Success criteria

This spec should eventually produce an implementation that:

- preserves the current agentic workflow shape
- removes most of the mechanical orchestration burden from prose
- makes runs durable and resumable
- makes execution state inspectable
- reduces skipped-step and lost-state failure modes
- preserves or improves review quality
- remains pleasant to invoke directly and easy for the agent to apply automatically

## Open design preference captured here

The current expectation is that `execute-plan` remains the primary orchestrated workflow, while adjacent workflows may later reuse shared orchestration/review primitives. The design should therefore optimize for modularity without requiring an all-at-once rewrite of other skills first.
