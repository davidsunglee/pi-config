## Overview

Decouple the footer extension's color values from theme token aliases. Replace the current `FIELD_COLORS` object (which uses theme token names like `"accent"`, `"dim"`) with a `THEME_COLORS` map that stores direct color values per theme. Remove `FIELD_COLORS` entirely.

## Color Fields (11 user-configurable + 1 fixed)

Split the current `model` field into three:

| Key | What it colors | Notes |
|---|---|---|
| `modelName` | Model name (e.g. `claude-sonnet-4`) | Was part of `model` |
| `thinkingMode` | Thinking level (e.g. `• high`) | Was part of `model` |
| ~~`modelProvider`~~ | Provider prefix (e.g. `(anthropic)`) | **Always dim** — not user-configurable, use `theme.fg("dim", ...)` |
| `tokens` | Input & output token counts | Unchanged |
| `cost` | Cost value / subscription indicator | Unchanged |
| `cache` | Cache read & write counts | Unchanged |
| `contextUsage` | Context % when ≤70% (escalates to warning/error) | Unchanged |
| `contextWindow` | Context window capacity | Unchanged |
| `branch` | Git branch name | Unchanged |
| `pwd` | Working directory path | Unchanged |
| `statuses` | Extension status texts | Unchanged |
| `symbols` | All glyphs: ↑ ↓ R W $ / % • and parens | Unchanged |

The 11 user-configurable keys form the `FooterColors` type. `modelProvider` stays hardcoded to `theme.fg("dim", ...)`.

## Color Value Format

Accept any of:
- **Named ANSI colors:** `"red"`, `"green"`, `"yellow"`, `"blue"`, `"magenta"`, `"cyan"`, `"white"`, `"gray"`/`"grey"`, and bright variants (`"brightRed"`, etc.)
- **Hex strings:** `"#ff8800"`
- **ANSI 256 color numbers:** `208`

When pre-populating defaults, use **named ANSI colors**.

## Configuration Structure

A single `THEME_COLORS` map at the top of the extension file, keyed by theme name:

```ts
type FooterColors = {
  modelName: string | number;
  thinkingMode: string | number;
  tokens: string | number;
  cost: string | number;
  cache: string | number;
  contextUsage: string | number;
  contextWindow: string | number;
  branch: string | number;
  pwd: string | number;
  statuses: string | number;
  symbols: string | number;
};

const THEME_COLORS: Record<string, Partial<FooterColors>> = {
  // User adds entries here to override defaults for specific themes
  // "dracula": { modelName: "magenta", cost: "yellow" },
};
```

- **Partial overrides allowed** — missing keys fall back to auto-populated defaults.
- `modelProvider` is excluded from this type (always dim).

## Default Pre-population (Runtime)

Use `getResolvedThemeColors(themeName)` from the theme module — it returns `Record<string, string>` with all tokens resolved to hex values. Map each footer field to its corresponding theme token:

| Field | Theme token | Rationale |
|---|---|---|
| `modelName` | `accent` | Matches current FIELD_COLORS |
| `thinkingMode` | `accent` | Matches current FIELD_COLORS |
| `tokens` | `muted` | Matches current FIELD_COLORS |
| `cost` | `warning` | Matches current FIELD_COLORS |
| `cache` | `muted` | Matches current FIELD_COLORS |
| `contextUsage` | `muted` | Matches current FIELD_COLORS |
| `contextWindow` | `dim` | Matches current FIELD_COLORS |
| `branch` | `success` | Matches current FIELD_COLORS |
| `pwd` | `dim` | Matches current FIELD_COLORS |
| `statuses` | `dim` | Matches current FIELD_COLORS |
| `symbols` | `borderMuted` | Matches current FIELD_COLORS |

Merge logic: `{ ...resolvedDefaults, ...THEME_COLORS[themeName] }` — user values win.

## Applying Colors in Render (the resolved open question)

`theme.fg()` only accepts `ThemeColor` token names, so we **cannot** pass hex/named/256 values through it. Instead:

### New helper: `applyColor(value: string | number, text: string, mode: "truecolor" | "256color"): string`

Handles all 3 formats:
- **Named ANSI** → lookup table to standard SGR codes (e.g. `"red"` → `\x1b[31m`, `"brightCyan"` → `\x1b[96m`)
- **Hex** → truecolor: `\x1b[38;2;r;g;bm` / 256color: downconvert via weighted RGB distance (replicate the theme module's `hexTo256` logic)
- **Number** → `\x1b[38;5;Nm`
- All reset with `\x1b[39m` (foreground reset only, matching `theme.fg()` behavior)

Get color mode via `theme.getColorMode()`.

### Special cases that still use `theme.fg()` directly:
- `modelProvider` → `theme.fg("dim", ...)`
- Context usage escalation (>70% warning, >90% error) → `theme.fg("warning", ...)` / `theme.fg("error", ...)`

## Theme Switch Behavior

When the user switches themes mid-session:
1. `invalidate()` fires
2. Re-call `getResolvedThemeColors()` for the new theme to get fresh defaults
3. Look up new `theme.name` in `THEME_COLORS`
4. Merge: user overrides on top of resolved defaults → store as the active color set
5. Next `render()` uses the merged colors via `applyColor()`

Access the current theme name via `theme.name` (available on the Theme class).

## What to Remove

- Delete the existing `FIELD_COLORS` object entirely
- Replace all `theme.fg(FIELD_COLORS.xxx, ...)` calls with `applyColor(activeColors.xxx, ..., mode)` (except `modelProvider` and context escalation)

## Implementation Notes

- `getResolvedThemeColors` is exported from `@mariozechner/pi-coding-agent` theme module. Import: `import { getResolvedThemeColors } from "@mariozechner/pi-coding-agent"` (verify exact export path).
- The hex→256 downconversion logic should replicate `hexToRgb` + `rgbTo256` from the theme module (uses weighted Euclidean distance with 6×6×6 cube and 24-step grayscale ramp).
- Named ANSI color table (16 standard colors): `black`=30, `red`=31, `green`=32, `yellow`=33, `blue`=34, `magenta`=35, `cyan`=36, `white`=37, `gray`/`grey`=90, `brightRed`=91, `brightGreen`=92, `brightYellow`=93, `brightBlue`=94, `brightMagenta`=95, `brightCyan`=96, `brightWhite`=97.
