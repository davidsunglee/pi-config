## Problem

Every subagent in the workflow starts with fresh context and independently discovers codebase structure. The planner (opus) does the deepest exploration — following imports, tracing data flow, understanding interfaces — and this is the most expensive read work in the pipeline. Coders in the same wave often re-read overlapping files (shared interfaces, shared types). If define-spec is added, it will also need codebase reads. Nobody shares notes.

## Proposal

Add an optional **scout agent** that runs on a cheap model (haiku) before plan generation. Given a todo/spec, the scout performs task-scoped codebase reconnaissance and produces a structured **codebase brief** that downstream agents consume instead of doing their own exploratory reads.

The scout is not a whole-codebase indexer. It's scoped to the task at hand — find the files, interfaces, dependency chains, patterns, and conventions relevant to this specific piece of work.

## Where it fits in the pipeline

```
todo → [scout] → [define-spec] → generate-plan → execute-plan
```

All three bracketed stages are optional. The scout's output (codebase brief) flows into define-spec and/or the planner:

- **define-spec** consumes the brief to ground spec writing in actual codebase reality
- **planner** consumes the brief to skip expensive exploratory reads and focus on plan structure and task decomposition
- **coders** may optionally receive relevant excerpts from the brief in their task prompts, but this is lower-value since they read current files directly

The scout does NOT re-run between execution waves. By wave 2, wave 1 has changed files, making the brief stale. Coders already get file lists in their task prompts and read on cheaper models.

## What the scout produces

A structured codebase brief, written to a file at `.pi/briefs/TODO-<id>-brief.md` (keyed by todo ID for deterministic lookup by downstream consumers like define-spec and generate-plan). Contents:

1. **Relevant files** — files directly related to the task, with one-line descriptions of what each does
2. **Key interfaces and types** — the contracts that the task will need to understand or modify
3. **Dependency graph** — how the relevant files relate to each other (imports, calls, implements)
4. **Patterns and conventions** — coding patterns, naming conventions, error handling approaches used in the affected area
5. **Test coverage** — existing tests for the affected code, test patterns used
6. **Potential risk areas** — files or interfaces that are widely depended on and dangerous to modify

## Design questions

### 1. When should the scout run?

Options:

- **Always before planning.** Simple, but adds latency and cost for well-scoped tasks where the relevant files are already known.
- **Only when the todo/spec doesn't identify files.** Skip the scout if the input already names specific files and modules. Requires a heuristic to judge "does this input have enough codebase context?"
- **User-triggered.** Explicit opt-in, like define-spec. Least friction for simple tasks but requires the user to judge when it's worthwhile.

### 2. Is haiku good enough?

The scout needs to:
- Follow imports and trace dependency chains — haiku can do this reliably
- Identify patterns and conventions — less certain at haiku tier
- Characterize design intent of interfaces — risky at haiku tier

A scout that mischaracterizes an interface wastes more planner time than no scout at all. Options:
- **Haiku with structured output constraints** — force the scout to report what it literally sees (file contents, import statements) rather than interpret design intent
- **Sonnet for the scout** — more expensive but more reliable interpretation. Still much cheaper than opus doing the same reads.
- **Haiku with a confidence signal** — scout flags areas where it's uncertain, planner knows to verify those

> **Note from define-spec design (2026-04-15):** Both define-spec and the planner consume the scout brief as trusted input. The brief needs to include judgment-heavy content (patterns, conventions, risk areas) for these consumers to ask informed questions and skip their own exploration. This argues for sonnet over haiku — the cost difference is small relative to the cost of a bad plan from a mischaracterized brief.

### 3. How does the planner consume the brief?

Options:
- **Injected into the planner prompt** alongside the todo/spec. Simple, but adds tokens to an already large prompt.
- **Referenced as a file** that the planner reads. Keeps the prompt lean, but the planner might not weight it correctly.
- **Replaces the planner's exploratory phase.** The planner prompt says "a scout has already mapped the relevant codebase — use this brief instead of doing broad file discovery." Most token-efficient, but requires the planner to trust the scout's work.

### 4. What's the codebase size threshold?

For a 20-file project, everyone can just read everything — the scout adds cost without saving any. The scout earns its place in larger codebases where the planner would otherwise spend significant tokens on exploration. Is there a heuristic (file count, repo line count, or just user judgment)?

### 5. Brief format and staleness

- Should the brief include literal file contents (expensive but self-contained) or just summaries with paths (cheap but requires downstream agents to still read files)?
- Should the brief have an explicit "generated at" timestamp and git SHA so downstream agents know its freshness?

### 6. Is the brief required input or advisory? (merged from `7e1fa3d2` item 8)

Generate-plan today verifies the scout brief file exists and passes the path to the planner, but the planner is not required to read it. "Best non-obvious context" runs on the honor system. Pick one:

- **(A) Required.** Enforce in the planner prompt ("you MUST read and incorporate the scout brief"), and add a reviewer check ("did the plan incorporate brief findings?"). Makes the brief load-bearing.
- **(B) Advisory — drop the field.** Remove the `Scout brief:` line from spec/plan preamble, simplify define-spec and generate-plan accordingly. Appropriate if briefs are aspirational in practice.

Recommend A if scout briefs are genuinely load-bearing, B if they're aspirational. Size: small either way. This question should be answered as part of the overall scout design — it's the contract half of the brief format decision in Q5.

## Token cost estimate

Rough comparison for a medium-complexity task in a large codebase:

| Approach | Exploratory reads | Model | Estimated cost |
|----------|------------------|-------|---------------|
| No scout (current) | Planner does all discovery | opus | 150-200K input tokens at opus pricing |
| With scout | Scout does discovery, planner consumes brief | haiku + opus | ~100K haiku + ~80K opus (shorter planner prompt, less exploration) |

The savings come from shifting exploratory reads from opus to haiku. The planner still reads the brief and the task-critical files directly, but skips the broad "follow every import" phase.
