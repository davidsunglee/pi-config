Currently the `execute-plan` skill leaves the caller/coordinator doing substantial work beyond dispatching coder subagents for task execution and verifier subagents for task verification. For example, the caller runs integration tests at the end of waves, and there may be other heavyweight operations embedded in the main-agent procedure.

Goal: audit `execute-plan` and move heavier operations into dedicated subagents where practical, especially integration test execution, so the caller primarily orchestrates/dispatches and handles high-level decisions.

Notes:
- Preserve existing verification guarantees and failure reporting.
- Identify other non-trivial caller-side work besides integration tests.
- Consider whether to reuse verifier agents or introduce a dedicated integration/regression verification prompt/agent flow.
- Update docs/prompts/tests as needed.

Additional scope constraint:
- The caller/coordinator for the `execute-plan` skill should not personally take on work that involves writing/editing source code or debugging problems.
- If execution, verification, integration tests, or review reveals code changes or debugging are needed, that work should be delegated to an appropriate subagent instead of being performed directly by the caller.
- The caller should remain responsible for orchestration, dispatch, aggregation, and user-facing decisions—not implementation/debugging.

Additional investigation item:
- Look into why, when the hybrid re-review fails, the flow continues to perform a final verification review.
- Desired behavior: if the hybrid re-review fails, skip final verification review and go straight back to remediation, continuing the remediation/re-review loop.
- Ensure the execute-plan/refinement flow does not spend caller or verifier effort on final verification when re-review has already identified unresolved issues.
