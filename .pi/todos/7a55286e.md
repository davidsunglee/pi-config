## Summary
`agent/skills/execute-plan/` is a monolith (SKILL.md ~748 lines plus ~5 supporting docs) that bundles wave orchestration, per-task verification, the three-set integration regression model, the retry/intervention state machine, refine-code invocation, and branch finalization. That surface is the workflow's widest blast radius for its own bugs and the hardest part to change safely. Shrinking it — by extracting subsystems into their own skills (where the shape warrants a skill) or into code modules (where it does not) — would reduce cognitive load, make each subsystem individually testable, and let other skills reuse pieces without pulling in the whole orchestrator.

## Goal
Refactor `execute-plan` into a thin orchestrator that composes smaller, focused skills and local modules. The orchestrator should coordinate the sequence; the subsystems should own their own rules, inputs, and outputs.

## Extractions

Not every subsystem is skill-shaped. Skills describe procedures for an agent to follow (subagent dispatch, prompt contracts, judgment calls, policy). Algorithms and bookkeeping belong in code modules that the orchestrator calls. The split below reflects that.

### New standalone skills

- **Verifier protocol** — evidence collection (`Verify:` recipes, truncation rules), per-criterion PASS/FAIL parsing, protocol-error handling on malformed output, visible-file scoping, verifier-subagent dispatch contract. Skill-shaped: it's a subagent-dispatch procedure with a structured output contract and explicit failure-mode policy.
- **Wave dispatcher** — dependency-ordered wave construction from a plan, parallel coder dispatch within a wave (bounded by pi-subagent MAX_PARALLEL_TASKS), pacing modes (always-pause / auto-continue / pause-on-failure), response classification (DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED). Skill-shaped because the dispatch-and-classify loop is the procedure, and other orchestrators (not just `execute-plan`) could plausibly want it.
- **Branch / worktree pre-flight** — Step 0 detection logic: is-git-repo, on-worktree vs. main/develop, feature-branch reuse with dirty-workspace warning, slug derivation, new-worktree fallback. Skill-shaped as a self-contained pre-flight procedure any workflow entering a branch-scoped change could invoke.

### Code modules (not skills)

- **Integration regression model** — the three-set (`baseline_failures`, `deferred_integration_regressions`, `new_regressions_after_deferment`) reconciliation algorithm. Core is set algebra on test results; belongs in a typed module. The thin policy layer around it (defer menu on intermediate waves, final-gate blocking) stays in `execute-plan` as orchestration, calling into the module.
- **Retry / intervention state machine** — shared retry budget across Blocked / DONE_WITH_CONCERNS / Verifier FAIL, sub-task split inheritance, budget reset on user-triggered "retry again." This is bookkeeping with decision rules, not agent procedure. Extract as a module; `execute-plan` calls it to decide whether to advance, retry, or surface to the user.

### Resulting shape of execute-plan

`execute-plan` becomes: pre-flight (skill) → load plan → for each wave { wave-dispatch (skill) → verifier (skill) → regression-check (module) → retry-decision (module) → commit } → refine-code → final gate → finalize.

Each arrow is either a skill invocation or a module call. `execute-plan` itself holds only orchestration and policy it can't delegate.

## Non-goals
- Not changing the workflow's semantics. Same gates, same invariants, same artifacts. This is purely a structural refactor.
- Not redesigning the regression model, retry policy, or verifier protocol — just relocating them.
- Not touching `define-spec`, `generate-plan`, or `refine-code` as part of this work (other than updating cross-references).

## Constraints
- The three-set regression invariant (no double-counting, no baseline mutation, no regressions shipping on final wave) must survive the split exactly.
- Existing plans in `.pi/plans/` and `.pi/plans/done/` must continue to execute under the refactored skill without reformatting.
- Subagent dispatch contracts (coder / verifier / refiner prompt shapes) should not change unless an extraction forces it; if they do, document the change.
- New skills live in this repo under `agent/skills/` and must be self-contained — no dependency on external superpowers skills for their core procedure.

## Acceptance criteria
- `execute-plan/SKILL.md` is substantially smaller (target: well under half its current size) and reads as orchestration, not policy.
- Each new skill (verifier, wave-dispatcher, worktree-preflight) lives in its own directory under `agent/skills/` with its own SKILL.md, inputs/outputs documented, and can be invoked directly by other skills.
- Each extracted module lives alongside the skills it supports with a clear import surface and documented invariants.
- A fixture plan (small, synthetic) exercises the refactored workflow end-to-end: dispatch → verify → regression reconciliation → retry → commit → finalize. Passes before and after the split with identical artifacts modulo paths.
- Cross-references from `define-spec` and `generate-plan` to `execute-plan` still resolve, and `refine-code` is still invoked at the right point.
- A short migration note documents where each old `execute-plan` section now lives, so anyone reading the old file can find the new home.

## Open questions
- Should the new skills be siblings under `agent/skills/` or nested inside `execute-plan/`? Sibling is more reusable and lets other orchestrators invoke them; nested is clearer ownership. Sibling is probably right given the goal is reusability, but worth confirming.
- Where do the code modules live? Candidates: alongside `execute-plan/` as supporting files, or a shared `agent/skills/_lib/` (or similar) for modules reused across skills. Avoid scattering.
- Worktree-preflight overlaps conceptually with existing worktree practices elsewhere in the ecosystem — does the new skill need to diverge in behavior, or is it re-homing the same logic under a repo-local name for independence?
- Does the wave dispatcher need to own response-classification taxonomy (DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED), or should that live with the retry module since that's what branches on it?
