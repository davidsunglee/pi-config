## Goal

Replace the 300-line prose SKILL.md that drives execute-plan with a pi extension backed by a generic TypeScript core library. The extension handles all deterministic orchestration (plan parsing, wave computation, model resolution, subagent dispatch, state tracking, git operations, test running, TUI). The agent is only consulted for judgment calls.

## Problem statement

Today, `execute-plan` is effectively a large state machine encoded in prose. That works when the orchestrating model follows instructions perfectly, but the system remains fragile in predictable ways:

- skipped procedural steps
- lost or inconsistently reconstructed state between waves
- incorrect or hallucinated model resolution
- brittle retry handling
- weak resumability after interruption or failure
- poor observability into what happened and why
- UX limitations from keeping bookkeeping in prompts instead of code

The goal is not to remove agentic execution. The goal is to move orchestration mechanics into code and leave judgment-heavy decisions with agents.

## Motivation

1. **Model-independence.** Currently, only highly capable models follow the 13-step SKILL.md reliably. Moving deterministic steps to code means less capable (cheaper) models can orchestrate.
2. **Separation of duties.** Workflow orchestration is mechanical; judgment (handling BLOCKED, interpreting spec review results, deciding retries) is where an LLM adds value. Code should do the bookkeeping.
3. **Token efficiency.** The agent no longer spends tokens re-reading and following procedural instructions across waves.
4. **Consistent UX.** TUI interactions (settings confirmation, wave progress, failure handling) render identically regardless of which model is orchestrating.

## Core design principle

**Code orchestrates; agents judge.**

### Move into code

The extension should own the parts that are procedural, stateful, mechanical, and easy to verify:

- plan loading and structural validation
- extracting optional test command
- dependency parsing and wave computation
- model tier resolution from settings
- prompt/template loading and placeholder filling
- worker dispatch bookkeeping
- retry counting and retry state
- workspace/worktree bookkeeping
- per-wave commit mechanics
- test execution and baseline comparison
- review dispatch plumbing
- persistence of run state
- resume behavior
- cancellation handling
- progress reporting / TUI / status surfaces

### Keep agent-driven

The extension should continue delegating judgment-heavy work to agents/subagents:

- implementation of plan tasks via `plan-executor`
- interpretation of `DONE_WITH_CONCERNS`
- interpretation of `BLOCKED` / `NEEDS_CONTEXT`
- content-level verification against task acceptance criteria
- spec-compliance review reasoning
- final code review reasoning
- deciding whether a concern is acceptable or needs follow-up

The extension may encode routing heuristics, but should not try to replace LLM judgment with brittle deterministic rules where the task is inherently interpretive.

## Architecture

Three layers, each with a clear boundary:

### 1. Core library (`lib/execute-plan/`)

Pure TypeScript with no pi dependencies. Testable in isolation.

Responsibilities:
- **Plan parsing and validation** — read plan markdown, extract header/tasks/dependencies/risks/test-command, validate all 5 required sections present
- **Wave computation** — build dependency graph, assign tasks to waves, enforce ≤7 tasks per wave
- **Model tier resolution** — read `modelTiers` from settings.json, map task recommendations (cheap/standard/capable) to concrete model strings, resolve crossProvider tiers
- **Template filling** — read implementer-prompt.md, spec-reviewer.md, plan-reviewer.md, code-reviewer.md; fill placeholders deterministically
- **State management** — read/write `.pi/plan-runs/<plan>.state.json` (current wave, completed waves with commit SHAs, execution status, lock, settings, workspace info, baseline test results). This folds in TODO-307824c4.
- **Git operations** — detect branch/worktree status, stage+commit with structured messages after every wave, reset on retry, verify commit SHAs on resume
- **Test operations** — run test command, capture baseline, compare post-wave results against baseline to detect regressions (new failures vs pre-existing)
- **Plan lifecycle** — move plan to `done/`, close linked todo

Core library exports pure functions and a `PlanExecutionEngine` (or similar) that takes injected I/O interfaces:
```typescript
interface ExecutionIO {
  readFile(path: string): Promise<string>;
  writeFile(path: string, content: string): Promise<void>;
  exec(command: string, cwd: string): Promise<{ stdout: string; stderr: string; exitCode: number }>;
  dispatchSubagent(config: SubagentConfig): Promise<SubagentResult>;
  // ... other I/O
}
```

This keeps the core testable with mock I/O and makes it drivable from contexts other than a pi extension.

### 2. Pi extension (`extensions/execute-plan/index.ts`)

Thin wrapper. Registers a `/execute-plan` command and tools. Implements `ExecutionIO` using pi APIs.

Responsibilities:
- **Command registration** — `/execute-plan [path]` to start execution
- **Tool registration** — `execute_plan_judgment` (or similar) tool the agent calls when the extension hands off a judgment call
- **TUI** — settings confirmation dialog, wave progress display (widget or status), failure/retry interaction
- **Subagent dispatch** — spawn `pi --mode json -p --no-session` processes (same pattern as the existing subagent extension), with model override, prompt files, streaming progress updates
- **User interaction** — confirm dialogs (main branch warning, worktree choice), select dialogs (retry/skip/stop on failure), input dialogs (custom test command)
- **Agent handoff** — when a judgment call is needed (BLOCKED task, DONE_WITH_CONCERNS interpretation, retry strategy), the extension injects a message into the session with the relevant context and asks the agent to decide. The agent responds via the judgment tool.

### 3. Skill (thin, optional) (`skills/execute-plan/SKILL.md`)

Drastically reduced from current 300 lines. Tells the agent:
- "When the user wants to execute a plan, use `/execute-plan` or the `execute_plan` tool"
- "When the extension asks for your judgment (via injected message), evaluate the situation and respond via the `execute_plan_judgment` tool"
- Documents the types of judgment calls: BLOCKED handling, DONE_WITH_CONCERNS evaluation, spec review interpretation, retry vs escalate decisions

## Preconditions

The extension checks these before starting execution:

1. **Git repository required.** Checkpoint commits are mandatory (every completed wave is committed). If the working directory is not inside a git repo, the extension refuses to execute with a clear error: "execute-plan requires a git repository (commits are made after each wave)."
2. **Clean working tree recommended.** Warn if there are uncommitted changes — they'll be mixed into the first wave commit.

## Interaction flow

```
User: "execute the plan"
  │
  ├─► Agent recognizes intent, calls /execute-plan or execute_plan tool
  │
  ├─► Extension takes over:
  │     0. Precondition checks (git repo exists, warn if dirty)
  │     1. Parse plan (core lib)
  │     2. Check for existing state file → offer resume or restart
  │     3. Show settings TUI, get user confirmation
  │     4. Compute waves (core lib)
  │     5. Resolve models (core lib)
  │     6. Optionally set up worktree (core lib + user interaction)
  │        - If on main: prompt for worktree (default) or confirm committing to main
  │     7. Capture test baseline (core lib)
  │     8. Record pre-execution SHA, acquire lock, set status "running"
  │     9. For each wave:
  │        a. Fill worker prompts (core lib)
  │        b. Dispatch subagents (extension, parallel)
  │        c. Collect results
  │        d. If all DONE → verify, spec-check
  │        e. If BLOCKED/NEEDS_CONTEXT/DONE_WITH_CONCERNS → hand to agent
  │        f. Commit wave (always — mandatory checkpoint)
  │        g. Run integration tests, compare against baseline
  │        h. If test regression → present retry/skip/stop to user
  │           - On retry: git reset to undo wave commit, re-dispatch
  │        i. Update state sidecar (core lib), set wave status "done" with commitSha
  │        j. Update TUI progress
  │    10. Dispatch final code review (extension, cross-provider)
  │    11. Present review results
  │    12. Move plan to done, close linked todo, delete state file (core lib)
  │    13. Release lock, set status "completed"
  │    14. Offer branch completion (extension → skill handoff)
  │
  └─► Extension returns summary to agent context
```

## Concurrency policy

V1 assumes **one active `execute-plan` run per repository**. The lock mechanism in the state file enforces this. Starting a second run while one is active is rejected with a clear error, in exchange for simpler state handling and fewer concurrency hazards.

## Cancellation behavior

Cancellation supports two granularities:

1. **Stop after the current wave finishes** — all in-flight tasks in the wave complete, results are collected and committed if successful, then execution halts.
2. **Stop after the current task finishes** — remaining tasks in the wave are not dispatched, partial wave results are not committed, execution halts.

The extension makes the active cancellation mode visible in the TUI and persists cancellation state (`"status": "stopped"`, `"stoppedAt"`, `"stopGranularity"`) in the state sidecar. On resume, the extension reports how the previous run stopped and what remains.

## State management (folds in TODO-307824c4)

### State file

Location: `.pi/plan-runs/<plan-name>.state.json`

State files live in `.pi/plan-runs/`, separate from `.pi/plans/` which holds only plan files and their `done/`/`reviews/` subdirectories. This keeps the plans directory clean and avoids mixing authoring artifacts with execution state.

```json
{
  "plan": "2026-04-06-my-feature.md",
  "status": "running" | "stopped" | "completed",
  "lock": { "pid": 12345, "session": "abc.json", "acquiredAt": "2026-04-06T..." },
  "startedAt": "2026-04-06T...",
  "stoppedAt": null,
  "stopGranularity": null,
  "settings": {
    "execution": "parallel",
    "tdd": true,
    "finalReview": true,
    "specCheck": true,
    "integrationTest": true,
    "testCommand": "npm test"
  },
  "workspace": {
    "type": "worktree" | "current",
    "path": "/absolute/path/to/workspace",
    "branch": "plan/my-feature"
  },
  "preExecutionSha": "abc123",
  "baselineTest": { "exitCode": 0, "output": "...", "failingTests": [] },
  "waves": [
    { "wave": 1, "tasks": [1, 2], "status": "done", "commitSha": "def456" },
    { "wave": 2, "tasks": [3, 4], "status": "done", "commitSha": "ghi789" },
    { "wave": 3, "tasks": [5], "status": "in-progress" }
  ]
}
```

Key design decisions:
- **Wave-level granularity for v1.** Each wave is either fully done (committed) or not. No task-level checkpointing within a wave — if a wave is interrupted, it re-runs from scratch.
- **`commitSha` always populated for done waves.** Checkpoint commits are mandatory, so every completed wave has a corresponding git commit.
- **`status` field for crash detection.** Set to `"running"` when a wave starts, `"stopped"` on clean user-initiated stop, `"completed"` at the end. If state says `"running"` on resume, the previous execution didn't exit cleanly.
- **`lock` for concurrent execution guard.** PID + session + timestamp, same pattern as todo locks. Prevents two terminals from executing the same plan. Stale locks (dead PID) are detected and can be stolen with user confirmation.
- **`workspace` block** records where execution is happening. On resume, the extension verifies the workspace still exists and is on the expected branch.
- **Baseline test stored and trusted on resume.** Only incorrect if the user rewrote git history, which we don't guard against in v1.

### Resume flow

1. Find state file for the plan in `.pi/plan-runs/`.
2. **Check lock.** Dead PID → stale, warn "previous execution may not have exited cleanly." Live PID → block, "plan is already being executed in another session."
3. **Verify workspace.**
   - If `workspace.type === "worktree"`: verify path exists. If gone → "Can't resume — worktree at `<path>` no longer exists." Offer restart (new worktree).
   - If `workspace.type === "current"`: verify we're on the expected branch. If different branch → warn, offer restart.
4. **Cross-check git.** For each "done" wave, verify `commitSha` exists in `git log`. If any SHA is missing → warn "git history doesn't match state — someone may have rewritten history." Offer restart.
5. **Show resume confirmation.** Display stored settings and progress: "Resuming plan X — waves 1-2 done, starting wave 3. Settings: [summary]. Continue / Restart / Cancel?"
6. **Acquire lock, set status "running", resume** from first non-done wave.

### State file lifecycle

- **Created** in `.pi/plan-runs/` when execution starts (after settings confirmation).
- **Updated** after each wave completes.
- **Deleted** on successful completion (when plan moves to `done/`).
- **Overwritten** if user chooses "restart" on resume prompt.
- **Left behind** on crash or stop — that's the whole point, it enables resume.

## Mandatory checkpoint commits

Every completed wave is committed. This is not configurable.

**Commit message format:**
```
feat(plan): wave <N> - <plan goal summary>

- Task <X>: <task title>
- Task <Y>: <task title>
```

**Consequences:**
- Git repo is a hard precondition for execute-plan.
- Resume can always cross-verify state against git history.
- Retry always has a commit to undo (`git reset --soft HEAD~1` + `git reset HEAD`).
- Final code review range (`BASE_SHA..HEAD_SHA`) always spans real commits.
- If on main/master/develop (no worktree, no feature branch): warn user once before first commit — "You're on `<branch>`. Every wave will commit directly. Continue?" If declined, suggest setting up a worktree or switching to a feature branch — do not proceed without commits.

## Boundary: what the agent decides vs. what code decides

### Code decides (deterministic)
- Plan parsing, validation, and wave computation
- Model tier resolution from settings.json
- Template filling (all placeholders)
- Subagent dispatch (model, prompt, parallelism)
- Git operations (commit after every wave, staging, reset on retry)
- Test running and baseline comparison
- State file read/write, lock management
- Spec review dispatch (mechanical), code review dispatch (mechanical)
- TUI rendering and progress tracking
- Plan lifecycle (move to done, close todo, delete state file)
- Resume: workspace verification, git cross-check, state restoration
- Cancellation: signal handling, state persistence, granularity enforcement

### Agent decides (judgment)
- Interpreting BLOCKED status — provide more context? re-dispatch with better model? break task down?
- Interpreting DONE_WITH_CONCERNS — are the concerns serious? address now or note for later?
- Interpreting spec review results — are the failures real? re-dispatch implementer with findings?
- Interpreting final code review results — fix critical issues? proceed with minor issues?
- Retry strategy when automatic retries exhaust — try different approach? skip? stop?
- Worktree branch name suggestion (agent suggests, user confirms via TUI)
- Whether a task that's not explicitly `cheap`/`standard`/`capable` should be assigned what tier

### User decides (via TUI)
- Execution settings (parallel/sequential, TDD, review, spec check, integration test)
- Resume vs restart (when state file found)
- Main branch commit warning acknowledgment
- Worktree setup confirmation
- Retry/skip/stop on wave failure or test regression
- Cancellation granularity (stop after wave vs stop after task)
- Final branch disposition (merge, PR, keep, discard)

## Shared primitives

Even though this effort only implements `execute-plan`, the core library should factor out reusable primitives that other workflows can later share:

- plan file parser / validator
- dependency graph + wave builder
- model tier resolver
- prompt-template loader / filler
- run-state persistence layer
- review-dispatch helper(s)
- git / worktree helper(s)
- test baseline + comparison helper(s)
- execution reporting / progress primitives

These primitives should make later migration of adjacent workflows (especially `generate-plan`) easier without requiring an all-at-once rewrite.

## TUI interactions

The extension should provide consistent, well-designed TUI for:

1. **Settings confirmation** — bordered panel showing plan summary + settings grid, with start/customize/quit options. Note: checkpoint commit is not shown as a setting — it's always on. If no git repo, execution is blocked before reaching this screen.
2. **Resume prompt** — when state file found: show progress summary, offer continue/restart/cancel
3. **Main branch warning** — confirm dialog when about to commit directly to main
4. **Wave progress** — widget (above or below editor) showing current wave, completed waves, task statuses, with real-time updates as subagents stream
5. **Failure handling** — select dialog for retry/skip/stop with context about what failed
6. **Cancellation feedback** — show active cancellation mode and what's still in flight
7. **Review summary** — formatted display of code review findings

## Subagent dispatch

Follow the existing subagent extension pattern:
- Spawn `pi --mode json -p --no-session` with `--model`, `--tools`, and `--append-system-prompt`
- Parse JSON events from stdout for streaming progress and final results
- Support parallel dispatch with concurrency limit
- Track per-task usage stats (tokens, cost, turns)

The extension should reuse or factor out common subagent dispatch code from the existing subagent extension where possible, rather than duplicating the spawn/parse logic.

## Migration strategy

1. Build core library with tests against mock I/O
2. Build extension, initially alongside the existing SKILL.md
3. Replace SKILL.md with a thin stub that points to the extension
4. Validate on a real plan execution
5. Remove the old SKILL.md content once confident

The existing prompt templates (implementer-prompt.md, spec-reviewer.md, code-reviewer.md, plan-reviewer.md) remain as-is — the core library reads and fills them. The plan-generator agent/skill is untouched.

## Constraints

- **No standalone CLI.** The extension is the entry point; it's driven from pi.
- **Core library has no pi imports.** It depends only on Node.js built-ins and its injected I/O interface.
- **Prompt templates stay as markdown files.** The extension reads them at runtime; they are not compiled into code.
- **Existing plan format unchanged.** Plans generated by the current plan-generator work with the new extension.
- **Backward compatible settings.json.** The `modelTiers` structure in settings.json is unchanged.
- **Git repo required.** Checkpoint commits are mandatory; execute-plan cannot run outside a git repository.

## Non-goals

- Changing the plan-generator or its output format
- Changing the plan-executor agent definition
- Building a standalone CLI tool
- Changing the subagent dispatch mechanism (still spawns pi processes)
- Redesigning the prompt templates
- Task-level checkpointing within a wave (v2 consideration)
- Multi-user/team concurrency (v2 consideration)
