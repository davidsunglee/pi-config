Build a pi extension that customizes the colors of individual fields in the pi footer, tied into the theme system.

## Default footer layout (reference)
The built-in footer renders 2-3 lines:
- **Line 1:** `~/path (branch) ‚Ä¢ session-name`
- **Line 2 left:** `‚Üëinput ‚Üìoutput Rcache Wcache $cost context%/contextWindow (auto)`
- **Line 2 right:** `(provider) model-name ‚Ä¢ thinking-level`
- **Line 3 (optional):** extension statuses

Context usage is colored yellow >70% and red >90%. The extension must preserve this behavior.

## Configurable color fields
All fields individually configurable via theme token names (e.g. `"accent"`, `"success"`, `"muted"`), auto-adapting on theme switch.

| Config key | Applies to |
|---|---|
| `model` | Model name (and provider prefix, thinking level) |
| `tokens` | Input **and** output token counts (shared) |
| `cost` | Cost value (and subscription indicator) |
| `cache` | Prompt cache read **and** write counts (shared) |
| `contextUsage` | Context usage percentage (when ‚â§70%; overridden by warning/error at higher levels) |
| `contextWindow` | Context window capacity value |
| `branch` | Git branch name |
| `pwd` | Working directory path |
| `statuses` | Extension status texts |
| `symbols` | All glyphs: `‚Üë`, `‚Üì`, `R`, `W`, `$`, `/`, `%`, parens, etc. |

## Context usage color escalation
Must preserve the default footer's behavior:
- ‚â§70%: use `contextUsage` field color (normal)
- \>70%: override to `"warning"` (yellow)
- \>90%: override to `"error"` (red)

The `/` separator between context usage and context window uses the `symbols` color.

## Color source
- Hardcoded in the extension using theme token references
- References theme tokens via `theme.fg(tokenName, text)` so colors adapt automatically when the theme changes

## Layout
- Keep the default 2-line layout: pwd/branch on line 1, stats left + model right on line 2, optional statuses on line 3

## Location
- Global: `~/.pi/agent/extensions/custom-footer.ts`

## API surface available to extensions
From `ctx` (ExtensionContext / ExtensionCommandContext):
- `ctx.sessionManager.getEntries()` ‚Äî all session entries (use getEntries not getBranch to include pre-compaction)
- `ctx.model` ‚Äî current model (`.id`, `.reasoning`, `.contextWindow`, `.provider`)
- `ctx.getContextUsage()` ‚Äî returns `{ tokens, contextWindow, percent }` or undefined
- `ctx.cwd` ‚Äî current working directory
- `ctx.modelRegistry.isUsingOAuth(model)` ‚Äî subscription check

From `footerData` (ReadonlyFooterDataProvider):
- `footerData.getGitBranch()` ‚Äî git branch or null
- `footerData.getExtensionStatuses()` ‚Äî ReadonlyMap of status texts
- `footerData.getAvailableProviderCount()` ‚Äî number of providers
- `footerData.onBranchChange(cb)` ‚Äî subscribe to branch changes

From ExtensionCommandContext (in session_start handler):
- `ctx.getThinkingLevel()` ‚Äî current thinking level
- `ctx.getSessionName()` ‚Äî session display name
