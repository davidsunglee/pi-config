## Recommendation

Move the **deterministic orchestration** around `generate-plan` into code, while keeping **plan synthesis / decomposition** in the LLM.

At the same time, extract the **plan contract** into a shared library so both `generate-plan` and `execute-plan` depend on the same parsing / validation rules.

This should follow the same design principle used in `execute-plan`: **use code for deterministic contract enforcement and orchestration; use LLMs where judgment is required.**

## Why this is worth doing

### 1. `execute-plan` now treats the plan format as an executable contract
The repo already has code in `agent/lib/execute-plan/plan-parser.ts` that parses and validates plans with concrete structural expectations:
- required sections
- task numbering and task block format
- dependency syntax
- optional test command extraction
- source todo extraction

This means the plan format is no longer just guidance for humans; it is effectively an API consumed by automation.

### 2. `generate-plan` still leaves too much deterministic behavior in skill prose
The current `agent/skills/generate-plan/SKILL.md` describes deterministic logic in natural language:
- input source resolution
- prompt assembly
- source todo propagation
- review model selection
- review output path derivation
- review parsing
- review-note appending
- re-generation loop rules

Those behaviors are better represented in code than in a skill file, especially now that downstream execution is stricter.

### 3. Shared contract reduces drift
Today, the code that executes a plan and the prompts/skills that generate a plan can drift apart. The system will be more robust if:
- `generate-plan` writes plans that are checked against the exact same contract `execute-plan` consumes
- future plan format changes happen in one place
- review / repair logic can use validator output directly

### 4. Token savings are real but secondary
Some prompt text can be removed once deterministic behavior moves to code, but the primary motivation is **correctness and consistency**, not token optimization.

## Proposed architecture

### A. Introduce a shared plan-contract library
Create a shared library that represents the canonical plan format and validation rules.

Possible location:
- `agent/lib/plan-contract/`
- or `agent/lib/plan/`

Core responsibilities:
- parse plan markdown into structured data
- validate required sections and references
- expose typed plan data structures
- expose helper utilities for plan metadata / paths where appropriate

Likely modules:
- `types.ts` — shared plan types
- `parser.ts` — markdown → structured plan object
- `validator.ts` — structural validation rules
- `paths.ts` — helper functions for plan/review filenames if useful
- `notes.ts` — append / merge review notes in canonical format if useful

### B. Make both generate and execute flows depend on the shared contract
`execute-plan` should consume the shared library instead of owning its own parser/validator implementation.

`generate-plan` should use the same shared library to:
- validate newly generated plans before they are treated as executable
- run targeted edit-based repair against the existing plan when validation or review finds issues
- keep the emitted markdown aligned with the parser’s expectations

## Scope of code to move from skill into implementation

### Move into code
The following parts are deterministic and should be implemented in code:

1. **Input normalization**
   - todo ID → full todo contents
   - file path → file contents
   - freeform description → raw text

2. **Prompt construction**
   - canonical planner prompt
   - inclusion of cwd/repo context
   - source todo propagation
   - output filename guidance
   - fixed formatting constraints required by the plan contract

3. **Post-generation validation gate**
   - parse generated plan
   - run structural validation
   - fail / retry / repair if invalid

4. **Review orchestration**
   - choose model tier
   - derive review output path
   - dispatch review agent
   - parse review result
   - append informational notes to plan when appropriate

5. **Canonical plan/review file handling**
   - plan path derivation
   - review path derivation
   - source metadata handling
   - standardized `## Review Notes` behavior

### Keep in the LLM
The following parts should stay LLM-driven because they require judgment:
- architecture synthesis
- task decomposition
- dependency reasoning
- task sizing
- acceptance criteria drafting
- risk identification
- format-specific footgun identification
- model recommendation per task

## Recommended implementation shape

### Preferred approach
Implement a **coded `generate-plan` wrapper** (tool/extension/command) around the existing `plan-generator` agent rather than trying to replace the planner with code.

Suggested high-level flow:
1. Resolve input into canonical source text
2. Build canonical planner prompt in code
3. Dispatch `plan-generator`
4. Validate generated plan with shared plan-contract lib
5. Run plan review
6. If validation or review finds issues, pass the existing plan plus specific findings back to the model and instruct it to edit in place rather than regenerate from scratch
7. Repeat targeted edit/review cycles up to 10 times, surfacing remaining findings if the loop does not converge
8. If fundamental structural issues remain after 2 edit cycles, allow partial regeneration of only the affected section(s) as an escape hatch
9. Append non-blocking review notes and return final plan path + review status

### Minimal high-value version
If the full wrapper is too much for one pass, the first implementation should at least add:
- a shared plan-contract lib
- a coded post-generation validation gate before the plan is handed to `execute-plan`

That directly addresses the main risk: malformed or drifting plans reaching stricter execution logic.

## Migration plan

### Phase 1: Extract the plan contract
- Move plan types / parsing / validation out of `agent/lib/execute-plan/`
- Replace execute-plan-local imports with shared library imports
- Preserve existing behavior and tests
- Add tests proving the shared library accepts current valid plans and rejects malformed ones

### Phase 2: Add generate-plan runtime wrapper
- introduce a command/tool-backed implementation for `generate-plan`
- move deterministic orchestration out of `agent/skills/generate-plan/SKILL.md`
- reduce the skill to a thin user-facing entry point that delegates to code

### Phase 3: Add validation + targeted edit loop
- after generation, validate via shared contract
- on validation failure or reviewer findings, feed explicit findings back into an edit prompt against the existing plan
- prefer targeted in-place editing over full regeneration so correct sections stay stable and the loop converges
- allow up to 10 edit/review cycles; typical convergence should be 2-3 cycles
- if fundamental structural issues remain after 2 edit cycles, allow partial regeneration of only the affected section(s)
- if max cycles are exhausted, surface remaining findings to the user rather than silently accepting the result

### Phase 4: Move review orchestration into code
- derive output paths in code
- select model tiers in code
- parse review result in code
- append review notes in a deterministic, canonical format

## Deliverables

### Shared contract deliverables
- shared plan types
- shared markdown parser
- shared validator
- tests covering valid and invalid plans
- documentation of the canonical plan contract

### Generate-plan deliverables
- coded wrapper for generate-plan orchestration
- canonical prompt builder
- validation gate
- targeted edit-based repair flow over the existing plan (up to 10 cycles, usually 2-3)
- review orchestration in code
- simplified `generate-plan` skill that delegates to implementation

### Execute-plan deliverables
- refactor to consume shared plan-contract lib
- no plan-format logic should remain duplicated between generate and execute paths

## Acceptance criteria

1. There is one canonical implementation of the plan contract used by both generation and execution flows.
2. `execute-plan` no longer owns unique parser/validator logic that can drift from generation.
3. `generate-plan` validates generated plans before presenting them as executable.
4. A malformed plan produced by the generator is caught before execution begins.
5. If validation fails, the system can surface or use concrete validator errors rather than vague prompt-only guidance.
6. Deterministic review orchestration no longer lives only in skill prose.
7. The remaining `generate-plan` skill text is thin and user-facing, not the source of truth for control flow.
8. Existing valid plans continue to parse and execute successfully after the refactor.
9. When repair is needed, the system edits the existing plan in place by default rather than regenerating the whole document.
10. The wrapper never silently accepts a non-converged review/edit loop; remaining findings are surfaced to the user when limits are reached.

## Non-goals

- Do **not** replace LLM planning/decomposition with algorithmic task generation.
- Do **not** over-constrain the planner with excessive coded heuristics for architecture/task design.
- Do **not** undertake a large rewrite if a wrapper + shared contract provides the needed reliability.

## Risks / cautions

- Over-scoping this into a “full coded planner” would likely have poor ROI.
- If markdown remains the executable plan format, parser/renderer constraints still matter; validation remains essential.
- The contract should be strict enough to protect execution, but not so strict that harmless wording changes break valid plans.

## Design decisions (2026-04-12)

### Review → edit, not full regenerate
When the plan reviewer finds issues, the wrapper should pass the **existing plan plus specific findings** back to the model and instruct it to **edit in place** instead of regenerating the whole plan.

**Rationale:** full regeneration is expensive and tends not to converge. Each rewrite is stochastic, so fixing one issue can easily reintroduce another. Targeted editing preserves the parts of the plan that were already correct and should usually converge in 2-3 cycles.

### Repair loop limits
- default behavior: targeted edits against the current plan
- max cycles: 10
- if the max is exhausted: surface remaining findings to the user rather than silently accepting the plan
- escape hatch: if the reviewer still reports **fundamental structural issues** after 2 edit cycles, allow partial regeneration of only the affected section(s), not the entire plan

## Open questions

1. Should the shared contract library stay markdown-native, or should generation move toward a structured intermediate representation later (JSON/YAML → canonical markdown rendering)?
2. Should review-note appending be part of the shared plan-contract lib, or a generate-plan-specific helper?
3. Should filename/path derivation live in the shared contract, or in generate/execute-specific helpers?
4. Is the first step a full generate-plan extension/tool, or just contract extraction + validation gate?

## Suggested first slice

Start with the smallest slice that meaningfully reduces risk:
1. extract shared plan parsing / validation into a common lib
2. switch `execute-plan` to consume that shared lib
3. add a coded post-generation validation step for `generate-plan`
4. only then move the rest of generate-plan orchestration into code

That sequence yields most of the correctness benefit early while keeping scope controlled.
