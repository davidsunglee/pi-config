## 1. Spec refinement and readiness

### Refinement has no structured completeness gate

**Problem:** The refine workflow is a prompt injection: "Ask me for the missing details..." There's no schema for what constitutes "refined enough" and no "ready for planning" gate.

**Suggestions:**
1. Define a lightweight spec-ready checklist the refine agent works toward — e.g. problem statement, success criteria, scope boundaries, constraints
2. Have the agent explicitly declare readiness for plan generation instead of implicitly ending the conversation
3. Incorporate lessons from Superpowers's `brainstorming` skill as a reference for how to guide ambiguity reduction and collaborative problem framing:
   - https://github.com/obra/superpowers/blob/main/skills/brainstorming
4. Document which parts of that brainstorming flow should carry over into refinement (for example: structured exploration of options, surfacing constraints, and converging toward a clearer problem statement) versus which parts should remain separate from generate-plan

## 2. Execution handoff and follow-through

### Worktree UX gap

**Problem:** Auto-detection of workspace mode is silent. User doesn't get clear feedback about which path was taken or why. Reusing a feature branch from a previous abandoned run can silently mix work from two plans.

**Suggestions:**
1. Always log the decision explicitly: "Using existing worktree at .worktrees/plan/foo (detected feature branch `plan/foo`)" vs "Creating new worktree at .worktrees/plan/bar"
2. Warn on branch reuse: if the detected feature branch has uncommitted changes or commits not associated with the current plan, warn before proceeding
3. Consider making worktree creation the unconditional default unless the user explicitly opts out

### No plan resumability

**Problem:** Step 11 says "leave the plan file so it can be resumed" but there's no resume mechanism. The state sidecar `.pi/plan-runs/<plan>.state.json` tracks progress but no "resume from wave N" workflow exists.

**Suggestion:** When execute-plan receives a plan with an existing state file, offer to resume from the last successful wave rather than starting over.

### BLOCKED escalation should be immediate

**Problem:** When using wave pacing option (b) "never pause," a BLOCKED task's context is lost by the time the user sees the report at the end. Subsequent waves that depend on the blocked task are wasted.

**Suggestion:** For BLOCKED status specifically, always surface immediately regardless of wave pacing setting.

## 3. Model matrix resolution order

### models.json should support project-local override

**Problem:** Both generate-plan and execute-plan read model tiers from `~/.pi/agent/models.json`. This is a user-level config with no way to override per project. Different projects may need different model assignments (e.g., a project with a large codebase might need `capable` where a smaller one uses `standard`).

**Suggestion:** Look for `.pi/agent/models.json` in the project directory first, fall back to `~/.pi/agent/models.json`. This matches the existing pi convention where project-level config (`.pi/`) takes precedence over user-level config (`~/.pi/`).
