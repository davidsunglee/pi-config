## 1. Execution handoff and follow-through

### Worktree reuse UX and safety

**Problem:** `execute-plan` now surfaces the workspace mode in the settings summary (`current workspace` vs `new worktree`), so the old "silent auto-detection" concern is mostly addressed. The remaining gap is reuse safety and explicitness: the user still does not get a concrete log of which workspace path is being used, and reusing an existing feature branch/worktree can still silently mix work from an abandoned or unrelated run into the current plan.

**Suggestions:**
1. Log the decision explicitly with path and reason, e.g. `Using existing worktree at .worktrees/foo (detected feature branch \`foo\`)` vs `Creating new worktree at .worktrees/bar`
2. Warn on branch/worktree reuse: if the detected workspace has uncommitted changes, or commits not obviously associated with the current plan, warn before proceeding
3. Consider making new worktree creation the unconditional default unless the user explicitly opts into reusing the current workspace

### No plan resumability

**Problem:** `execute-plan` still says the plan can be resumed later, but the skill does not define a resume workflow. The state sidecar `.pi/plan-runs/<plan>.state.json` can track progress, but there is no documented `resume from last successful wave` path in the skill.

**Suggestion:** When `execute-plan` receives a plan with an existing state file, offer to resume from the last successful wave rather than starting over.

### BLOCKED escalation should be immediate

**Problem:** When using wave pacing option (b) "never pause," a `BLOCKED` task's context is lost by the time the user sees the report at the end. Subsequent waves that depend on the blocked task are wasted.

**Suggestion:** For `BLOCKED` status specifically, always surface immediately regardless of wave pacing setting.

## 2. Model matrix resolution order

### `model-tiers.json` should support project-local override

**Problem:** Both `generate-plan` and `execute-plan` read model tiers from `~/.pi/agent/model-tiers.json`. This is a user-level config with no way to override per project. Different projects may need different model assignments (for example, a large codebase might need `capable` where a smaller one uses `standard`).

**Suggestion:** Look for `.pi/agent/model-tiers.json` in the project directory first, then fall back to `~/.pi/agent/model-tiers.json`. This matches the existing pi convention where project-level config (`.pi/`) takes precedence over user-level config (`~/.pi/`).
