## Findings

### High

#### 1. Generate-plan edit loop does not specify the output path for surgical plan edits
- **Files:**
  - `agent/skills/generate-plan/SKILL.md:102-111`
  - `agent/skills/generate-plan/edit-plan-prompt.md:21-28`
  - `agent/agents/planner.md:31-33`
- **What:** Step 4.3 says the `planner` should write the edited plan back to the same path, but the filled edit prompt does not include that path, and the dispatch does not pass an `output:` path either.
- **Why it matters:** The `planner` contract says it writes to the output path specified in the task prompt. On the first review failure, the surgical edit flow is underspecified and cannot reliably overwrite the original plan.
- **Recommendation:** Include the original plan path explicitly in `edit-plan-prompt.md` and/or pass it via the dispatch contract so the overwrite target is unambiguous.

### Medium

#### 2. Refine-code expects a different review verdict format than the code-review template defines
- **Files:**
  - `agent/skills/requesting-code-review/review-code-prompt.md:93`
  - `agent/skills/requesting-code-review/review-code-prompt.md:146`
  - `agent/skills/refine-code/refine-code-prompt.md:59-61`
- **What:** The formal review template says `**Ready to merge?** [Yes/No/With fixes]`, while `refine-code` says to assess verdict based on the string `"Ready to merge: Yes"`. The example in the template uses colon syntax, but the declared format uses question-mark syntax.
- **Why it matters:** This is a parsing contract mismatch. The code refiner may fail to recognize a clean review or may parse verdicts inconsistently.
- **Recommendation:** Normalize the review template and the refine-code protocol to one exact verdict format and reference that exact format everywhere.

#### 3. `coder` agent output format conflicts with `execute-task-prompt.md`
- **Files:**
  - `agent/agents/coder.md:45-58`
  - `agent/skills/execute-plan/execute-task-prompt.md:79-99`
- **What:** `coder.md` defines an output format with `## Completed`, `## Files Changed`, and optional concerns. The execute-task prompt says to use an exact structure that additionally requires `## Tests` and `## Self-Review Findings`.
- **Why it matters:** The same worker receives conflicting reporting requirements. That reduces deterministic output and can break orchestration or downstream parsing assumptions.
- **Recommendation:** Make the agent definition and the worker prompt require the same exact report schema.

### Low

#### 4. `max_iterations_reached` option wording is inconsistent between `execute-plan` and `refine-code`
- **Files:**
  - `agent/skills/execute-plan/SKILL.md:403-406`
  - `agent/skills/refine-code/SKILL.md:73-78`
- **What:** `execute-plan` offers `Keep iterating / Proceed with issues / Stop execution`, while `refine-code` still offers `Continue iterating / Proceed / Stop`.
- **Why it matters:** The flow is still understandable, but it violates the suite-wide consistency this plan aimed to establish.
- **Recommendation:** Reword `refine-code/SKILL.md` to match the execute-plan wording exactly.
