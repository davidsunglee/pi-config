# Wave 5 Task 16 Code Review

## Strengths
- Wave dispatch is wired through `TaskQueue` rather than any bulk-dispatch API, and the queue forwards per-task completion/progress back into engine progress events (`agent/lib/execute-plan/engine.ts:780-800`, `agent/lib/execute-plan/task-queue.ts:38-100`). That matches the intended singular-dispatch architecture from the task spec (`/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:848-853,885-886`).
- The wave loop persists meaningful state on the happy path: waves are marked `in-progress` before execution, committed waves are written back via `updateWaveStatus(...)`, and final review emits a typed `code_review_completed` event carrying a parsed `CodeReviewSummary` (`agent/lib/execute-plan/engine.ts:500-552`, `agent/lib/execute-plan/engine.ts:680-689`, `agent/lib/execute-plan/engine.ts:1185-1198`).
- The focused verification baseline is clean: `cd agent && node --experimental-strip-types --test lib/execute-plan/engine.test.ts` passed 39/39, and `cd agent && npx tsc --noEmit --target ESNext --module NodeNext --moduleResolution NodeNext --strict --esModuleInterop --skipLibCheck --allowImportingTsExtensions lib/execute-plan/engine.ts lib/execute-plan/engine.test.ts lib/execute-plan/index.ts` completed without diagnostics.

## Issues
### Critical
- Stop/cancellation paths still fall through the normal success-completion lifecycle. `executeWaves()` only returns `void`, and every halt path just `return`s (`agent/lib/execute-plan/engine.ts:520-552`, `agent/lib/execute-plan/engine.ts:727-728`, `agent/lib/execute-plan/engine.ts:871-873`, `agent/lib/execute-plan/engine.ts:1033-1034`, `agent/lib/execute-plan/engine.ts:1203-1205`, `agent/lib/execute-plan/engine.ts:1237-1239`). Back in `execute()`, there is no stopped/completed distinction; the engine always continues into release-lock, move-plan-to-done, delete-state, and `execution_completed` emission (`agent/lib/execute-plan/engine.ts:389-419`). I reproduced this with a targeted harness that returned `{ action: "stop" }` from `requestJudgment`: progress still ended with `execution_completed`, the plan was moved into `.pi/plans/done/`, and the state file was deleted. That violates the required halt semantics for `stop`, retry exhaustion, and both cancellation modes (`/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:834-844,861-871,879-881,891-892`).

### Important
- Retry metadata is not fully persisted. `persistTaskRetryState()` always writes `lastContext: null` and `lastModel: null` (`agent/lib/execute-plan/engine.ts:1275-1302`), and the task-level `retry` / `provide_context` branches never pass the chosen `response.context` or `response.model` into state before redispatch (`agent/lib/execute-plan/engine.ts:904-980`). The same pattern is repeated for wave retries (`agent/lib/execute-plan/engine.ts:614-629`, `agent/lib/execute-plan/engine.ts:658-673`) and final-review retries (`agent/lib/execute-plan/engine.ts:1210-1221`). Task 16 explicitly requires persisting attempt count, last failure, optional model override, and appended context before each retry (`/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:858-859`), and the acceptance criteria require that "all retry metadata" be written through `updateState` (`/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:887`). The current tests only assert that retry entries exist, not that the required fields are correct (`agent/lib/execute-plan/engine.test.ts:1155-1205`, `agent/lib/execute-plan/engine.test.ts:1438-1515`, `agent/lib/execute-plan/engine.test.ts:1616-1681`).
- Final-review retries do not dispatch any fix-up work. On `response.action === "retry"`, `executeFinalReview()` persists `retryState.finalReview` and immediately loops back to another `code-reviewer` invocation (`agent/lib/execute-plan/engine.ts:1182-1224`). There is no implementer/fix-up dispatch carrying the review findings. The plan requires the engine to "dispatch fix-up work with the review findings in the prompt, then re-run final review" (`/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:879`). As written, the retry loop can only re-review the same workspace. The corresponding test also stops at persistence and never asserts a fix-up pass (`agent/lib/execute-plan/engine.test.ts:1616-1681`).

### Minor
- The new Task 16 tests do not fully lock the anchored acceptance criteria:
  - the stop-after-task test checks only that no `wave_completed` event is emitted, but never verifies that `taskQueue.abortAfterCurrent()` was invoked (`agent/lib/execute-plan/engine.test.ts:1787-1890` vs. `/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:843-844,891`);
  - the live-progress test exercises retry-time direct `io.dispatchSubagent(...)`, not the initial `TaskQueue.run(...)` forwarding path that the spec calls out (`agent/lib/execute-plan/engine.test.ts:1894-1944` vs. `/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:851,893`);
  - the regression test proves `requestTestRegressionAction()` is called, but it does not assert `resetWaveCommit()` and `retryState.waves` persistence on the regression-retry path (`agent/lib/execute-plan/engine.test.ts:1079-1153` vs. `/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:868-870,894`).

## Recommendations
- Make `executeWaves()` report an explicit outcome (`completed` vs. `stopped`) or throw a sentinel stop/cancel result so `execute()` only performs plan-finalization (`movePlanToDone`, todo close, state deletion, `execution_completed`) on true success.
- Thread retry metadata through the persistence helpers: record `lastContext` and `lastModel` for task, wave, and final-review retries, and add assertions on those serialized fields instead of only checking that retry objects exist.
- Implement the missing final-review remediation loop: dispatch a fix-up worker with the review findings in the prompt, then re-run code review against the updated workspace.
- Tighten the focused tests around the spec-anchored behaviors: spy on `taskQueue.abortAfterCurrent()`, cover progress forwarding through the initial TaskQueue path, and verify the regression-retry path persists `retryState.waves` and resets the wave commit before retrying.

## Assessment
I would not sign off Task 16 yet. The happy-path orchestration is largely there, the singular-dispatch/TaskQueue structure is sound, and the focused tests/typecheck are green, but the engine still treats multiple stop conditions as successful completion, drops required retry metadata, and omits the spec-mandated fix-up loop for final review retries. Those are task-level correctness gaps, not polish items.
