### Strengths
- The implementation largely matches the intended architecture: a pure TS core in `agent/lib/execute-plan/*`, a thin extension wrapper in `agent/extensions/execute-plan/*`, and a much thinner `agent/skills/execute-plan/SKILL.md`. The separation of concerns is clear and testable.
- The state/update model is thoughtfully designed. `updateState()` plus typed retry/cancellation fields gives the engine a solid persistence surface, and the TUI contract around `code_review_completed` is much cleaner than the old prose-skill flow.
- Test coverage is broad and targeted. I verified `cd agent && npx tsc --noEmit` and `cd agent && node --experimental-strip-types --test lib/execute-plan/*.test.ts extensions/execute-plan/*.test.ts`; both passed, with 316/316 tests green.
- Formatter extraction in `agent/extensions/execute-plan/tui-formatters.ts` is a good call: it keeps most rendering logic unit-testable and makes the TUI layer much easier to reason about.

### Issues

#### Critical (Must Fix)
- **Worker/process failures can be silently treated as successful wave execution, and the plan can still be archived as completed.** `TaskQueue.run()` swallows dispatch errors (`agent/lib/execute-plan/task-queue.ts:72-89`), and `dispatchWaveTasks()` simply skips any task missing from the results map (`agent/lib/execute-plan/engine.ts:845-847`). That means an implementer spawn failure can disappear entirely. I reproduced this with a targeted `node --experimental-strip-types --input-type=module` harness where `dispatchSubagent()` always threw: the engine still returned `outcome: 'completed'` and moved the plan into `.pi/plans/done/`. This is a correctness bug, not just a missing edge-case test.
- **Final-review retry exhaustion is reported as success.** In `executeFinalReview()`, the retry loop persists review retry state and dispatches fix-up work, but when retries are exhausted it unconditionally returns `true` (`agent/lib/execute-plan/engine.ts:1155-1169,1207-1238,1257-1260`). I reproduced this with a targeted harness where every `code_review` judgment returned `{ action: 'retry' }`: after 3 review attempts, the engine still returned `outcome: 'completed'` and archived the plan. That directly violates the intended halt/retry contract.

#### Important (Should Fix)
- **Task retry budgets do not actually resume from persisted state.** The startup flow loads `existingState.retryState` (`agent/lib/execute-plan/engine.ts:257`), but task-level retry handling resets to `let attempts = 0` every time (`agent/lib/execute-plan/engine.ts:890-1015`). I reproduced this with a harness seeded with `retryState.tasks["1"].attempts = 2`; the resumed run still performed 4 task dispatches before stopping. This falls short of the plan’s persisted-retry requirement (`/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:793-808,832-838`).
- **Final-review retries are built against stale review metadata and ignore retry context.** `headSha` is computed once before the final-review retry loop (`agent/lib/execute-plan/engine.ts:1155-1169`), so re-reviews after fix-up commits still point at the old HEAD. Separately, the retry fix-up prompt ignores `JudgmentResponse.context` entirely and only includes `summary.rawOutput` (`agent/lib/execute-plan/engine.ts:1224-1230`), even though the retry semantics in `agent/lib/execute-plan/types.ts:255-259` say retry context should be appended to the prompt. I confirmed the latter with a harness: a retry context of `Use a safer API` never appeared in the fix-up prompt.
- **The globally registered judgment tool is backed by a module-global bridge pointer, so concurrent starts can clobber an active execution’s resolver.** The extension stores only one `currentBridge` (`agent/extensions/execute-plan/index.ts:58-60`), overwrites it per run (`agent/extensions/execute-plan/index.ts:187`), reads it from callbacks (`agent/extensions/execute-plan/index.ts:311-315`), and nulls it in `finally` (`agent/extensions/execute-plan/index.ts:489`). A second `/execute-plan` invocation can replace the resolver for a first run before the repo-lock rejection even returns, causing pending judgments to fail or hang.
- **The subagent JSON-stream parser overwrites, rather than accumulates, assistant text chunks.** In `dispatchWorker()`, each text part from `message_end` does `finalOutput = part.text` (`agent/extensions/execute-plan/subagent-dispatch.ts:306-312`). If the assistant output is chunked, earlier text—including `STATUS:` or section headers—can be lost, producing bogus parse failures or downgraded `BLOCKED` results. The current tests cover parsing final output strings, but not multi-part `message_end` payloads.

#### Minor (Nice to Have)
- **Corrupted state files still throw hard instead of being handled as a resume/lock-validation problem.** `readState()` does a raw `JSON.parse` with no recovery path (`agent/lib/execute-plan/state-manager.ts:93-98`). That conflicts with the plan’s explicit risk mitigation to handle parse errors on resume (`/Users/david/Code/pi-config/.pi/plans/2026-04-10-execute-plan-extension.md:1223-1226`). A targeted Node check with malformed JSON reproduced the throw.
- **Worktree path construction trusts the editable branch string too much.** `createWorktree()` derives the directory name via `branch.replace(/^plan\//, "")` and feeds it straight into `join(worktreeDir, dirName)` (`agent/lib/execute-plan/worktree-ops.ts:62-64`). A quick Node check showed `path.join('/repo/.worktrees', '../../etc/passwd')` resolves to `/etc/passwd`, so a branch containing traversal segments can escape the intended worktree root.
- **`loadAgentConfig()` says parse failures return `null`, but malformed frontmatter still throws.** The read is guarded, but `parseFrontmatter(...)` is not (`agent/extensions/execute-plan/subagent-dispatch.ts:158-173`). A targeted malformed-agent repro raised `YAMLParseError` instead of returning `null`.
- **`commitWave()`/`getHeadSha()` do not validate git exit codes.** `git add`, `git commit`, and `git rev-parse HEAD` results are ignored (`agent/lib/execute-plan/git-ops.ts:38-56,105-108`). In a targeted mock-IO repro, `commitWave()` returned an empty SHA even though all git commands reported failure. Some call sites fail later because `updateWaveStatus()` rejects empty SHAs, but the helper contract is still unsound, and the final-review fix-up commit path does not check the returned SHA at all.

### Plan vs Implementation Assessment
- **`closeTodo()` completion note vs. unchanged body:** the plan is at fault here, not the implementation. Task 11 first says `closeTodo` should “append completion note” (`...execute-plan-extension.md:594-595`), but the same task later requires the round-trip test to keep the Markdown body unchanged (`...execute-plan-extension.md:605-613`). The current implementation in `agent/lib/execute-plan/plan-lifecycle.ts` preserves the body byte-for-byte, which is the only behavior consistent with the stronger compatibility requirement.
- **Persisted retry-state resume semantics:** the implementation is at fault. The plan explicitly requires resume to continue from persisted retry metadata (`...execute-plan-extension.md:793-808,832-838`), but task-level retries restart from zero in `handleTaskResult()`.
- **Final-review retry loop semantics:** the implementation is at fault. The plan’s requirement to retry fix-up work and re-run review until accepted/stopped/max-attempts (`...execute-plan-extension.md:874-895`) is sound; the current code still reports success on exhaustion and does not fully honor retry context or updated diff metadata.
- **Global judgment-tool design:** the plan is sound; the implementation is at fault. Registering the tool once globally while keeping per-execution pending promises is a good design (`...execute-plan-extension.md:27-35,983-1034`). The module-global `currentBridge` collapses that into a single mutable slot.
- **Corrupted state handling:** the implementation is at fault relative to the plan’s risk mitigation. The plan explicitly calls out handling parse errors on resume (`...execute-plan-extension.md:1223-1226`), but `readState()` still throws directly on malformed JSON.

### Recommendations
- Make `TaskQueue.run()` surface dispatch failures instead of swallowing them. Either reject the whole queue on infrastructure failure or record a synthetic failure result, and make `dispatchWaveTasks()` fail if any scheduled task has no result.
- Rework `executeFinalReview()` so retry exhaustion returns a stopped/failed outcome, recompute `HEAD` after each fix-up commit, and append `JudgmentResponse.context` to the fix-up prompt. Add regression tests for exhaustion, updated SHA usage, and retry-context propagation.
- Carry persisted task retry counts into `handleTaskResult()` on resume rather than resetting to zero.
- Replace the module-global `currentBridge` with execution-scoped routing keyed by run id/session id, so a second invocation cannot steal or clear the first execution’s resolver.
- Harden boundary/error handling: sanitize branch-derived worktree paths, catch malformed state/agent files, and make git helpers validate `exitCode` before returning success data.
- Add focused tests for the missing edge cases above: multi-chunk `message_end` payloads, malformed state files, malformed agent frontmatter, throwing `dispatchSubagent()`, and final-review exhaustion.

### Assessment
**Ready to merge?** No

**Reasoning:** The overall architecture is strong and the targeted suite is green, but there are still correctness holes that can mark failed work as successful and archive plans anyway. The silent worker-failure path and final-review-exhaustion path are merge blockers.
